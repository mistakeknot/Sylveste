# fd-ipc-authorization-boundary -- Review Findings

## Summary

The IPC authorization model is fundamentally sound: identity is derived from the filesystem directory path, not from container-supplied fields, and Docker mount isolation ensures containers can only write to their own group's IPC namespace. However, the dual-poller architecture (both Node and Rust poll the same IPC directories) creates race conditions where authorization can be applied by different processes with different authorization logic, and the Rust GroupRegistry's 10-second sync window creates a permissive authorization gap during startup and after group registration.

## Findings

### [P1] Rust IPC watcher has no task authorization -- delegates blindly to Node

- **File**: `/home/mk/projects/Sylveste/apps/intercom/rust/intercomd/src/ipc.rs:308-329`
- **Issue**: `process_tasks()` parses the IPC task file and forwards it to the Node host via `delegate.forward_task()` without performing any authorization check. It passes `ctx.is_main` (directory-derived, trustworthy) to Node via the HTTP body. However, because Rust itself performs zero authorization on tasks, ALL task authorization is deferred to Node's `processTaskIpc()`. This creates an implicit contract: if Rust ever stops forwarding to Node and handles tasks natively, there is no authorization layer in Rust for task operations (schedule, pause, resume, cancel, register_group).
- **Scenario**: If the strangler-fig migration progresses and Rust starts handling `schedule_task` natively (without the Node callback), a non-main container could schedule tasks targeting other groups because no Rust-side authorization exists for task operations. The `IpcDelegate` trait's `forward_task` signature (line 66) does not enforce that the delegate perform authorization.
- **Fix**: Add a Rust-side authorization check in `process_tasks()` before delegating. At minimum, for `IpcTask::RegisterGroup`, reject if `!ctx.is_main`. For `IpcTask::ScheduleTask`, verify the target JID belongs to the source group (using the GroupRegistry). This ensures defense-in-depth regardless of which process handles the task.

### [P1] Dual IPC poller race: Node and Rust both poll the same directories

- **File**: `/home/mk/projects/Sylveste/apps/intercom/src/ipc.ts:45-198` and `/home/mk/projects/Sylveste/apps/intercom/rust/intercomd/src/ipc.rs:234-258`
- **Issue**: Both Node (`startIpcWatcher`) and Rust (`IpcWatcher::run`) poll the same `data/ipc/` directory tree on a 1-second interval. Both read, process, and unlink files. For messages, Node applies its own authorization check (line 79-82 of `ipc.ts`) while Rust applies a slightly different check (line 279 of `ipc.rs`). For tasks, Node applies full authorization (lines 229-435 of `ipc.ts`) while Rust applies none (line 319-321 of `ipc.rs`). A file could be processed by either poller depending on timing.
- **Scenario**: A non-main container writes a `register_group` task. If Rust picks it up first, it forwards to Node with `is_main=false` (from directory), Node rejects it -- correct. But the race means some files go through Rust's authorization path and some through Node's. For messages, if Node picks up a file, it uses `registeredGroups[data.chatJid]` (in-memory, always fresh). If Rust picks it up, it uses `GroupRegistry::folder_for_jid()` which may be up to 10 seconds stale (see P2). Different authorization decisions for the same operation depending on which poller wins the race.
- **Fix**: Designate one process as the exclusive IPC poller. Since Rust handles queries natively already, make Rust the sole poller and have it forward messages and tasks to Node via the existing callback endpoints. Alternatively, separate the IPC directories: Rust polls `data/ipc-rust/` and Node polls `data/ipc-node/`, with containers writing to the appropriate one based on which operations Rust handles natively vs. delegates.

### [P2] Rust GroupRegistry has a 10-second stale window for message authorization

- **File**: `/home/mk/projects/Sylveste/apps/intercom/rust/intercomd/src/ipc.rs:694-750`
- **Issue**: The Rust `GroupRegistry` is populated by `sync_registry_loop()`, which fetches registered groups from the Node host every 10 seconds (line 712). During the first 12 seconds of startup (2-second delay + 10-second interval), the registry is empty. An empty registry means `is_authorized_target()` (line 578-583) returns `false` for all non-main messages, which is a safe default (deny). However, between sync cycles, a newly registered group exists in Node's in-memory state and SQLite but not in Rust's GroupRegistry. During this window, Rust would deny legitimate messages from the new group if it polls the IPC file first.
- **Scenario**: Main group registers "team-eng" via IPC. Node processes the `register_group` task and updates its in-memory `registeredGroups`. Team-eng's container writes a message. If Rust's IPC poller picks up the message before the next 10-second registry sync, `is_authorized_target` returns false because team-eng's JID is not yet in the Rust registry. The message is logged as "Unauthorized IPC message attempt blocked" and discarded (not moved to errors -- it's `remove_file`'d at line 298).
- **Fix**: Either (1) trigger an immediate registry sync after Node processes a `register_group` task (push-based update), or (2) fall through to Node for authorization when Rust's registry has no entry for a JID (treat unknown as "defer to Node" rather than "deny"). Option 1 is simpler: add a `POST /v1/ipc/registry-changed` callback that Rust listens for.

### [P2] register_group writes to SQLite only -- invisible to Rust daemon's Postgres-based group state

- **File**: `/home/mk/projects/Sylveste/apps/intercom/src/index.ts:67-89` (registerGroup function) and `/home/mk/projects/Sylveste/apps/intercom/src/db.ts:686-704` (setRegisteredGroup)
- **Issue**: When a group is registered via IPC `register_group`, the Node `registerGroup()` function calls `setRegisteredGroup()` which writes to SQLite only (line 690-703 of `db.ts`). It also updates the in-memory `registeredGroups` map. However, the Rust daemon reads groups from Postgres (`pool.get_all_registered_groups()` at `/home/mk/projects/Sylveste/apps/intercom/rust/intercomd/src/process_group.rs:99`). A group registered via IPC exists only in SQLite and Node's memory -- it is invisible to Rust's process_group_messages, which means Rust will skip processing messages for this group (line 102: `None => return Ok(true)`).
- **Scenario**: Main container registers "team-eng" via the IPC `register_group` task. Node stores it in SQLite and in-memory. Rust's `sync_registry_loop` fetches the group from Node's `/v1/ipc/registered-groups` endpoint, so it appears in Rust's IPC authorization GroupRegistry. But when an actual message arrives for team-eng in the outbox, Rust's `process_group_messages` looks up the group in Postgres (`groups.read().await.get(chat_jid)`), finds nothing, and skips it. The group can receive IPC-level authorization but cannot actually have its messages processed by Rust.
- **Fix**: This is explicitly noted as out of scope for this review (covered by fd-dual-database-state-divergence). Noted here for completeness because it interacts with the IPC authorization boundary: a group can pass IPC authorization checks but fail at the process_group level.

### [P2] Node query handler accepts sourceGroup and isMain but ignores them for read operations

- **File**: `/home/mk/projects/Sylveste/apps/intercom/src/query-handlers.ts:205-233`
- **Issue**: The Node `handleQuery()` function receives `sourceGroup` and `isMain` as parameters (line 208-209) but does not use them for any authorization decision. All query types (run_status, sprint_phase, search_beads, spec_lookup, review_summary, next_work, run_events, research) are accessible to all groups regardless of `isMain` status. The Rust SylvesteAdapter applies `is_main` checks for write operations (line 162 of `sylveste.rs`) but not for reads. Node handles no write operations via this path.
- **Scenario**: A non-main container can query `run_events`, `search_beads`, `spec_lookup`, and `next_work` -- all of which expose full project state including other groups' work items. While this may be intentional (read-only queries are lower risk), the inconsistency between "reads are open" and "writes require main" is undocumented. If any of these queries returns sensitive data (e.g., `search_beads` returning another group's task prompts), non-main groups have access.
- **Fix**: If this is intentional, document the authorization policy in the `handleQuery` function. If non-main groups should have limited visibility, add filtering based on `sourceGroup` to queries like `search_beads` and `run_events`.

### [P3] Container-supplied isMain and groupFolder fields in IPC JSON are redundant and potentially confusing

- **File**: `/home/mk/projects/Sylveste/apps/intercom/container/shared/ipc-tools.ts:110,121,132` and `/home/mk/projects/Sylveste/apps/intercom/container/agent-runner/src/ipc-mcp-stdio.ts:200,219,238`
- **Issue**: Container-side IPC tools write `isMain` and `groupFolder` fields into the task JSON files (e.g., `pause_task` at ipc-tools.ts:110). These fields come from the container's `ContainerInput`, which is host-controlled. However, the host-side IPC processors never read these container-supplied fields -- they derive identity from the directory path. The `IpcTask` Rust struct at `/home/mk/projects/Sylveste/apps/intercom/rust/intercom-core/src/ipc.rs:49-68` includes `group_folder: Option<String>` fields on PauseTask, ResumeTask, CancelTask, but Rust ignores them, using `ctx.group_folder` from the directory instead. Node's `processTaskIpc` similarly uses the `sourceGroup` parameter (from the directory) rather than `data.groupFolder`.
- **Scenario**: No direct security impact because the host ignores these fields. However, the presence of these fields is misleading and could lead a future developer to accidentally use `task.group_folder` instead of `ctx.group_folder`, bypassing the directory-derived identity model. A container that crafts a task JSON with `"groupFolder": "main"` and `"isMain": true` would have no effect today, but the fields' presence invites mistakes.
- **Fix**: Remove `isMain` and `groupFolder` fields from container-written task JSON. If they need to exist for logging or debugging, rename them to `_source_group` and `_source_is_main` to signal they are informational, not authoritative.

### [P3] IPC file poller does not validate directory entries against symlink attacks

- **File**: `/home/mk/projects/Sylveste/apps/intercom/rust/intercomd/src/ipc.rs:236-249` and `/home/mk/projects/Sylveste/apps/intercom/src/ipc.ts:49-57`
- **Issue**: Both IPC pollers use `read_dir` / `readdirSync` to enumerate group directories under `data/ipc/`. They check `is_dir()` / `isDirectory()` but do not check whether the directory entry is a symlink. If an attacker can create a symlink at `data/ipc/main -> data/ipc/team-eng/`, the poller would read files from team-eng's directory but treat them as coming from "main" (since the directory name is "main"), granting main-group privileges.
- **Scenario**: Requires write access to `data/ipc/` outside a container, which is normally protected by filesystem permissions and Docker mount isolation. Inside the container, the mount is `data/ipc/{group}` -> `/workspace/ipc`, so a container cannot create entries in the parent `data/ipc/` directory. Risk is limited to host-level compromise, but defense-in-depth suggests validating.
- **Fix**: Use `lstat` / `lstatSync` instead of `stat` / `statSync` to detect symlinks, and skip symlinked directories. In Rust, use `entry.file_type()` which already reads the directory entry type without following symlinks (this is correct behavior in Rust's `read_dir`). In Node, change `fs.statSync` at line 50 of `ipc.ts` to `fs.lstatSync` to avoid following symlinks.
