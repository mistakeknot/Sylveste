## Task Context

Intercom is a dual-process system (Node host + Rust IronClaw daemon) where messages flow from Telegram/WhatsApp through Node, are stored in Postgres by Rust, then polled every 1 second and dispatched to Docker containers via a per-group queue. The redesign aims to replace polling and fire-and-forget HTTP callbacks with reliable event-driven IPC.

## Decision Lens

Prioritize connection lifecycle correctness (what happens when Node restarts mid-session, when Postgres connection drops, when a container dies with pending output) over raw latency numbers. A transport that requires manual reconnect handling is a reliability hazard.

## Review Approach

### 1. Postgres LISTEN/NOTIFY assessment
- Evaluate whether a single persistent connection in tokio-postgres is sufficient, how connection drops are detected and reconnected, and whether 8KB NOTIFY payload limit constrains message routing

### 2. Unix domain socket vs HTTP callbacks
- Evaluate UDS between Rust and Node as a replacement for HttpDelegate HTTP callbacks: does it eliminate the reqwest timeout issues, and can Node's Grammy event loop handle a persistent UDS server alongside HTTP?

### 3. Filesystem IPC watcher overhead
- Review ipc.rs poll_once(): assess minimum poll interval for acceptable latency vs. inotify-based file watching, and whether sorted JSON file approach creates ordering hazards under concurrent writes

### 4. Container stdout protocol alternatives
- Examine the OUTPUT marker protocol: assess whether replacing stdout parsing with a Unix socket or named pipe between container and runner eliminates partial-read and truncation risks

### 5. Group registry sync race
- Analyze sync_registry_loop (10s poll from Node): identify whether this creates a startup race where intercomd processes IPC for unregistered groups, and whether event-push would close it

### 6. Bidirectional vs unidirectional IPC
- Compare single-direction (Rust→Node) vs bidirectional socket IPC: assess whether Node needs to push events back to Rust and whether bidirectional complicates ownership

## Success Criteria

- The chosen transport handles Node process restart without losing in-flight IPC messages or requiring manual connection re-establishment
- Container output extraction does not depend on parsing sentinel markers from an unbounded stdout stream
- LISTEN/NOTIFY adoption (if chosen) includes reconnect logic with backoff and a fallback poll to prevent silent loss
