## Task Context

Intercom is a dual-process system (Node host + Rust IronClaw daemon) where messages flow from Telegram/WhatsApp through Node, are stored in Postgres by Rust, then polled every 1 second and dispatched to Docker containers via a per-group queue. The redesign aims to replace polling and fire-and-forget HTTP callbacks with reliable event-driven IPC.

## Decision Lens

Treat any design where a slow or misbehaving group can delay, corrupt, or observe another group's messages as a regression, regardless of latency savings. Authorization checks in the current polling path must be audited for presence in any replacement.

## Review Approach

- Review GroupQueue in queue.rs and process_group.rs: would LISTEN/NOTIFY deliver all groups on a single shared channel? Does per-group dispatch correctly re-introduce isolation after shared notify?

### 2. IPC authorization preservation

### 3. Outbox multi-tenant behavior
- Assess outbox pattern: if a Postgres outbox stores messages from all groups, can a single slow consumer block delivery for others? Does schema need per-group partitioning?

### 4. Per-group cursor concurrency
