## Severity Calibration

- **P0**: A new NOT NULL column is added without a DEFAULT, causing the migration to fail mid-table on production data and leaving a partially-altered table with no rollback.
  - When: Occurs on any non-empty table when the migration runs against production and the error handling does not wrap ALTER TABLE in a transaction.
- **P0**: Migration version tracking does not lock the migrations table before inserting, allowing two concurrent startup processes to both apply the same migration, resulting in duplicate indexes or double-applied data transforms.
  - When: Triggered when the service restarts under load or during a rolling deploy with multiple replicas.
- **P1**: A rollback migration drops a column that a still-deployed binary is actively writing, causing all writes to that column to silently succeed with data going nowhere.
  - When: Occurs during a blue-green or canary deployment where the old binary runs against the rolled-back schema.

When in doubt: describe the failure scenario. If it wakes someone at 3 AM, it is P0/P1. If it degrades quality over weeks, it is P2.
