## Task Context

Intercom is a dual-process system (Node host + Rust IronClaw daemon) where messages flow from Telegram/WhatsApp through Node, are stored in Postgres by Rust, then polled every 1 second and dispatched to Docker containers via a per-group queue. The redesign aims to replace polling and fire-and-forget HTTP callbacks with reliable event-driven IPC.

## Decision Lens

Prioritize migration paths that can be rolled back at each step over big-bang rewrites. Treat any step that requires both processes to be down simultaneously as a coordination hazard that needs explicit mitigation.

## Review Approach

### 1. Node-only capabilities inventory
- Map which Node capabilities have no Rust equivalent: Grammy (Telegram SDK with inline keyboards), Baileys (WhatsApp), summarizer.ts (GPT-5.3 Codex call), stream-accumulator.ts (real-time Telegram edit) — assess replaceability

### 2. Orchestrator sidecar fallback
- Review orchestrator.enabled flag in intercom.toml: does sidecar mode provide a safe fallback during migration? Is the flag granular enough to migrate one subsystem at a time?

### 3. Cross-boundary endpoint audit
- Examine host-callback.ts and intercomd-client.ts: classify every Node↔Rust endpoint as (a) already redundant, (b) migration-only, or (c) permanently required

### 4. Dual-persistence cutover path
- Assess SQLite↔Postgres state: identify which tables are still written by Node and whether collapse requires hard cutover or can be done table-by-table

### 5. Service startup ordering
- Review systemd ordering (intercomd.service Before= intercom.service): does single-process eliminate this, or does container/channel initialization create an equivalent dependency?

### 6. Partial collapse cost-benefit
- Evaluate keeping both processes but replacing HTTP with LISTEN/NOTIFY: does the resulting single IPC hop actually simplify, or just shift complexity from polling to event subscription?

## Success Criteria

- Each migration step can be deployed independently with the previous step still providing fallback — no coordinated downtime
- After collapse, the number of active IPC channels is strictly less than before — measurably simpler
- The SQLite dual-write is eliminated before the Node process is removed — no orphaned persistence layer
