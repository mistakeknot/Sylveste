## Task Context

Auraken's profile architecture is described in the PRD as a three-tier system with bi-temporal entities, Zettelkasten-style linking, and a FluxMem-inspired adaptive synthesis layer. This is one of the most ambitious personal knowledge architectures proposed for a conversational AI -- most competitors (ChatGPT, Gemini) use flat key-value memory. The architecture must work at conversation speed, grow over months/years without degradation, and represent things that are genuinely hard to represent (thinking patterns, contradictions, cross-domain connections). The expansion to product recommendations, health, and finance adds new entity types and linking patterns to an already complex graph.

## Review Approach

### 1. Three

- Three-tier architecture viability: Episode (immutable) -> Entity (bi-temporal) -> Synthesis (derived) is elegant on paper. How does entity extraction actually work at conversation scale? What is the precision/recall tradeoff for thinking pattern detection vs. factual extraction?

### 2. Bi

- Bi-temporal entity modeling: Entities have valid_from/valid_until timestamps, which is sophisticated. But how does the system decide when a thinking pattern has changed vs. when it is contextually different? Values autonomy at work might be true in March and false in July -- or it might be true with one manager and false with another.

### 3. Knowledge graph maintenance burden: The Zettelkasten

- Knowledge graph maintenance burden: The Zettelkasten-style entity linking (A-MEM) requires maintaining connections between entities. Who maintains these links -- the LLM automatically, the user manually, or some hybrid? Auto-linking has high false-positive risk; manual linking creates unsustainable maintenance burden.

### 4. Synthesis layer token economics: The synthesis layer gene...

- Synthesis layer token economics: The synthesis layer generates 300-500 token compact summaries for system prompt injection. As the profile grows over months/years, how does the system decide what makes it into the synthesis? Is there a compression/fidelity tradeoff that degrades profile quality?

### 5. Retrieval architecture: When the agent needs to draw on a...

- Retrieval architecture: When the agent needs to draw on accumulated context mid-conversation, what is the retrieval pipeline? Full profile injection, RAG over entities, or something else? How does retrieval latency affect conversation flow?

### 6. Profile decay

- Profile decay and contradiction resolution: Over months of conversation, some patterns become stale, some contradict newer observations. How does the system handle profile decay without losing valuable historical context? How does it distinguish changed mind from contextual variation?

## Severity Calibration

- **P1**: Entity extraction precision is too low -- the system generates a high volume of weakly-supported thinking pattern claims that pollute the synthesis layer, causing the agent to make confident but wrong assertions about how the user thinks.
  - When: No confidence scoring for extracted entities, no minimum evidence threshold, no user review mechanism for contested entities.
- **P2**: The synthesis layer compression loses critical nuance -- a 500-token summary of months of conversation flattens contradictions and context-dependent patterns into oversimplified labels, undermining the warm data promise.
  - When: Fixed token budget with no adaptive structure, no preservation of contradiction pairs in synthesis, no mechanism to expand synthesis for complex profiles.
- **P2**: Knowledge graph links accumulate without pruning, creating a maintenance debt that degrades retrieval quality over time -- the Zettelkasten graveyard problem where everything is connected to everything.
  - When: No link strength decay, no pruning mechanism, no relevance-weighted retrieval over the graph.

When in doubt: describe the failure scenario. If it wakes someone at 3 AM, it is P0/P1. If it degrades quality over weeks, it is P2.
