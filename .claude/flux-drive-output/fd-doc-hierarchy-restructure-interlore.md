# Flux Drive: Doc Hierarchy Migration Risk Review -- interlore PRD

## Summary

The PRD introduces MISSION.md as a new root document and promotes VISION.md alongside PHILOSOPHY.md as co-equal siblings. The interlore plugin itself is well-scoped. However, the doc hierarchy restructure (F1) has **severe migration gaps**: 111 files reference PHILOSOPHY.md, 47 structural test suites hardcode it as a required file, 48 AGENTS.md files contain the "Philosophy Alignment Protocol" boilerplate, and the canonical `plugin-standard.md` defines the exact AGENTS.md header that every plugin copies. The PRD addresses none of this.

---

## Findings

### P0: No Migration Plan for 111 Cross-References to PHILOSOPHY.md

**Location:** PRD F1 acceptance criteria (lines 24-28)

**Issue:** The PRD's acceptance criteria for F1 list exactly 5 items, all focused on the root project. It does not acknowledge or plan for the 111 files across the codebase that reference `PHILOSOPHY.md`. The breakdown:

- **48 AGENTS.md files** in `interverse/*/AGENTS.md` contain the "Philosophy Alignment Protocol" boilerplate with `[PHILOSOPHY.md](./PHILOSOPHY.md)` links (158 total occurrences across these 48 files)
- **47 `test_structure.py` files** in `interverse/*/tests/structural/` hardcode `"PHILOSOPHY.md"` in their `required` file list -- these are CI-blocking structural tests
- **52 PHILOSOPHY.md files** exist across the ecosystem (52 in interverse plugins, 2 in os/ pillars, 1 at project root)
- **interlab `plugin-benchmark.sh`** (line 193) iterates over required files including PHILOSOPHY.md
- **Clavain `project-onboard` skill** (SKILL.md lines 32, 66; SKILL-compact.md lines 13, 36) generates PHILOSOPHY.md from a template during onboarding
- **Clavain `watchables.yaml.tmpl`** (line 22) watches `PHILOSOPHY.md` by path
- **interscribe SKILL.md** (line 205) lists PHILOSOPHY.md in its reference directory tree
- **docs/demarch-vision.md** has 5+ inline references to specific PHILOSOPHY.md sections (e.g., "See PHILOSOPHY.md section Earned Authority")

**Failure scenario:** If the restructure changes the semantics or role of PHILOSOPHY.md (even without renaming it), every one of these 48 AGENTS.md boilerplate sections becomes misleading. If the restructure ever promotes MISSION.md to where PHILOSOPHY.md cross-refs point, 47 test suites will fail on every plugin.

**Smallest fix:** Add an acceptance criterion to F1: "Cross-reference inventory completed and migration approach decided (update in place, scripted batch, or deferred to follow-up bead)." The PRD does not need to execute the migration, but it must acknowledge the blast radius and assign it.

### P0: plugin-standard.md Canonical Boilerplate Must Be Updated

**Location:** `/home/mk/projects/Demarch/docs/canon/plugin-standard.md` lines 122-145 (AGENTS.md Standard Header)

**Issue:** `plugin-standard.md` defines the exact AGENTS.md header that every plugin is expected to copy. This header is:

```markdown
## Canonical References
1. [`PHILOSOPHY.md`](./PHILOSOPHY.md) -- direction for ideation and planning decisions.

## Philosophy Alignment Protocol
Review [`PHILOSOPHY.md`](./PHILOSOPHY.md) during:
- Intake/scoping
- Brainstorming
...
```

If MISSION.md becomes the root and VISION.md is introduced as a sibling, this boilerplate must change. The PRD does not mention `plugin-standard.md` at all. Since plugin-standard.md is labeled "the canonical reference -- not a suggestion" (line 3), any hierarchy change that does not update it creates an immediate contradiction: the standard says PHILOSOPHY.md is the canonical reference for ideation/planning, but the new hierarchy says MISSION.md is the root.

**Failure scenario:** New plugins created after the restructure (or existing plugins regenerated by interscribe/interdoc) will use the old boilerplate. Clavain's `project-onboard` skill generates PHILOSOPHY.md from a template -- it has no template for MISSION.md or VISION.md.

**Smallest fix:** Add an acceptance criterion: "plugin-standard.md AGENTS.md Standard Header updated to reflect MISSION/VISION/PHILOSOPHY hierarchy." And a separate criterion: "Clavain project-onboard templates updated with MISSION.md.tmpl."

### P1: MISSION.md vs PHILOSOPHY.md Semantic Boundary Is Undefined

**Location:** PRD F1 (lines 19-28), Brainstorm "Key Decisions" section (lines 37-41)

**Issue:** The brainstorm gives a one-line distinction: "Philosophy = how we build (design bets, principles). Vision = where we're going." But it says nothing about what MISSION.md contains that is currently NOT in PHILOSOPHY.md's opening section.

PHILOSOPHY.md currently opens with (lines 1-5):

> "The design bets, tradeoffs, and convictions that inform everything else. CLAUDE.md says *how to work here*. AGENTS.md says *what to build and how*. This document says *why these tradeoffs and not others*."

This is already a mission-level statement. The three principles that follow ("Every action produces evidence," "Evidence earns authority," "Authority is scoped and composed") read as mission axioms, not implementation philosophy.

The PRD's acceptance criteria say "MISSION.md exists at project root with one-paragraph mission statement" but do not define:
- What content moves OUT of PHILOSOPHY.md's opening section into MISSION.md
- Whether PHILOSOPHY.md's three principles stay, move, or get duplicated
- What question MISSION.md answers that PHILOSOPHY.md's existing "why these tradeoffs" framing does not

**Failure scenario:** MISSION.md and PHILOSOPHY.md's opening paragraphs overlap. Agents reading both get conflicting or redundant framing. The hierarchy becomes decoration rather than structure.

**Smallest fix:** Add an acceptance criterion: "PHILOSOPHY.md opening section updated to remove mission-level content that moved to MISSION.md. Semantic boundary documented in doc-structure.md: MISSION answers 'why do we exist,' PHILOSOPHY answers 'what design bets follow from that mission,' VISION answers 'where are we going next.'"

### P1: VISION.md Is a Rename/Promotion of docs/demarch-vision.md -- 18+ References Unaddressed

**Location:** PRD F1 (line 21), existing file at `docs/demarch-vision.md` (v3.4, 2026-03-13)

**Issue:** The project already has a mature vision document at `docs/demarch-vision.md` (version 3.4, actively maintained). The PRD says "Establish VISION.md" at project root as a sibling of PHILOSOPHY.md. It does not acknowledge that `demarch-vision.md` exists, nor specify whether VISION.md is:
- A rename/move of `docs/demarch-vision.md` to `VISION.md`
- A new summary document that links to `docs/demarch-vision.md`
- A replacement

18 files reference `demarch-vision` by name (including `docs/architecture.md`, `docs/demarch-reference.md`, `CONVENTIONS.md`, `.interwatch/watchables.yaml`, and multiple plans/PRDs).

**Failure scenario:** If VISION.md is created as a new file without addressing `demarch-vision.md`, the project has two vision docs with no defined relationship. If it is a rename, 18 cross-references break silently (no CI catches broken markdown links).

**Smallest fix:** Add an acceptance criterion: "VISION.md is {rename of | summary pointing to} docs/demarch-vision.md. Old path either removed with redirect or retained as symlink. Cross-references updated."

### P1: Precedence in Conflicts Between MISSION, VISION, and PHILOSOPHY Is Unspecified

**Location:** PRD F1, Brainstorm "Key Decisions" (lines 37-41)

**Issue:** The brainstorm says "VISION.md and PHILOSOPHY.md are siblings (both derived from mission, neither derives from the other)." But when vision says "ship fast" and philosophy says "review first," what wins? The existing project has a clear answer -- PHILOSOPHY.md is authoritative (it is the first canonical reference in every AGENTS.md, and the alignment protocol says to adjust plans to align with it or create follow-up work to update it).

Introducing MISSION.md as root without defining a conflict resolution rule means:
- Does MISSION.md override PHILOSOPHY.md? (Root overrides children?)
- Does PHILOSOPHY.md override VISION.md? (Current implicit behavior?)
- Are siblings truly equal? (Then who arbitrates?)

**Failure scenario:** An agent sees a PRD that conflicts with PHILOSOPHY.md but aligns with VISION.md. The existing alignment protocol says "adjust the plan to align [with PHILOSOPHY.md] or create follow-up work to update PHILOSOPHY.md explicitly." Does this still apply? Does the agent now also check MISSION.md? The protocol becomes ambiguous.

**Smallest fix:** doc-structure.md updated hierarchy section must include a "conflict resolution" rule, e.g.: "MISSION.md is immutable baseline. When VISION and PHILOSOPHY conflict, PHILOSOPHY wins for implementation decisions, VISION wins for scope/direction decisions. Conflicts are surfaced as interlore proposals."

### P2: doc-structure.md Claims "auto-loaded by interdoc" -- PRD's F1 Says to Remove This But No Replacement Rationale Given

**Location:** `docs/canon/doc-structure.md` line 87; PRD F1 acceptance criteria line 26

**Issue:** doc-structure.md currently says:

> "PHILOSOPHY.md stays at project root (auto-loaded by interdoc's philosophy alignment protocol)."

The PRD's F5 acknowledges interdoc doesn't actually load PHILOSOPHY.md (it is inert prose). F1 acceptance criteria say "doc-structure.md no longer claims 'auto-loaded by interdoc' for PHILOSOPHY.md placement rationale." But the PRD does not specify what the *replacement* rationale for PHILOSOPHY.md's placement should be. After F5 fixes interdoc to actually load PHILOSOPHY.md, the "auto-loaded by interdoc" claim becomes true again. The acceptance criterion is contradictory with F5.

**Smallest fix:** Clarify: F1 removes the false claim *now*. F5 makes the loading real. doc-structure.md should be updated twice: once in F1 (remove false claim, replace with "PHILOSOPHY.md stays at root for agent accessibility"), once in F5 (add "loaded by interdoc during generation" once that is actually wired).

### P2: 47 Structural Test Suites Hardcode PHILOSOPHY.md as Required -- No Batch Update Strategy

**Location:** `interverse/*/tests/structural/test_structure.py` line 35 (pattern across 47 plugins)

**Issue:** Every plugin's structural test suite includes a line like:

```python
required = ["CLAUDE.md", "PHILOSOPHY.md", "LICENSE", ".gitignore"]
```

These tests assert that PHILOSOPHY.md exists at the plugin root. If the hierarchy restructure changes plugin-standard.md to require MISSION.md alongside (or instead of) PHILOSOPHY.md, all 47 test suites need updating. This is a mechanical change but the PRD does not mention it.

Additionally, `interlab/scripts/plugin-benchmark.sh` line 193 iterates the same required-file list.

**Smallest fix:** If PHILOSOPHY.md remains required at the plugin level (likely -- the PRD only adds MISSION.md at project root), document explicitly that plugin-level PHILOSOPHY.md is unaffected. If MISSION.md is also required at plugin level, add a bead for the batch update.

This is a quality-of-life improvement, not a blocker.
