# Plan: interop — Unified Integration Fabric

## Feature Ordering

```
Phase 1: F1 (core daemon + event bus)
Phase 2: F2 (beads) + F5 (filesystem) — parallel, both depend only on F1
Phase 3: F4 (Notion) + F3-partial (GitHub Issues/PRs, no file sync) — parallel
```

### Task 1.2: Event type + Adapter interface

**Files:** `internal/adapter/adapter.go`, `internal/adapter/event.go`, `internal/adapter/errors.go`
**Steps:**
1. `event.go` — `Event` struct: `ID string`, `EntityID string`, `EntityKey string`, `Type EventType` (typed constants: `EntityCreated`, `EntityUpdated`, `EntityDeleted`, `EntityConflict`), `SourceAdapter AdapterID`, `Timestamp time.Time` (interop receive-time, NOT external timestamp — fixes P1-BSC-02), `SchemaVersion int`, `RoutingHints map[string]string`, `TracePoints []TracePoint`, `Payload EventPayload` (sealed interface with unexported `isEventPayload()` method)
   **EntityKey canonical format** (P0 fix: all adapters MUST use these formats — CollisionWindow and AncestorStore key on EntityKey, mismatched keys cause silent sync failure):
   - Beads: `beads:issue:<bead-id>` (e.g. `beads:issue:sylveste-bcok`)
   - Filesystem: `fs:file:<relative-path>` (relative to watched dir, e.g. `fs:file:docs/README.md`)
   - Notion pages: `notion:page:<notion-page-id>` (e.g. `notion:page:abc-123-def`)
   - Notion databases: `notion:db:<notion-db-id>`
   - GitHub issues: `github:issue:<owner>/<repo>#<number>` (e.g. `github:issue:mistakeknot/interop#42`)
   - GitHub files: `github:file:<owner>/<repo>:<path>` (e.g. `github:file:mistakeknot/interop:docs/README.md`)
   **Cross-system entity correlation:** When two adapters refer to the same logical entity (e.g. a Notion page synced to a local file), both must emit events with EntityKeys that resolve to the same entry in a `entity_correlation` table (new, in identity module). The correlation table maps `(source_key, target_key)` pairs established at first sync. CollisionWindow and AncestorStore use the canonical (first-registered) key. Without this, cross-adapter collision detection is impossible.
2. `adapter.go` — `Adapter` interface: `Start(ctx context.Context) error`, `Stop() error` (must close Emit channel before returning — fixes P1-AIC-03), `HandleEvent(Event) error` (must return within 5s, long ops dispatched to goroutine — fixes P1-AIC-01), `Emit() <-chan Event`, `ID() AdapterID`. Optional: `HealthChecker` (`Health() HealthStatus`), `StateSyncer` (`FullSync(ctx) error`)
3. `errors.go` — typed errors: `ErrMalformed` (discard), `ErrTransient` (retry), `ErrConflict` (escalate to SyncJournal), `ErrShuttingDown` (requeue), `ErrUnresolvableTarget` (dead-letter, not silent drop)
**Test:** Compile check; mock adapter implements interface

### Task 1.4: EventBus with per-entity serialization + Emit validation

**Files:** `internal/bus/bus.go`, `internal/bus/bus_test.go`
**Steps:**
1. `EventBus` struct: registered adapters, `CollisionWindow` (field, not separate — P0 fix), per-entity dispatch channels (`sync.Map` keyed by `EntityKey` → `chan Event`, created on demand, GC'd after 60s idle — `sync.Map` required: concurrent adapter goroutines create channels simultaneously, plain map races. P0 fix from plan review), dispatch `sync.WaitGroup` (tracks all goroutines spawned by dispatch — P0 fix for shutdown)
2. `Emit(e Event) error` — validates before enqueueing: non-empty EntityID, non-nil Payload, valid Type, non-empty SourceAdapter. Invalid → logged + dropped (contamination barrier — fixes P0-6 from flux-review)
3. `Register(a Adapter)` / `Unregister(a AdapterID)`. **Register after StartDispatching:** must atomically spawn a dispatch goroutine for the new adapter (P0 fix: late-registered adapters must not be silently ignored)
4. **Two-stage dispatch loop** (P0 fix: entity channels must be both filled AND drained):
   - **Stage 1 (adapter → entity channel):** Per-adapter goroutine reads from `a.Emit()`, sets `e.Timestamp = time.Now()`, calls `Emit(e)` which routes to per-entity channel.
   - **Stage 2 (entity channel → adapters):** Per-entity consumer goroutine reads from entity channel, calls `collisionWindow.Check(e)` (P0 fix: wiring CollisionWindow into dispatch path). On collision → route pair to `ConflictResolver.ResolvePair`. No collision → call `dispatchToAdapters(e)` (unexported — P1 fix: prevents callers from bypassing per-entity serialization). `dispatchToAdapters` fans out to all other adapters' HandleEvent, skipping source adapter.
   - **Consumer goroutine lifecycle:** Created on demand with entity channel. Tracked in dispatch WaitGroup. Exits when entity channel is closed during shutdown drain.
5. `Requeue(e Event)` — for panic recovery in adapter goroutine pools. **Retry limit:** max 3 requeues per event (tracked via `Event.RetryCount`), then dead-letter to SyncJournal with `ErrMaxRetries` (P1 fix: prevent infinite requeue loops)
6. **Priority lanes deferred** — remove three-lane priority select from Phase 1. Single FIFO channel per entity. Priority field remains on Event struct for future use but dispatch does not differentiate. Avoids YAGNI complexity; add when a concrete consumer needs express delivery.
**Test:** 2 mock adapters, emit from A, verify B receives; emit from B, verify A receives; emit invalid event, verify rejected; emit same EntityKey from A and B simultaneously, verify serialized (not interleaved); **emit opposing events within TTL → collision detected (end-to-end bus+collision test)**; late Register after StartDispatching → adapter receives events

### Task 1.5: CollisionWindow

**Files:** `internal/bus/collision.go`, `internal/bus/collision_test.go`
**Steps:**
1. `CollisionWindow` struct: `map[EntityKey]pendingEvent` protected by `sync.Mutex` (P0 fix: concurrent reads/writes from dispatch goroutines race without mutex) with configurable TTL (default 5s). **Owned by EventBus as a field** (P0 fix: not a standalone struct — must be wired into the Stage 2 dispatch path in Task 1.4)
2. `Check(e Event) CollisionResult` — called by the per-entity consumer goroutine (Task 1.4 Stage 2) before dispatch. Returns `NoCollision`, `Collided(pair)`, or `Buffered`
3. On collision: route pair to `ConflictResolver.ResolvePair(a, b Event)` — write `ConflictRecord` to SyncJournal with `status=pending` (fixes P1-ACQ-01: externally observable via `/conflicts`)
4. No collision: add to window, dispatch after TTL expires or next tick
5. `StartFlusher(done <-chan struct{}, dispatch func(Event))` — **must be called by Daemon.Run()** (P0 fix: flusher was never started). Ticks at `ttl/2`, dispatches expired non-colliding events
6. **Shutdown drain:** `Flush()` must be called during graceful shutdown before `close(done)` to release pending non-colliding events. Events remaining after Flush are logged as dropped with their EntityKeys for manual recovery
**Test:** Two opposing events within TTL → conflict detected; two events outside TTL → both dispatched; same-source events → no collision; **end-to-end: emit from adapter → collision detected → ConflictRecord written to SyncJournal**; shutdown Flush releases pending events

### Task 1.6: SyncJournal (SQLite)

**Files:** `internal/journal/journal.go`, `internal/journal/journal_test.go`, `internal/journal/schema.sql`
**Steps:**
1. SQLite table: `sync_operations` (id, event_id, entity_id, source_adapter, dest_adapter, status [begun/completed/failed/conflict], started_at, completed_at, error_msg, conflict_record JSON). **Second table:** `conflict_records` must include `status TEXT NOT NULL DEFAULT 'pending'` column (values: `pending`, `resolved`) — P1 fix: MCP `list-conflicts`/`resolve-conflict` require status filtering
2. Methods: `Begin(eventID, entityID, src, dst) error`, `Complete(eventID) error`, `MarkFailed(eventID, err) error`, `ResolveConflict(entityID, strategy, val1, val2, winner) ConflictRecord`
3. `ListPending() []SyncOp` — for retry on startup
4. `ListConflicts(status string, limit int) []ConflictRecord` — filters by status (P1 fix), limit 0 → default 100
5. **SQLite hardening** (P1 fix — applies to all three databases: journal, ancestor, dedup):
   - WAL mode enabled
   - `PRAGMA busy_timeout = 5000` — prevents SQLITE_BUSY on concurrent process access
   - `db.SetMaxOpenConns(1)` — SQLite is single-writer; serializes all writes, makes shutdown predictable
   - All public methods accept `context.Context` as first parameter, use `ExecContext`/`QueryContext` variants — allows cancellation during shutdown drain
**Test:** Begin+Complete round-trip; Begin+MarkFailed round-trip; ListPending after crash (Begin without Complete); restart persistence (close DB, reopen, verify data); ListConflicts filters by status; ListConflicts(limit=0) returns default count

### Task 1.7: AncestorStore (SQLite)

**Files:** `internal/ancestor/store.go`, `internal/ancestor/store_test.go`
**Steps:**
1. SQLite table: `ancestors` (entity_id PRIMARY KEY, content_hash, content BLOB, adapter_pair, updated_at)
2. `Get(ctx context.Context, entityID string) (Ancestor, error)`, `Put(ctx context.Context, entityID string, a Ancestor) error`, `Delete(ctx context.Context, entityID string) error`
3. Indexed by entity_id for fast lookup
4. **Restart persistence test** (fixes P0-1): store N ancestors, close store, reopen, verify all N retrievable with identical content
5. **SQLite hardening:** Same as Task 1.6 — WAL mode, `busy_timeout=5000`, `SetMaxOpenConns(1)`, context-aware queries
6. **Cross-store write ordering invariant** (P1 fix): When a three-way merge completes, `AncestorStore.Put` MUST be called BEFORE `SyncJournal.ResolveConflict`. If the process crashes between the two writes, the ancestor is stale but the journal shows no completed conflict — restart retries correctly. The reverse order (journal first) would leave a "resolved" conflict with a stale ancestor, causing incorrect merge base on next cycle. Document this invariant in `ConflictResolver` and enforce with an integration test that verifies ordering.
**Test:** Put+Get round-trip; restart persistence; concurrent Put from different goroutines (SQLite serialized writes); **cross-store ordering: ancestor updated before journal conflict record**

### Task 1.8: ConflictResolver with per-adapter-pair policies

**Files:** `internal/conflict/resolver.go`, `internal/conflict/resolver_test.go`, `internal/conflict/policy.go`
**Steps:**
1. `ConflictPolicy` per adapter pair: `beads+github` → beads authoritative for status, GitHub for PR state; `beads+notion` → beads for status, Notion for prose content; `notion+filesystem` → three-way merge with AncestorStore (add `MergeStrategy` field to Policy: `lww` or `three-way`); configurable in YAML (fixes P1-BSC-04)
2. `ResolvePair(a, b Event) (Event, ConflictRecord)` — applies policy. When `MergeStrategy == "three-way"`, fetches ancestor from AncestorStore and calls `ResolveThreeWay`. Returns winning event and a `ConflictRecord` value — **caller is responsible for persisting** the ConflictRecord to SyncJournal (keeps resolver testable without live DB)
3. `ResolveThreeWay(base, left, right) (merged, ConflictRecord, error)` — three-way merge. Uses full 32-byte SHA-256 digest for content comparison (P1 fix: 8-byte prefix truncation has collision risk). Returns `ErrUnresolvable` when field-level divergence cannot be auto-merged → dead-letter to SyncJournal. Field-level comparison even on convergent state (fixes supply-chain P0-5: same close-state, different close-notes → dead-letter)
4. LWW uses `interop receive-time` monotonic clock, NOT external timestamps (fixes P1-BSC-02)
5. **Post-merge ancestor update** (P1 fix — mandatory postcondition): After `ResolveThreeWay` returns a merged result, the caller MUST call `ancestorStore.Put(entityID, Ancestor{Content: merged, Hash: sha256(merged)})` BEFORE persisting the ConflictRecord to SyncJournal (per Task 1.7 cross-store ordering invariant). Failure to update the ancestor causes stale merge base → false conflicts on every subsequent sync cycle
6. **AncestorStore access for adapters** (P1 fix): Adapters that need three-way merge (Notion, GitHub filesync) receive `*ancestor.Store` at construction time via dependency injection. The bus-level dispatch path calls `ResolvePair` (which internally fetches the ancestor), so adapters do NOT call AncestorStore directly for conflict resolution — only the ConflictResolver does. Adapters may read ancestors for their own FullSync reconciliation logic.
**Test:** Per-pair policy selection; three-way merge produces correct output using full SHA-256 digest; field-level conflict on convergent state detected; LWW uses interop timestamp; **post-merge: ancestor updated before conflict record persisted**; **three-way merge with stale ancestor → correct re-merge (not false conflict)**

### Task 1.9: Daemon lifecycle + graceful shutdown

**Files:** `internal/daemon/daemon.go`, `internal/daemon/daemon_test.go`
**Steps:**
1. `Daemon` struct: holds EventBus, SyncJournal, AncestorStore, registered adapters, HTTP server (health + webhooks), MCP server
2. `Run(ctx context.Context) error` — start all components, block until context cancelled
3. Graceful shutdown via `signal.NotifyContext(ctx, syscall.SIGTERM, syscall.SIGINT)` — **sequenced drain** (P0 fix: goroutines must complete before stores close):
   1. Close HTTP listener (stop new webhooks)
   2. Call `bus.StopEmitting()` — reject new `Emit()` calls
   3. Stop all adapters (calls `a.Stop()` which closes `Emit()` channels)
   4. Wait for adapter dispatch goroutines to exit (Stage 1 WaitGroup — reads from `a.Emit()`)
   5. Call `collisionWindow.Flush()` — release pending non-colliding events for final dispatch
   6. Drain remaining entity channel events (Stage 2 consumer goroutines process buffered events)
   7. Wait for all `dispatchToAdapters` handling goroutines to complete (Stage 2 WaitGroup — P0 fix: these were previously untracked and could outlive journal.Close)
   8. `close(done)` — signal any remaining select loops
   9. Close SyncJournal and AncestorStore (safe: all goroutines that write to them have exited)
   10. Close MCP server
   11. Exit
   Each wait step has a configurable timeout (default 30s total across all waits). On timeout expiry: log unfinished goroutine count + pending entity keys, force exit.
4. Health endpoints: `/health` (liveness — process alive), `/ready` (readiness — all adapters started, each adapter's `Ready() <-chan struct{}` closed)
5. Structured JSON logging via `slog` with adapter context fields
6. MCP/webhook lifecycle coupling: if either server crashes, trigger graceful shutdown of both (fixes flux-review daemon insight)
**Test:** SIGTERM triggers orderly shutdown; in-flight events complete before exit (verify via SyncJournal: no `begun` entries without `completed` after shutdown — P1 fix: drain test must verify events actually complete); **journal.Close() not called until all handling goroutines exit** (P0 fix: inject a slow HandleEvent, verify it completes before journal closes); health returns 200 after start; ready returns 503 before all adapters ready, 200 after
