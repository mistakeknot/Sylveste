# Clavain Quality Gates

## Why This Journey Matters

Shipping code without review is fast but fragile. Shipping code through manual review is safe but slow. Quality gates automate the judgment call: which changes need which level of review? A one-line typo fix doesn't need five specialized agents poring over it. A security-sensitive authentication refactor needs every lens available.

Clavain's quality gates system (`/clavain:quality-gates`) auto-selects reviewer agents based on what changed, dispatches them in parallel, synthesizes findings, and produces a verdict. The developer gets the right amount of review — no more, no less — without manually choosing reviewers or reading raw agent output.

## The Journey

The developer finishes implementing a feature. Tests pass, linting clean. They run `/clavain:quality-gates`. The gate system analyzes the diff:
- What files changed? (scheduler, config, tests)
- What domains are affected? (concurrency, data access, user-facing CLI)
- How large is the change? (8 files, 900 insertions)
- Is anything security-sensitive? (no credentials, no auth changes)

Based on this analysis, the gate auto-selects relevant agents from the available fleet. Two agents always run: fd-architecture (module boundaries) and fd-quality (naming, idioms, language detection). Risk-based agents are added conditionally: fd-safety for auth/crypto/secrets changes, fd-correctness for migrations/concurrency, fd-performance for hot-path code. The gate caps at 5 agents total, prioritizing by risk when more are eligible — the triage log shows which agents were considered and why each was included or excluded. Project-specific agents generated by `/flux-gen` get a category bonus if relevant. Note: `/flux-gen` must be run at least once to generate project-specific agents; without them, only the core fleet is available.

Agents run in parallel, each writing to `{run-id}-{agent-name}.md` in `.clavain/quality-gates/` (using `.partial` suffix while in progress). Each produces findings tagged with severity levels: P0 (critical — blocks merge), P1 (important — should fix), P2 (suggestion), IMP (improvement — optional). Note: these P0-P2 labels are *finding severity* within a review, distinct from the CUJ-level criticality in frontmatter.

The synthesis is delegated to `intersynth:synthesize-review`, which deduplicates overlapping findings using a curated test corpus of known-duplicate patterns, merges consistent observations, and produces a compact verdict. The developer gets a single report:

```
Verdict: needs-changes
Gate: FAIL
P0: 0 | P1: 1 | P2: 2 | IMP: 0

Top findings:
- [P1] RankBeads sort is not stable; equal-priority beads may shuffle — fd-correctness (2/3 agents)
- [P2] Consider table-driven tests for boost clamping — fd-quality
```

The developer addresses the P1 finding (makes the sort stable), notes the P2 suggestions for later, re-runs the gate, and ships on PASS. The gate records the outcome — findings addressed, findings dismissed — feeding back into Interspect's agent performance tracking (fail-open: if Interspect is unavailable, the gate still works, but performance data is not recorded).

For simple changes (docs, config, small fixes), the gate may return: "Change is low-risk. Review optional." This behavior is user-configurable — developers can set thresholds for when review is mandatory vs. optional (e.g., "always review", "skip under N lines", "skip docs-only changes"). The default is smart skip.

When quality gates produce P0 or P1 findings on security-sensitive code, the gate suggests escalating to `/interpeer` for model-diverse validation. Interpeer provides a non-Claude perspective (GPT-5.2 via Oracle, or Codex), catching blind spots that same-model review misses. This is an optional escalation, not part of the default flow.

## Success Signals

| Signal | Type | Assertion |
|--------|------|-----------|
| Agent selection matches change type | qualitative | Selected agents are relevant to the domains touched; triage log shows inclusion/exclusion reasoning |
| Parallel dispatch completes in under 90 seconds | measurable | All agents return findings within 90s |
| Synthesis deduplication verified against test corpus | measurable | Curated corpus of known-duplicate findings correctly merged; distinct findings preserved |
| Critical findings caught by injection test | measurable | Known-injected critical issues are detected; injection harness runs as part of gate CI |
| Low-risk changes get proportionally less review | measurable | Single-file doc changes trigger ≤2 agents (configurable threshold) |
| Findings are actionable (not vague) | qualitative | Developer can fix the issue from the finding description alone |
| Gate outcome feeds Interspect tracking | measurable | Agent performance data updated after each gate run (fail-open) |

## Known Friction Points

- **Agent quality varies** — some agents produce consistently useful findings, others are noisy. Interspect tracking helps but calibration takes time. Workaround: `/interspect:interspect-correction` to record agent errors as high-quality evidence.
- **Synthesis can merge too aggressively** — distinct issues in the same domain may be merged into one finding, losing nuance. Mitigation: intersynth preserves both when conflicting recommendations exist.
- **Interpeer is expensive** — cross-AI review via Oracle uses significant tokens and requires Oracle infrastructure (Xvfb, Chrome). Not practical for every change — reserved for P0/P1 findings on security-sensitive code.
- **Project agents need generation** — `/flux-gen` must be run first to generate project-specific agents. Without them, only generic agents are available. Future: auto-detect and suggest generation on first gate run.
