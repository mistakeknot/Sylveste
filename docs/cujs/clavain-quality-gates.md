# Clavain Quality Gates

## Why This Journey Matters

Shipping code without review is fast but fragile. Shipping code through manual review is safe but slow. Quality gates automate the judgment call: which changes need which level of review? A one-line typo fix doesn't need five specialized agents poring over it. A security-sensitive authentication refactor needs every lens available.

Clavain's quality gates system (`/clavain:quality-gates`) auto-selects reviewer agents based on what changed, dispatches them in parallel, synthesizes findings, and produces a verdict. The developer gets the right amount of review — no more, no less — without manually choosing reviewers or reading raw agent output.

## The Journey

The developer finishes implementing a feature. Tests pass, linting clean. They run `/clavain:quality-gates`. The gate system analyzes the diff:
- What files changed? (scheduler, config, tests)
- What domains are affected? (concurrency, data access, user-facing CLI)
- How large is the change? (8 files, 900 insertions)
- Is anything security-sensitive? (no credentials, no auth changes)

Based on this analysis, the gate auto-selects relevant agents from the available fleet: fd-architecture (module boundaries), fd-correctness (data consistency, race conditions), fd-quality (naming, idioms, test coverage). It skips fd-safety (no security-sensitive changes) and fd-game-design (not a game). Project-specific agents generated by `/flux-gen` get a category bonus if relevant.

Agents run in parallel. Each produces findings with severity (critical, important, suggestion, nitpick). The synthesis layer deduplicates overlapping findings, merges consistent observations, and ranks by severity. The developer gets a single report:

```
Verdict: APPROVE with suggestions (3 findings)

Critical: 0
Important: 1 — fd-correctness: RankBeads sort is not stable; equal-priority beads may shuffle between calls
Suggestion: 2 — fd-quality: consider table-driven tests for boost clamping
```

The developer addresses the important finding (makes the sort stable), notes the suggestions for later, and ships. The gate records the outcome — findings addressed, findings dismissed — feeding back into Interspect's agent performance tracking.

For simple changes (docs, config, small fixes), the gate may return: "Change is low-risk. Review optional." The developer can skip review or run it anyway.

For cross-AI second opinions, `/clavain:interpeer` escalates to a different AI provider (GPT-5.2 via Oracle, or Codex) for an independent review. This is the nuclear option — expensive but catches blind spots that same-model review misses.

## Success Signals

| Signal | Type | Assertion |
|--------|------|-----------|
| Agent selection matches change type | qualitative | Selected agents are relevant to the domains touched |
| Parallel dispatch completes in under 90 seconds | measurable | All agents return findings within 90s |
| Synthesis deduplicates overlapping findings | measurable | No two findings describe the same issue |
| Critical findings are never missed | measurable | Known-injected critical issues are caught |
| Low-risk changes get proportionally less review | measurable | Single-file doc changes trigger ≤2 agents |
| Findings are actionable (not vague) | qualitative | Developer can fix the issue from the finding description alone |
| Gate outcome feeds Interspect tracking | measurable | Agent performance data updated after each gate run |

## Known Friction Points

- **Agent quality varies** — some agents produce consistently useful findings, others are noisy. Interspect tracking helps but calibration takes time.
- **Synthesis can merge too aggressively** — distinct issues in the same domain may be merged into one finding, losing nuance.
- **No incremental review** — gates review the full diff, not just the delta since last review. Re-running gates after addressing findings re-reviews everything.
- **Interpeer is expensive** — cross-AI review via Oracle uses significant tokens and requires Oracle infrastructure (Xvfb, Chrome). Not practical for every change.
- **Project agents need generation** — `/flux-gen` must be run first to generate project-specific agents. Without them, only generic agents are available.
