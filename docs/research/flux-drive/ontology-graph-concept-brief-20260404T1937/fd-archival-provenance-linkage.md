### Summary

The concept brief faces the same fundamental challenge archivists solved with multi-level description and authority control: how to create a unified discovery layer over heterogeneous collections without violating the principle that each collection's internal arrangement is sovereign. The brief correctly identifies the "unify retrieval, not storage" constraint from PHILOSOPHY.md but does not carry this constraint through to concrete design implications. Three structural gaps emerge from archival science: (1) the entity graph is conceived as a flat topology rather than a hierarchical description with levels that carry semantic meaning, (2) authority control and entity resolution are conflated rather than treated as distinct operations with different cost profiles, and (3) the brief lacks the archival insight that description granularity should be non-uniform — reference entities need full description while detail entities need only stub entries with pointers.

### Issues Found

APL-1. **P1: Flat entity graph loses hierarchical context** — The "Unified Entity Graph" capability (lines 48-49) describes "beads, sessions, agents, artifacts, discoveries all linked" but does not address how the graph preserves the hierarchical nesting that gives entities their contextual meaning. In archival science, a document is described within its series within its fonds — removing the hierarchical context makes the document less discoverable, not more. The Sylveste equivalent: a function exists within a module within a pillar; a bead exists within an epic within a sprint. A flat graph that links a function directly to a bead without preserving the module-epic nesting produces "show me everything related to X" results that are overwhelming and lack the contextual scaffolding that makes them interpretable. **Failure scenario**: An agent querying "everything related to `routing_resolve_agents`" gets back 200 flat nodes (beads, sessions, commits, test results, discoveries) with no hierarchical grouping — the agent cannot distinguish between the function's architectural context (Clavain > routing > model-selection) and its incidental associations (sessions that happened to call it), making the query result less useful than grepping the codebase directly.

**Recommended fix**: The concept brief should specify that the ontology graph preserves multi-level description: entities are grouped into hierarchical containers (pillar > module > file > function; epic > sprint > bead) and graph traversal respects these levels by default, with "flatten" as an explicit opt-in for cross-hierarchy queries.

APL-2. **P1: Authority control vs. entity resolution conflated** — The brief mentions "Cross-system entity resolution" (implicit in Capability 1, line 48) without distinguishing between two fundamentally different operations that archivists keep rigorously separate. Authority control creates canonical entity records — "this entity exists, here is its authoritative description, here are its identifiers across systems." Entity resolution matches ad-hoc references to those canonical records — "this string in a session log probably refers to that canonical entity." Conflating these means either (a) the ontology requires all subsystems to register entities through a canonical interface (authority control without resolution — brittle, high adoption friction) or (b) the ontology infers all entity identities from observation (resolution without authority — produces false matches that accumulate over time). **Failure scenario**: Without canonical entity records, the ontology resolves "fd-architecture" in an interspect evidence file and "fd-architecture" in a flux-drive triage table as the same entity by string matching. When the agent name changes or when two different agents share a name across plugin versions, the resolution silently breaks — every historical relationship attributed to the old entity is now attributed to the new one, corrupting the graph without any error signal.

**Recommended fix**: Add a section on "Entity Registration" that distinguishes: (a) canonical entity records maintained by each subsystem (authority files — each plugin declares its entities), and (b) a resolution layer that matches observed references to canonical records with scored confidence, surfacing low-confidence matches as candidates rather than assertions.

APL-3. **P2: No granularity tiers — uniform description depth assumed** — The "Minimal Viable Ontology" question (line 67) asks "which entity types and relationships would provide 80% of the value" but does not address granularity within entity types. Archival science learned through decades of practice (Greene & Meissner 2005, "More Product, Less Process") that item-level description is unsustainable — coarse description at the series/collection level delivers most of the discovery value. The concept brief lists "Files, functions, classes, modules, tests, dependencies, imports, AST nodes" (line 21) as development entities without distinguishing between reference entities (modules, plugins — stable, frequently queried) and detail entities (AST nodes, individual tool calls — volatile, rarely queried directly). Loading both into the same graph layer at the same granularity produces millions of fine-grained nodes that make traversal slow and results noisy.

**Recommended fix**: Define three description tiers: (1) reference entities (plugins, agents, epics, modules) — always in the graph, fully described; (2) aggregate entities (files, beads, sessions) — in the graph with summary metadata, detail on demand; (3) detail entities (AST nodes, tool calls, individual commits) — indexed on demand with TTL, not permanent graph residents.

APL-4. **P2: No crosswalk mechanism between existing graph structures** — The brief lists six existing graph-like structures (lines 33-39) but does not describe how entities would be mapped between them. Archivists routinely maintain formal crosswalks between descriptive standards (MARC to EAD, Dublin Core to ISAD(G)) — explicit mapping tables that declare "this field in system A corresponds to this field in system B, with these transformations." Without crosswalks, the ontology must either require all subsystems to adopt a single canonical schema (which PHILOSOPHY.md forbids) or allow unconstrained string labels that defeat the purpose of typed schema.

**Recommended fix**: Add a "Schema Crosswalks" section describing how the ontology maps between subsystem-native schemas (beads issue schema, intercore run schema, cass session schema) using explicit declared mappings rather than schema convergence.

APL-5. **P1: Typed Schema + Actions creates implicit storage migration** — Capability 2 (lines 51-52) proposes "Formal type system for platform entities (Agent, Tool, Skill, Run, Artifact) with declared capabilities, constraints, and allowed actions." This sounds like a projection layer but operationally requires each subsystem to conform to the ontology's type declarations — if a plugin's "Skill" entity doesn't match the ontology's Skill type definition, it either fails validation or is silently misrepresented. This violates the archival principle of respect des fonds: each subsystem's internal descriptive practices should be sovereign. The "Actions" component (preconditions and effects declared in the schema) is particularly concerning — it implies the ontology prescribes what operations are valid on entities, shifting from observation to governance.

**Recommended fix**: Reframe Capability 2 as "Reference Profiles" rather than "Typed Schema" — each subsystem publishes a self-described profile of its entity types using its own vocabulary, and the ontology indexes these profiles for discovery without requiring conformance to a central type system. Actions remain the subsystem's responsibility, not the ontology's.

APL-6. **P2: No entity lifecycle model for dynamic entities** — The brief does not address entity lifecycle management. Archives have formal accession (adding material with provenance records) and deaccessioning (removing material while maintaining provenance). The ontology graph will ingest sessions, runs, and tool calls at high volume — without a lifecycle model, the graph accumulates unbounded stale nodes. A session from 6 months ago that touched a now-deleted file creates a zombie relationship that pollutes traversal results.

**Recommended fix**: Define entity lifecycle states (active, archived, tombstoned) with retention policies per entity type. Archived entities retain their relationships but are excluded from default traversals. Tombstoned entities are removed from the graph but leave provenance records of their former existence.

### Improvements

1. **Finding aids as projections** — The concept brief's "read-only projection" idea (line 68) maps directly to the archival concept of a finding aid: a document that describes a collection's contents without being the collection itself. Frame the ontology explicitly as a "finding aid generator" — each subsystem's entities are described by a finding aid that the ontology indexes, and the ontology's unified query is a union search across finding aids. This reframe makes the non-invasive nature of the projection layer concrete and operationally testable: if deleting the ontology graph leaves every subsystem fully functional, the design is correct. (fd-archival-provenance-linkage)

2. **Authority files as the connective tissue** — Rather than building entity resolution from scratch, adopt the archival authority file pattern: a shared registry of canonical entity identifiers (agents, plugins, developers) that multiple subsystems reference without duplicating. Each authority record has one canonical owner (the subsystem that creates the entity) and zero or more consumers (subsystems that reference it). This is lighter than a full ontology and directly addresses the "show me everything connected to this function" query without requiring cross-system schema convergence. (fd-archival-provenance-linkage)

3. **Respect des fonds as a design test** — Add a design principle: "The ontology must never require a subsystem to change its internal data model to be discoverable." This is the operational translation of respect des fonds and the PHILOSOPHY.md constraint. Use it as a litmus test for every proposed ontology feature: if implementing feature X requires beads to add a field, or cass to change its session schema, the feature violates the principle. (fd-archival-provenance-linkage)
