Verdict: needs-changes

### Summary

The concept brief's typed schema proposal faces the same challenge that clinical terminology systems solved over forty years: how to define a type system rigorous enough for machine reasoning but flexible enough to accommodate the reality that each subsystem describes the same concepts differently. Three patterns from health informatics directly apply. First, SNOMED CT's pre-coordination vs. post-coordination distinction warns that defining a schema entry for every possible entity-relationship-action combination produces combinatorial explosion — the ontology needs compositional types, not exhaustive enumeration. Second, FHIR's binding strength model (required/extensible/preferred/example) provides a proven framework for the formality gradient the ontology needs — some entity types demand strict typing while others need loose, extensible schemas. Third, the ongoing cost of maintaining cross-terminology maps (ICD-10 to SNOMED, LOINC to local codes) is the most expensive part of any terminology system — the concept brief does not account for this crosswalk maintenance burden.

### Issues Found

CTH-1. **P1: Pre-coordination trap — combinatorial schema explosion** — Capability 2 (lines 51-52) proposes "Formal type system for platform entities (Agent, Tool, Skill, Run, Artifact) with declared capabilities, constraints, and allowed actions." SNOMED CT discovered that pre-coordinating every possible concept — creating a dedicated node for every combination of finding, site, severity, and laterality — produces an unmanageable concept space. The ontology faces the same problem: if every combination of entity type (Agent, Tool, Skill, Run, Artifact, plus the 6 categories on lines 20-27), relationship type (blocks, parent-child, triggered, references, authored-by, etc.), and action type (create, close, dispatch, merge, etc.) needs a pre-defined schema entry, the schema grows combinatorially. With 20 entity types, 15 relationship types, and 10 action types, the exhaustive schema has 3,000 entries — most of which will never be queried. **Failure scenario**: A new plugin introduces a "recommendation" entity type with a "derived-from" relationship to discoveries and a "validate" action. Adding this to the ontology requires defining the entity schema, registering the relationship type, declaring the action's preconditions and effects, and updating all consumers that enumerate entity/relationship/action types. The friction discourages the plugin author from registering at all, so the ontology becomes incomplete.

**Recommended fix**: Adopt SNOMED CT's post-coordination model. Define atomic building blocks (entity types, relationship types, action types) independently, and compose them at query time. An agent asks "show me all entities of type Recommendation that have a derived-from relationship to any Discovery" — the ontology evaluates this compositionally without needing a pre-defined "Recommendation-derived-from-Discovery" schema entry.

CTH-2. **P1: No formality gradient across entity types** — The brief treats all entity types with the same schema rigor: "Typed objects. Every entity has a declared type... with properties and constraints" (line 9). Clinical informatics learned that different data elements need different binding strengths. FHIR defines four levels: **required** (must use this exact value set — e.g., patient gender), **extensible** (should use this value set, may add codes), **preferred** (recommended value set, alternatives acceptable), **example** (illustrative only). The Sylveste ontology needs the same gradient. Core types like Agent, Plugin, and Bead have stable, well-defined schemas with required fields — binding strength should be "required." Peripheral types like Discovery, Learning, and Pattern are inherently fuzzy, evolve rapidly, and have different shapes across subsystems — binding strength should be "extensible" or "preferred." A single formality level across all types either over-constrains the fuzzy types (plugins stop registering discoveries because the schema doesn't fit) or under-constrains the stable types (agent entity records lack the fields needed for reliable routing). **Failure scenario**: The ontology defines a strict "Discovery" schema with required fields (source, confidence, evidence). But interject discoveries have different fields than interknow learnings, and interspect evidence records have yet another shape. Each subsystem must either (a) transform its data to fit the canonical Discovery schema (losing information) or (b) stuff extra data into a generic "metadata" blob (defeating the purpose of typed schema). Over time, the "metadata" blob becomes the actual schema, and the typed fields are ignored.

**Recommended fix**: Define a binding strength for each entity type and each field within that type. Core entity types (Agent, Plugin, Bead, File) get "required" binding with strict schemas. Peripheral entity types (Discovery, Learning, Pattern) get "extensible" binding — the ontology provides a recommended schema but subsystems may add fields and omit optional ones. Query results for extensible types carry a completeness indicator: "this entity matched 4/7 recommended fields."

CTH-3. **P1: No freshness metadata — stale graph indistinguishable from empty graph** — The brief describes the ontology as a "read-only projection" (line 68) but does not address temporal validity. SNOMED CT publishes biannual releases; every concept carries a version and effective date. FHIR resources carry `meta.lastUpdated`. The ontology graph's relationships are populated by indexing subsystem stores, but indexing is asynchronous — a new session in cass may not appear in the graph for minutes or hours. Without per-entity freshness metadata, an agent querying "which sessions touched this file?" cannot distinguish between "no sessions touched this file" and "the sessions haven't been indexed yet." **Failure scenario**: An agent uses the ontology to check whether a file has been recently modified in any session before making an edit. The graph returns no results because cass indexing is 30 minutes behind. The agent proceeds with the edit, conflicting with work in a session that the graph didn't yet reflect. The conflict is undetected because the graph presented its incomplete state as authoritative.

**Recommended fix**: Every graph query result must include per-subsystem freshness metadata: `{subsystem: "cass", last_indexed: "2026-04-04T19:20:00Z", indexing_lag: "~15min"}`. Agents can then make informed decisions: "cass data is 15 minutes stale — fall back to direct `cass search` for real-time results."

CTH-4. **P2: Closed relationship type system creates extension friction** — The brief lists specific relationship types (line 14: "authored_by, blocked_by, triggered") suggesting a fixed enumeration. Clinical terminology systems learned that closed value sets create coordination costs when new concepts arise. ICD-10's fixed code set requires international committee approval for new codes — a process that takes years. If the ontology's relationship types are a closed enum, a new subsystem that introduces "inspired-by" (interlens), "drift-detected" (interwatch), or "cost-correlated" (interstat) relationships must go through a central schema migration. This friction discourages subsystem authors from expressing their domain-specific relationships, producing an ontology that covers only the relationships that existed at design time.

**Recommended fix**: Make relationship types an open, namespaced registry: `interlens:inspired-by`, `interwatch:drift-detected`, `interstat:cost-correlated`. Subsystems register their relationship types with a definition and query resolver. The ontology discovers new types automatically. Core relationship types (blocks, parent-child, references) are in the `core:` namespace and guaranteed stable.

CTH-5. **P2: No temporal validity for evolving schemas** — The brief acknowledges that "A code ontology changes every commit" (line 61) but does not address how schema evolution affects historical relationships. When a plugin renames an entity field, deprecates a relationship type, or splits one entity type into two, existing graph relationships become semantically ambiguous. SNOMED CT handles this with concept versioning: deprecated concepts are marked inactive but retained with a reason code (duplicate, outdated, erroneous) and a mapping to the replacement concept. Without this, the ontology graph silently accumulates historical relationships that use superseded schemas — queries return data that looks valid but whose semantics have shifted.

**Recommended fix**: Entity type definitions carry a version. When a type evolves, the old version is deprecated-not-deleted, and a migration mapping is declared ("v1.field_x maps to v2.field_y"). Historical relationships carry the schema version under which they were asserted. Agents can filter by schema version or accept the ontology's automatic migration.

CTH-6. **P2: No interface terminology — agents must speak the ontology's formal language** — Clinical informatics distinguishes between interface terminologies (the terms clinicians use in the UI: "heart attack") and reference terminologies (the formal codes in the backend: "SNOMED 22298006 - Myocardial infarction"). The concept brief's Capability 3 describes "Agent-Queryable Relationships" (lines 53-54) but does not address how agents express queries. If agents must use the ontology's formal type names and relationship labels ("MATCH (s:Session)-[:MODIFIED]->(f:File)"), the ontology becomes a specialized query language that every agent must learn. If agents use natural language ("which sessions modified this file?"), the ontology needs a mapping layer from natural-language terms to formal graph constructs.

**Recommended fix**: Build an interface terminology layer: a mapping from natural-language query patterns ("sessions that touched X", "who worked on X", "what depends on X") to formal graph traversals. Agents query in natural language; the ontology translates. This is how FHIR's search parameters work — each resource type declares named search parameters ("patient", "date", "status") that map to specific FHIRPath expressions.

### Improvements

1. **FHIR's "resources with references" as the design pattern** — FHIR solved the same problem at internet scale: heterogeneous health data across thousands of independent systems, each with its own vocabulary, unified through typed resources with typed references between them. The key insight was that resources are self-contained (you can understand a Patient resource without reading the referenced Organization) but references enable traversal (you can follow the reference to get the Organization if needed). This maps directly to the ontology graph: each subsystem's entities are self-contained, but the ontology adds typed references between them that enable cross-system traversal without requiring subsystems to embed each other's data. (fd-clinical-terminology-harmonization)

2. **Value set expansion as the query model** — FHIR's `$expand` operation on value sets — "give me all concepts that are children of Lung Disease" — provides a model for ontology queries. Rather than requiring agents to know the exact entity types, the ontology supports subsumption queries: "show me all entities that are a type of Infrastructure" expands to Plugin, Skill, Command, Hook, MCP Tool, Agent, Configuration. This requires a shallow type hierarchy (2-3 levels maximum) with is-a relationships between entity types. (fd-clinical-terminology-harmonization)

3. **ConceptMap as the crosswalk primitive** — FHIR's ConceptMap resource provides a formal, versionable, directional mapping between concept systems. Each mapping entry declares source code, target code, equivalence type (equivalent, wider, narrower, inexact), and a human-readable comment. Adopting this as the crosswalk primitive for the ontology means every mapping between subsystem schemas is explicit, versioned, and queryable — "how does beads' issue.status map to intercore's run.state?" returns a ConceptMap with declared equivalences and known gaps. (fd-clinical-terminology-harmonization)
