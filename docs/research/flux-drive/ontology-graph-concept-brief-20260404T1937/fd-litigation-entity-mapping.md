Verdict: risky

### Summary

E-discovery practitioners build entity relationship maps under the harshest conditions imaginable: terabytes of heterogeneous data from custodians who never intended their data to be connected, under legal hold obligations that make both over-connection and under-connection consequential, within weeks rather than months. The concept brief's ontology graph faces a milder version of these same challenges but is missing the defensive patterns that litigation support takes for granted. The most critical gap is provenance: in litigation, every entity link must be defensible — a lawyer must be able to explain why two documents are connected, what evidence supports the connection, and at what confidence level. The concept brief proposes cross-system entity relationships with no provenance model, meaning agents will traverse links they cannot audit and make routing decisions based on graph assertions whose confidence is unknown. This is a P0 finding because it directly enables the failure mode where incorrect entity resolution propagates through the dispatch system. The brief also lacks a proportionality analysis, person-entity deduplication strategy, and iterative enrichment model — all of which are standard practice in litigation entity analytics.

### Issues Found

LEM-1. **P0: Cross-system entity links lack provenance metadata** — Capability 1 (lines 48-49) proposes linking "beads, sessions, agents, artifacts, discoveries" into a unified graph, and Capability 3 (lines 53-54) enables agents to "traverse relationships at runtime." Neither capability addresses how relationships carry provenance. In litigation, the Federal Rules of Evidence require that every connection between entities be supported by evidence — an email thread, a timestamp overlap, a shared custodian. The Relativity analytics platform stores provenance for every computed link: `{method: "email-threading", confidence: 0.92, evidence: ["msg-id-match", "header-chain"], source_system: "exchange"}`. Without equivalent provenance in the ontology graph, an agent traversing "this function was modified in session X, which had a P0 failure" cannot verify whether the session-to-function link is (a) a confident assertion from git blame, (b) a temporal co-occurrence inference from cass, or (c) a speculative match from string similarity. **Failure scenario**: The ontology links bead `Sylveste-a3f` to session `abc-123` based on temporal overlap (both active at the same time). An agent queries the graph and concludes this session "worked on" the bead. Based on this, the agent inherits the session's failure history and deprioritizes further work on the bead. But the temporal overlap was coincidental — the session was working on a different bead entirely. The false link propagated through the dispatch system, causing the bead to be incorrectly deprioritized for multiple sprints. No one detects the error because the link has no provenance metadata to audit.

**Recommended fix**: Every cross-system relationship must carry: `{source_subsystem: string, method: "explicit-reference|temporal-cooccurrence|identifier-match|causal-chain|string-similarity", confidence: "confirmed|probable|speculative", evidence: [string], created_at: timestamp}`. Agents MUST be able to filter query results by confidence level: `--min-confidence=probable` excludes speculative links. Low-confidence matches are surfaced as candidates rather than asserted as facts. This is the single most important design requirement for the ontology graph — without it, the graph is an authoritative-looking source of unauditable assertions.

LEM-2. **P1: No proportionality analysis** — The concept brief asks "What's the minimal viable ontology — which entity types and relationships would provide 80% of the value?" (line 67) but provides no quantitative framework for answering this question. E-discovery law (Federal Rules of Civil Procedure, Rule 26(b)(1)) requires that discovery efforts be "proportional to the needs of the case" — not every document in every custodian's collection is reviewed. The litigation equivalent of the ontology question is: "Which custodians and document types do we need to process to find 80% of the relevant evidence?" This is answered by sampling: process a small random sample, measure the hit rate per custodian and document type, then scale up collection for high-yield sources. **Failure scenario**: Without proportionality analysis, the ontology team indexes all 6 entity categories at full granularity. The indexing takes months. When usage data finally arrives, it shows that 90% of agent queries only traverse beads-to-sessions and sessions-to-files — two relationship types across three entity types. The remaining 3+ categories and 10+ relationship types were indexed at significant cost but are rarely queried.

**Recommended fix**: Before building the ontology, instrument the ecosystem to measure what cross-system queries agents actually attempt (and fail). Run a 2-week shadow experiment: intercept agent queries that would benefit from cross-system traversal (e.g., agent asks "what sessions modified this file?" and manually runs `cass context <file>`), log the query patterns, and rank entity types and relationships by query frequency. Build the ontology's first phase to cover only the top-3 query patterns, then expand based on demand.

LEM-3. **P1: Person-entity resolution not addressed as a first-class concern** — The brief lists "Agent entities" (line 23) but does not address the person-entity resolution problem that litigation analytics considers foundational. In litigation, the same person appears across data sources under different identifiers: personal email, work email, phone number, Slack handle, document author metadata. E-discovery platforms build identity resolution as their first step — before any relationship analysis. In Sylveste, the same developer appears as: a GitHub username in commits, a `claimed_by` value in beads, a session UUID in cass, a `CLAUDE_SESSION_ID` in hook state, and an agent name in fleet-registry. **Failure scenario**: An agent queries "what has developer X worked on?" The ontology returns results only from the subsystem whose identifier was used in the query. Results from other subsystems where the same developer appears under a different identifier are invisible. The "developer-centric view" that motivated the ontology remains fragmented.

**Recommended fix**: Build identity resolution as a first-class graph primitive. Maintain an identity mapping table: `{person: "mk", identifiers: [{system: "git", id: "mk"}, {system: "beads", id: "mk"}, {system: "cass", id: "session-xxx"}, {system: "fleet", id: "opus-4"}]}`. The mapping is initially populated from explicit declarations (git config, beads config) and progressively enriched by observation (temporal co-occurrence of identifiers in the same work context). Identity mappings are auditable and correctable — a false identity merge can be split.

LEM-4. **P1: No iterative enrichment model — big-bang indexing assumed** — The brief implicitly assumes the ontology is built by comprehensively indexing all subsystem stores. Litigation entity maps are built iteratively: Phase 1 is broad collection with keyword search (cheap, covers all custodians at low resolution). Phase 2 is concept clustering (identifies document groups that are likely relevant). Phase 3 is targeted review (human reviewers add entity detail and relationship confidence to high-value document clusters). Each iteration adds entity detail and relationship confidence where it matters most. **Failure scenario**: The ontology team builds a comprehensive indexer that processes all 6 subsystem stores in a single pass. The initial indexing takes 3 hours. During the indexing period, the graph is either empty or misleadingly incomplete. Agents fall back to direct subsystem queries, establishing usage patterns that persist after the graph is ready. By the time the graph is fully indexed, agents have already built workflows around direct subsystem access and see no reason to switch.

**Recommended fix**: Design for iterative enrichment. Phase 1: create coarse entity stubs for all subsystems cheaply (entity_type, entity_id, subsystem, created_at — 4 fields, fast harvest). Phase 2: add relationship stubs between entities that co-occur in the same work context (same bead, same session, same file). Phase 3: deep-index specific entities on demand when an agent queries for detail the stub doesn't provide. Each phase delivers immediately usable query results — Phase 1 alone enables "does this entity exist in any subsystem?" queries.

LEM-5. **P2: Near-duplicate entity detection not addressed** — The same logical entity frequently appears in multiple subsystems with slightly different representations. A "routing model calibration" theme appears as a bead title, a discovery in interject, a solution doc in interknow, a lens in interlens, and a section in PHILOSOPHY.md. Litigation data processing dedicates significant effort to exact deduplication (MD5 matching) and near-duplicate detection (MinHash/SimHash) to avoid counting the same evidence multiple times. Without equivalent deduplication, the ontology graph will contain multiple nodes for the same logical entity, inflating relationship counts and making convergence analysis unreliable.

**Recommended fix**: Implement two-tier deduplication: (a) exact dedup via identifier matching within each subsystem (same bead ID = same bead), and (b) cross-system near-dedup via semantic similarity of entity titles/descriptions with a configurable threshold. Near-duplicates are linked as "probable-same-entity" relationships rather than silently merged — the graph presents them as candidates for human confirmation.

LEM-6. **P2: No access control on relationship traversal** — The brief's Capability 3 assumes all cross-system relationships are universally traversable. In litigation, some relationships are privileged (attorney-client communication) and must be withheld from production even though they exist in the graph. The Sylveste equivalent: an agent querying the graph should not necessarily be able to traverse from a bead to another agent's session details (exposing that agent's reasoning and tool usage). Multi-agent systems create implicit trust boundaries that the ontology's cross-system traversal could violate.

**Recommended fix**: Define relationship-level access control. Each relationship carries an access scope: `{scope: "public|session|agent|admin"}`. Agents querying the graph see only relationships at or below their access level. Public relationships (bead-to-file, plugin-to-skill) are universally visible. Session-scoped relationships (session-to-tool-calls) are visible only within the same session. Agent-scoped relationships (agent-to-calibration-data) are visible only to the agent that created them.

### Improvements

1. **Matter-centric model as the query paradigm** — Litigation organizes everything around a "matter" — the central entity to which all other entities relate. The Sylveste equivalent is the bead: everything connects to a work item. Design the ontology's primary query pattern around bead-centric traversal: given a bead, show me all sessions that worked on it, all files they touched, all commits they produced, all discoveries they made, all review findings they generated. This bead-centric view is the highest-value query pattern and should be the first one the ontology supports. It directly serves the "show me everything related to X" goal and aligns with the beads-as-source-of-truth philosophy. (fd-litigation-entity-mapping)

2. **Confidence scoring as a first-class query dimension** — In Relativity, every computed relationship has a confidence score that lawyers use to prioritize review: high-confidence links are reviewed first, low-confidence links are reviewed if time permits. The ontology should expose confidence as a query filter. An agent asking "what's related to this function?" gets back high-confidence results (git blame, explicit bead references) immediately and can optionally expand to low-confidence results (temporal co-occurrence, string similarity) if the high-confidence results are insufficient. This prevents the graph from overwhelming agents with speculative connections while still capturing weak signals for when they're needed. (fd-litigation-entity-mapping)

3. **Proportionality-driven phased rollout** — Rather than building the complete ontology and hoping agents use it, apply the litigation proportionality principle: build the minimum graph that serves the highest-value queries, measure adoption, and expand based on demand. Phase 1: beads-sessions-files triangle (the most common cross-system query). Phase 2: add agent-session relationships (enabling "who worked on this?"). Phase 3: add discovery-bead-file relationships (enabling "what was learned from this work?"). Each phase is independently valuable and independently measurable. If Phase 1 adoption is low, don't build Phase 2 — diagnose why. (fd-litigation-entity-mapping)
