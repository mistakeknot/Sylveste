### Summary

Spatial data infrastructure architects have spent three decades solving the exact problem the concept brief describes: building a queryable registry over independently maintained data stores where each data producer is an autonomous agency with its own schema, update cycle, and quality standards. The solution that survived contact with reality was the catalog-of-catalogs pattern — a registry that indexes metadata about features in agency data stores, rather than centralizing the data. The concept brief has not yet committed to this pattern or its alternative. Three critical SDI lessons apply: (1) the discovery threshold — the minimum metadata each subsystem must publish for the graph to be useful — must be defined and kept low enough for adoption; (2) identifier harmonization across subsystems requires a cross-reference mapping layer, not a unified ID namespace; and (3) the graph must deliver value at partial coverage, not require all subsystems to be indexed before any queries work. The "unify retrieval, not storage" principle from PHILOSOPHY.md is the correct starting point but needs operational detail.

### Issues Found

SDI-1. **P1: Catalog-of-catalogs vs. data warehouse not distinguished** — The "Unified Entity Graph" capability (lines 48-49) describes "One place to ask 'show me everything related to X'" but does not specify whether the graph stores entity data (warehouse) or metadata about entities with pointers to source stores (catalog). SDI architecture learned this distinction the hard way: Australia's national SDI (ASDI) initially attempted to warehouse state government geospatial data into a central repository, then abandoned the approach when agencies evolved their schemas independently and the warehouse fell permanently out of sync. The INSPIRE Directive in Europe adopted the catalog approach: each member state publishes metadata records to a shared catalog service (CSW), and the catalog returns pointers to authoritative data in national stores. **Failure scenario**: The ontology graph replicates beads issue data into its graph store. Beads evolves its schema (adds a `claimed_at` state field, renames `priority` to `urgency`). The graph's replicated data is now schema-inconsistent with the source. During the stale period (which grows longer with each subsystem release), agents querying the graph get data that looks valid but uses an outdated schema — `priority` still shows a value while the authoritative source now uses `urgency`. The graph has become a source of systematically incorrect data with no error signal.

**Recommended fix**: Commit explicitly to the catalog-of-catalogs pattern. The ontology graph stores: (a) entity type metadata (what types exist, what fields they have, what relationships connect them), (b) entity discovery records (this entity exists, here is its type, here is its subsystem, here is its native ID), and (c) relationship stubs (entity A relates to entity B via relationship R, with provenance). Full entity data retrieval is delegated to the source subsystem via its native API. The graph never caches entity field values.

SDI-2. **P1: No minimum metadata threshold for participation** — The brief asks "What's the minimal viable ontology?" (line 67) but frames this as which entity types to include, not what metadata each subsystem must provide per entity. ISO 19110 feature catalogs define a minimum metadata set for feature types: name, definition, attribute list, relationship list. This minimum is deliberately low — an agency can participate in the SDI by publishing a feature catalog even if its data quality is poor. Without an equivalent discovery threshold for the ontology, subsystem developers face an undefined cost: "How much work do I need to do before my plugin's entities appear in the graph?" If the answer is "implement a full entity registration API with typed schemas, relationship declarations, and action definitions" (as Capability 2 implies), many plugins will never participate. **Failure scenario**: The ontology defines a comprehensive entity registration interface. Of 60+ plugins, 5 invest the effort to register their entities. The graph covers 8% of the ecosystem. The "show me everything related to X" query returns results from 5 subsystems and silently omits 55 others. Users conclude the graph is useless because it misses most of the ecosystem. The graph is abandoned not because the design was wrong but because the adoption threshold was too high.

**Recommended fix**: Define a minimum discovery threshold — the absolute minimum metadata a subsystem must provide: `{entity_type: string, entity_id: string, subsystem: string, created_at: timestamp}`. This four-field record is enough for the graph to know an entity exists and to link it via temporal co-occurrence. Richer metadata (relationships, typed fields, actions) is progressive enhancement — subsystems that provide it get better query results, but subsystems that provide only the minimum are still discoverable. Model this explicitly after ISO 19115's "core metadata" — the 7 required fields that any metadata record must have.

SDI-3. **P1: Entity identifier problem — no cross-reference mapping layer** — The brief acknowledges "each subsystem manages its own entities with its own ID scheme" (line 28) but the proposed solution (unified query, line 14) does not address how entity IDs are harmonized. Geospatial data from different agencies uses different coordinate reference systems (CRS) — the Australian standard is GDA2020, New Zealand uses NZGD2000, and legacy data uses AGD66. Data integration requires either CRS transformation at query time (flexible but slow) or a common CRS convention (fast but adoption-heavy). The ontology faces the identical tradeoff: bead IDs (`Sylveste-xxx`), session UUIDs, file paths, commit SHAs, and plugin names are the "coordinate reference systems" of the platform. **Failure scenario**: The ontology stores cross-system relationships using native IDs: "Bead Sylveste-a3f is related to session abc-123-def." An agent queries for the bead's sessions. The ontology returns the session UUID. The agent needs to query cass using that UUID — but cass indexes by a different key (conversation_id, not session UUID). The agent cannot complete the traversal because the ontology linked entities using identifiers that the target subsystem doesn't accept as lookup keys.

**Recommended fix**: Maintain a cross-reference mapping layer: for each entity, the ontology stores all known identifiers — `{bead_id: "Sylveste-a3f", cass_session_id: "abc-123-def", git_commit: "e4d515e4", file_path: "os/Clavain/scripts/lib-routing.sh"}`. The mapping is populated by observation (indexing subsystem stores) not by requiring subsystems to register IDs. Each identifier carries its namespace (`beads:`, `cass:`, `git:`, `fs:`) so the ontology can route queries to the correct subsystem using the subsystem's native ID.

SDI-4. **P2: No data currency metadata** — SDI catalogs are required to declare temporal validity: `dateStamp` (when the metadata was created), `maintenanceFrequency` (how often the data is updated), and `temporalExtent` (the time period the data covers). These fields let consumers assess data currency — "this road network was last updated in 2023, so new subdivisions may be missing." The concept brief's "read-only projection" design (line 68) will have variable currency across subsystems: beads data may be seconds old (if the Dolt server is local), cass data may be minutes old (if indexing runs on a schedule), and git history may be hours old (if commit indexing is batched). Without currency metadata, agents treat all graph data as equally fresh.

**Recommended fix**: Every subsystem registration includes a currency declaration: `{last_indexed: timestamp, update_frequency: "realtime|minutes|hourly|daily", known_gaps: []}`. Graph query results include per-subsystem currency so agents can make informed freshness decisions.

SDI-5. **P2: Top-down type hierarchy will not survive plugin growth** — The brief lists 6 entity categories (lines 20-27) with specific types, suggesting a top-down design from current system analysis. INSPIRE's data specification process took 8 years (2007-2015) to define 34 spatial data themes, and the result still doesn't cover many real-world datasets. A top-down type hierarchy designed from Sylveste's current 60+ plugins will miss entity types from the next 20 plugins. If extending the hierarchy requires a coordinated schema migration, the type system becomes the bottleneck that slows ecosystem growth.

**Recommended fix**: Design the type system bottom-up with extension conventions. Core types (Agent, Plugin, Bead, File, Session) are defined centrally. Plugin-specific types are registered via namespace convention: `interlens:Lens`, `interwatch:DriftScore`, `interspect:Evidence`. The ontology discovers new types automatically as subsystems publish them. No central coordination needed.

SDI-6. **P2: All-or-nothing indexing assumed** — The brief's capabilities section describes a fully connected graph ("show me everything related to X" across all subsystems) without addressing the bootstrapping problem. SDI practitioners know that catalog value is monotonically increasing with coverage but non-zero at low coverage: even indexing one agency's data makes that agency's features discoverable to all other agencies. The concept brief should address progressive value delivery: what queries work with 2 subsystems indexed? 4? All 6+?

**Recommended fix**: Define a coverage matrix showing which queries become available at each coverage level. Example: with only beads + cass indexed, "which sessions worked on this bead?" works. Add git, and "which commits are associated with this bead's sessions?" works. Each additional subsystem unlocks new cross-system queries without requiring all subsystems first.

### Improvements

1. **The catalog-of-catalogs as the architectural pattern** — Frame the ontology explicitly as a Catalog Service for Web (CSW) analog: a registry that indexes metadata about entities in subsystem stores, returns pointers to authoritative data, and supports federated query across multiple subsystem catalogs. Each subsystem publishes a "feature catalog" (entity type definitions + entity discovery records) that the ontology indexes. The ontology never owns entity data — it owns only the index and the cross-references. This pattern has been proven at national scale (ASDI, NSDI, INSPIRE) across agencies that are far more heterogeneous and politically resistant to standardization than Sylveste's plugins. (fd-spatial-data-infrastructure)

2. **Metadata quality tiers instead of uniform requirements** — SDI practice recognizes that metadata quality varies enormously across agencies. Rather than a single quality bar, define tiers: Bronze (minimum 4-field discovery record), Silver (adds typed relationships and key fields), Gold (full schema-conformant entity records with actions). The ontology provides useful results at Bronze-level coverage for all subsystems and progressively richer results as subsystems upgrade to Silver and Gold. This mirrors the ISO 19115 core-vs-comprehensive distinction and prevents the adoption cliff where subsystems either fully conform or don't participate at all. (fd-spatial-data-infrastructure)

3. **Harvesting instead of registration** — SDI catalogs support two metadata publication models: push (agencies actively register metadata) and harvest (the catalog periodically crawls agency endpoints for metadata). The harvest model has dramatically higher adoption because it requires zero effort from the data producer after initial endpoint configuration. The ontology should support both: subsystems that implement a metadata endpoint get automatically harvested (like cass and beads, which already have queryable stores), while subsystems without queryable stores can push discovery records via a simple API. (fd-spatial-data-infrastructure)
