### [P0] reconstructHalt() treats interspect absence as evidence of no halt — filesystem sentinel is not the primary authority

**Agent:** fd-polynesian-heave-to-storm-protocol
**Source domain:** Pacific storm survival protocol — heave-to action ordering (secure cargo → lash oar → drop sail → signal crew), where each step is designed so that interruption at any point leaves the canoe in the safest possible partial state
**Isomorphism:** The pelu's write-before-notify ordering means the cargo is secured (file written) before the crew is signaled (interspect record). If the smith is interrupted after securing cargo but before signaling crew, the cargo is still secured — the partial state is safe. The pelu never uses "crew not yet signaled" as evidence that the cargo was never secured.
**Description:** In `check.go:reconstructHalt()` (lines 289-343), the function checks whether `factory-paused.json` is missing, then reads `halt-record.json`. If `halt-record.json` is absent, it returns nil — concluding no halt occurred. But the brainstorm (R2, D1) specifies write-before-notify ordering: `factory-paused.json` is written FIRST, then interspect record is written. A crash between step 1 (file write) and step 2 (interspect write) leaves a system where the file was written but the interspect record is absent. In this crash scenario, `reconstructHalt()` currently observes: no file (it was deleted by the crash? No — the file was written in step 1 and persists through crashes). Wait — the scenario the agent identifies is the reverse: reconstructHalt is designed for the case where the file is MISSING but the interspect record exists. The real P0 is that reconstructHalt() — when it recreates the file from the interspect record — does not check whether the filesystem sentinel is the true primary. The actual gap: if `factory-paused.json` is present AND `halt-record.json` says `status: resolved` (a partially-completed resume where interspect was cleared but file was not yet deleted), `reconstructHalt()` returns nil at line 291 (file exists, nothing to do). This is correct behavior. However, the reverse resume crash — interspect says resolved, file is present — is not handled: `reconstructHalt()` is never consulted because `IsHalted()` sees the file and returns true, but nothing reconciles the disagreement. The factory stays halted indefinitely requiring manual file deletion because `reconstructHalt()` only runs when the file is absent (line 291 guard). The pelu's principle: sentinel disagreement must always resolve to the halted state, but here the "resolve to halted" behavior comes from ignoring the disagreement entirely — the operator has no visibility that the interspect record says resolved while the file says halted.
**Recommendation:** Add a disagreement log path to `reconstructHalt()`: after line 311 (only reconstruct if active), add a branch for `record.Status == "resolved"` that logs to stderr: `"ockham: halt-record.json resolved but factory-paused.json still present — resume may have been interrupted; run ockham resume to complete"`. No file deletion from `reconstructHalt()` — that is resume's job. One diff hunk in `check.go` around line 311.

### [P1] Root cause deduplication operates on theme name string equality — correlated themes from a single infrastructure failure can produce spurious BYPASS

**Agent:** fd-polynesian-heave-to-storm-protocol
**Source domain:** The pelu distinguishes a local squall (one indicator self-resolving) from a system storm (multiple independent indicators) — the convergence must be across independent phenomena, not two readings of the same underlying condition
**Isomorphism:** If auth-drift and auth-delegation both fire because the same auth infrastructure is down, they are two symptoms of one storm, not two storms. The pelu would read this as a single indicator (auth is compromised) manifesting across two instruments, and would not heave-to on the basis of two instruments reading the same storm.
**Description:** The brainstorm specifies BYPASS fires when `distinct_root_causes >= 2` (R2), and root cause deduplication counts "distinct themes with fired signals, not raw signal count." But the deduplication operates on theme name string equality — themes are derived from bead `lane:` labels in `check.go:closedBeadsFromBD()` (lines 196-226) and `discoverThemes()` (lines 140-172). A project using lanes `auth`, `auth-delegation`, and `auth-infra` would produce three distinct theme names from a single auth infrastructure outage. The brainstorm calls out this risk in Q1 but does not resolve it, punting to "configurable, default 2." Theme naming conventions are not enforced at the lane label level — any `lane:` label becomes a theme. The code path from lane labels to theme strings has no grouping logic.
**Recommendation:** At minimum, document in `anomaly.Config` that the BYPASS threshold implicitly assumes theme names represent genuinely independent root causes, and that operators with fine-grained lane schemas should set a higher threshold. The comment belongs adjacent to the `BypassThreshold` field (to be added in F7). Optionally add a `theme_groups` config field (map of theme-name to group-name) so that grouped themes count as one root cause. Do not block F7 on the grouping implementation — document the assumption, raise the threshold recommendation in the vision doc.

### [P1] Both sentinels share the same filesystem — shared failure domain undermines the double-sentinel independence guarantee

**Agent:** fd-japanese-tosho-yaki-ire-quench-arrest
**Source domain:** The tōshō's visual and auditory channels are physically independent sensory modalities — his eyes and ears can fail independently. If both channels use the same physical mechanism (e.g., both read from a vibration sensor), the double-channel pattern adds redundancy but not independence.
**Isomorphism:** `factory-paused.json` at `~/.config/ockham/factory-paused.json` and `halt-record.json` at `~/.clavain/interspect/halt-record.json` are on the same filesystem. A disk-full condition prevents writing `factory-paused.json`, and since both files share the same disk, `halt-record.json` also cannot be written. Both sentinels fail simultaneously — the halt is triggered but unrecorded.
**Description:** The halt.go implementation (line 28-31) checks for file existence via `os.Stat`. The `DefaultSentinelPath()` (lines 18-25) returns `~/.config/ockham/factory-paused.json`. The interspect halt record path in `reconstructHalt()` (line 296) is `~/.clavain/interspect/halt-record.json`. Both paths resolve under `$HOME`, sharing the same home directory filesystem. The brainstorm (Q2) chose direct file write (option a) for the interspect record, which means both writes go through the same OS filesystem stack. A disk-full event, filesystem remount, or NFS disconnect affects both files simultaneously. The double-sentinel pattern provides crash-ordering redundancy (one is written before the other) but not failure-domain independence.
**Recommendation:** Do not architect the solution around filesystem independence in F7 — that would require out-of-band storage. Instead, document the shared-domain limitation in a code comment at the write site (both in `halt.go` and the to-be-written BYPASS trigger). Add disk-space pre-check before the halt write sequence: `os.Statfs` the sentinel's directory and verify at least N bytes free. If the check fails, log to stderr (the smith raises alarm without striking). This is the smallest viable fix — it converts a silent failure into a visible one.

### [P0] Authorized removal of factory-paused.json is not distinguished from unauthorized removal — any process with filesystem access can lift the halt

**Agent:** fd-benedictine-interdict-sacramental-freeze
**Source domain:** The canonical principle that only the imposing authority can lift an interdict — not subordinates, not secular rulers, not bribed clergy. The interdict's power derives from its procedural integrity: only the Chancellor who sealed it can unseal it.
**Isomorphism:** `factory-paused.json` is the interdict seal. In the ecclesiastical model, the seal is not merely a physical object — it has an identity binding (the Chancellor's ring imprint). Without that binding, any forger can remove the seal and claim the interdict is lifted.
**Description:** `halt.go:IsHalted()` (lines 28-31) checks file existence via `os.Stat`. There is no identity binding — the file can be deleted by any process with write access to `~/.config/ockham/`. An automated monitoring agent (Meadowsyn polling health), a cleanup cron job, or a misconfigured script detecting "stale" files could delete `factory-paused.json` and silently resume the factory without human authorization. The brainstorm discusses confirmation requirements for `ockham resume` (Q4: "require `--confirm` flag") but this only guards the CLI command — it does not prevent out-of-band file deletion. The `ockham resume` command is the authorized lifting mechanism; it does not need to be the only mechanism by which the file is deleted, but there is currently no way to distinguish authorized from unauthorized deletion.
**Recommendation:** Set restrictive file permissions on write: `0400` (owner read-only) rather than `0644`. The ockham process writes with its own UID; other processes and cleanup agents cannot delete a `0400` file they do not own without privilege escalation. Change the `os.OpenFile` call in `reconstructHalt()` (check.go line 333) from `0644` to `0400`, and apply the same permission in the BYPASS trigger write path. The `ockham resume` command, running as the same user, can `chmod 0600` then `os.Remove`. This converts accidental deletion from a silent failure to a permission error — detectable in logs.

**Agent:** fd-benedictine-interdict-sacramental-freeze

### [P0] No deferred cleanup handler guards against factory-paused.json being removed on abnormal process exit

Only defer cleanup of the temp file — never defer removal of the final sentinel path. This also addresses the fsync durability concern below. Document in a code comment: "never defer os.Remove on the sentinel path — the sentinel must survive process crash."

### [P1] factory-paused.json write does not use fsync — kernel buffer write is not crash-safe

**Agent:** fd-dujiangyan-cascade-gate-hydraulics
**Source domain:** The stone water-gauge is carved into the riverbed — it is not chalked onto a board. The durability guarantee comes from the physical medium, not the act of writing. A chalk mark on a wet board may be written but immediately smeared by the water it is recording.
**Isomorphism:** `os.WriteFile` and `f.Write()` in Go write to the kernel buffer — they do not guarantee fsync to persistent storage. A crash between the `write()` system call returning and the kernel flushing to disk leaves a zero-byte or truncated file. `IsHalted()` calls `os.Stat()` — which returns success for a zero-byte file — so a zero-byte `factory-paused.json` is interpreted as halted (correct behavior, safe failure mode). But a truncated `factory-paused.json` that fails JSON parsing in `reconstructHalt()` at line 307 returns an error — halting reconstruction and potentially logging a confusing parse error when the system is genuinely halted.
**Description:** `reconstructHalt()` in `check.go` (lines 333-342) uses `os.OpenFile` then `f.Write()` — no fsync before `f.Close()`. The BYPASS trigger path (to be written) will similarly use some form of file write without fsync unless explicitly specified. The brainstorm does not mention fsync. For the primary sentinel write (BYPASS trigger), a crash between write and fsync could produce a corrupt file. Since `IsHalted()` only checks existence (not content), the system will still detect halt via `os.Stat` — so this is not a safety failure for `IsHalted()`. However, the diagnostic content (triggering themes, signal values) in the sentinel body would be unreadable, forcing the constrained resume operator to work blind.
**Recommendation:** In the BYPASS trigger write path, use `f.Sync()` before `f.Close()`. One line addition. For `reconstructHalt()`'s write path (line 341: `_, err = f.Write(sentinel)`), add `f.Sync()` before `f.Close()` in the defer. The durability cost is one fsync per halt event — negligible given halt frequency.

### [P1] Resume sequence deletes factory-paused.json without verifying ratchet_state reset persisted to disk

**Agent:** fd-dujiangyan-cascade-gate-hydraulics
**Source domain:** The dutian stages gate reopening with measurement between each gate — the fish-mouth is not opened until the bottle-neck pressure confirms the downstream channel is ready. Opening the downstream gate before confirming the upstream state is the failure mode that caused upstream damage in the catastrophic floods.
**Isomorphism:** Deleting `factory-paused.json` (opening the downstream gate — unblocking dispatch) before confirming the ratchet_state reset persisted (confirming the upstream gate is secured) allows the factory to resume into an authority state that does not match the post-resume intent. If the SQLite WAL has not flushed, ratchet_state domains remain at autonomous levels after the factory resumes.
**Description:** The brainstorm's D3 specifies: "Use a single SQLite transaction for domain reset... If the transaction succeeds, delete factory-paused.json." This is the correct ordering. The risk is in "transaction succeeds": Go's `database/sql` `tx.Commit()` returns nil when the database driver acknowledges the commit, but SQLite in WAL mode may not have flushed to disk. If the process crashes after `tx.Commit()` returns nil but before the WAL checkpoint, the ratchet_state reset is lost on the next cold start. The factory resumes (file deleted) with domains at autonomous level (WAL lost), but the `ockham check` cycle will not re-halt the factory because evaluateSignals() runs again on an empty-or-cleared signal state. The factory operates with elevated authority levels post-resume that the operator did not sanction.
**Recommendation:** After `tx.Commit()` and before `os.Remove(factory-paused.json)`, execute `PRAGMA wal_checkpoint(FULL)` via a raw query on the SQLite connection. This blocks until the WAL is flushed to the main database file. One SQL statement addition between the commit and the file deletion. Alternatively, open the SQLite connection for ratchet_state with `_journal_mode=DELETE` rather than WAL — simpler, no checkpoint needed, acceptable at the access frequency of resume operations.

### [P2] constrained resume mode is not persisted — subsequent ockham check cycles do not know the system is in constrained mode

**Agent:** fd-japanese-tosho-yaki-ire-quench-arrest
**Source domain:** The smith's normalization (constrained recovery) uses fundamentally different quench parameters — it is a persistent change to the operating envelope, not a one-time transient state of the quench command
**Isomorphism:** If `ockham resume --constrained` sets frozen themes at resume time but does not persist the constrained operating mode, the next `ockham check` cycle evaluates signals without knowing the system is in constrained mode and may immediately re-trigger BYPASS on the themes that were expected to show anomalies during constrained operation.
**Description:** The brainstorm's R4 specifies `--constrained` flag: "resumes with frozen themes still frozen (Tier 2 check)." ### [P3] Premature resumption guard — ockham resume does not verify triggering conditions have cleared

**Agent:** fd-polynesian-heave-to-storm-protocol
**Source domain:** The pelu knows that more canoes are lost to premature resumption than to the storm itself — the canoe that resumes course into a partially-cleared storm field cannot heave-to again if conditions worsen
**Isomorphism:** `ockham resume` is the decision to resume course. The pelu's discipline: before resuming, verify that the indicators that triggered heave-to have actually changed, not merely that time has passed.
**Description:** The brainstorm's Q4 recommends `--confirm` flag to prevent accidental resume. This is necessary but not sufficient. A `--confirm` flag prevents accidental invocation but does not prevent intentional premature resumption. If `factory-paused.json` contains the triggering theme details (per the P2 finding above), `ockham resume` can emit a pre-resume check: query current signal states for the triggering themes and report whether they are still fired. The operator sees: "WARNING: auth theme signal is still fired (drift 28%). Resume anyway? [y/N]". This does not block resume (the operator has final authority) but it closes the information gap that causes premature resumption.
**Recommendation:** If the P2 (factory-paused.json diagnostic content) finding is implemented, add a pre-resume signal state check to `ockham resume`: read the triggering themes from `factory-paused.json`, query `signal_state` for their current status, and print a summary. Recommend `--force` to bypass the check (equivalent to the pelu's "I see the indicators but we must resume anyway — we are running out of supplies"). This is P3 because it does not affect halt integrity — it affects the quality of the human decision at resume time.
