## P2 findings

### Migration archives the `.bak` — but when is it deleted?
T6 renames v1 JSON to `.v1.json.bak`. No task cleans it up. If the user downgrades, they need the `.bak` to resurrect. Fine for now. Document retention policy: "kept indefinitely until manual cleanup" in T10 docs.

### Test harness for `ic gate check` integration
T7 integration test "skip if `ic` binary not in test env; xfail with clear message" — this risks the test being skipped in CI permanently. Stronger: detect if `ic` is available, and if not, make the test FAIL in prod builds but skip in dev. Or: provide a fake `ic gate check` that parses v1 JSON (thin harness). Pick one approach before T7 starts.

### Commit granularity vs. reviewability
C4 bundles T5+T6+T7 (subcommand + migration + export). That's ~600 lines across 3 concerns. Reviewer fatigue risk. Split:
- C4a: T6 (migration) — reviewable in isolation
- C4b: T5 + T7 (subcommand + export)

### `hooks.json` record-vs-array format
T8 correctly calls out record format (per Demarch memory). Double-check: Clavain's existing hooks.json may be an array-style file — if so, migrating to record format is a breaking change for other hooks. Add T8.0: "Read existing hooks.json; if array format, coordinate with other hook authors or add alongside without format change if possible."

### Concurrent `Drain` test — goroutines vs. processes
T4's concurrent test uses goroutines. SQLite concurrency semantics differ between goroutines in the same process (shared cache) and processes (distinct handles). Production case is processes. Add a process-level test using `exec.Command` to spawn a second `calibrate-gate-tiers --auto` invocation.
