### Summary

The skill orchestrates multiple feedback loops: triage → dispatch → synthesis → compound-knowledge → next-run-triage (reinforcing); stage1 findings → expansion decision → stage2 findings → reaction round → synthesis convergence (reinforcing); trust scores → score multiplier → agent selection → finding quality → trust scores (reinforcing). Each loop has damping somewhere — exempt agents in dropout, a maximum slot ceiling, topology visibility rules — but the dampers are per-loop, not system-level. Without cross-loop monitoring, the emergent behavior over many review cycles is monotonic narrowing of which agents get called, which findings get surfaced, and which knowledge entries persist. The skill references FluxBench calibration logging (Step 2.2c) but no code path actually reads the log to adjust upstream decisions — the feedback is one-way.

### Issues Found

1. SY1. P1: Reinforcing loop on agent selection. Path: (a) agent X has low tier_bonus → (b) lower triage score → (c) less likely to run → (d) fewer observations → (e) stays in stub/generated tier. Adjacent loop: AgentDropout computes redundancy from Stage 1 coverage; if fd-performance never runs (because it's always Stage 2 and dropped), its adjacency signal never appears in Stage 1 findings, which cements its "redundant" status. Over many runs, the agent pool narrows to a stable subset. Fix: add a "diversity floor" — every N-th review (or when an agent hasn't run in M days), override the dropout/budget gates and launch the starved agent regardless. This is the explicit exploration term in an explore/exploit balance. The skill has `tier_bonus` that rewards `proven` agents (+1) but no symmetric mechanism to rescue starved agents.

2. SY2. P1: Convergence → fixative → convergence loop. `phases/reaction.md` Step 2.5.2b fires the "convergence" injection when `novelty < novelty_estimate_below`. That injection tells agents to increase novelty. But the fixative injection happens *during* reactions, and its effect on future runs isn't measured. If reactions become novel (good), the next run's convergence gate sees higher overlap (bad — each agent has unique findings) and fires again. If reactions don't become novel (fixative ineffective), convergence gate continues firing. No "fixative-effective" metric closes the loop. Fix: track fixative effectiveness across runs — if firing the same injection 3+ times without moving the metric, log it as "fixative-drift" and escalate to human.

3. SY3. P1: Primed-confirmation bullwhip. `phases/launch.md` prompt template (via prompt-template.md L50): "If any knowledge entry above matches a finding you would independently flag, note it as 'independently confirmed' in your findings." Compounding (synthesize.md L571-574) says: "If the agent noted 'independently confirmed': update the existing entry's `lastConfirmed` date and set `provenance: independent`". This is good in intent. But the signal "this finding was seen before, and another agent matched it without priming" is hard for an LLM to self-report reliably. In practice, agents tend to confirm primed findings because priming is effective. Over many runs, weak-evidence entries collect `lastConfirmed` dates and stay in the knowledge base indefinitely. Fix: add a signal-quality check — when an entry's `lastConfirmed` is updated, record which agent(s) confirmed and sample a control review (same input, no priming) every 10 compoundings to detect primed-only entries.

4. SY4. P2: Speculative launches decouple slot budget from reality. `phases/expansion.md` Step 2.2a.6: "Speculative launches do NOT count against the slot ceiling — they are bonus agents justified by Stage 1 evidence. Cap: Maximum 2 speculative launches during Stage 1." The slot ceiling is computed in Step 1.2b with scope + domain multipliers. A review with 4 slots can end up running 6 agents (4 scheduled + 2 speculative). The budget calculation in Step 1.2c.2 estimates costs for the selected agents, not for potential speculative launches. Actual token consumption can systematically exceed budget without any alert. Fix: include `speculative_reserve` in budget calculation (the expansion.md Step 2.2c already mentions `speculative_reserve = incremental_expansion.max_speculative × agent_defaults.review` but only in cross-model dispatch context — lift that to the general budget).

5. SY5. P2: Decay check is calendar-based, not review-based. `phases/synthesize.md` Silent Compounding L587-588: "If an entry has not been independently confirmed in the last 10 reviews (approximate by date: >60 days), move it to ...archive/". The approximation "10 reviews ~= 60 days" fails for projects that go dormant then burst-active. A project reviewed 2 times in a 70-day period loses useful knowledge entries. Fix: track review count per project (via `findings.json` log or a counter file) and archive on review-count threshold, not calendar.

6. SY6. P2: Streetlight effect on trust scoring. Trust multiplier (progressive-enhancements.md Step 2.1e) scales agent scores by historical precision. Agents that historically ran on easy inputs look high-precision; agents that ran on hard inputs look low-precision. The multiplier rewards specialization in easy domains. Fix: normalize trust scores by input difficulty (file count, diff size, domain complexity) before multiplying.

7. SY7. P2: Topology default is fully-connected. `phases/reaction.md` Step 2.5.2a: "If missing/disabled, use fully-connected (all agents see all findings)." Fully-connected increases convergence — each agent reads everyone's findings, makes its reaction conditional on theirs. Topology-enabled configurations (same role → full, adjacent → summary, otherwise → none) would reduce convergence. The default favors cascade. Fix: flip the default when config is missing, or make "enabled + role-based visibility" the out-of-box state.

8. SY8. P2: Configuration interactions unmapped. `config/flux-drive/budget.yaml`, `reaction.yaml`, `discourse-topology.yaml`, `discourse-fixative.yaml`, `discourse-lorenzen.yaml`, `model-registry.yaml`, `routing-overrides.json`. Each has documented defaults and individual effects. No file describes interactions — e.g., setting `dropout.enabled: false` + `budget.min_agents: 5` with a 4-slot scope creates an undefined state (under-subscribed to budget, but dropout can't prune). Fix: a `config/flux-drive/interactions.md` stating documented non-default combinations and their emergent behavior.

### Improvements

1. IMP-1. Add cross-loop monitoring. A nightly cron that reads the last 30 runs' findings.json, computes the 5 trend metrics (agent-launch diversity, finding-severity distribution, compounding rate, trust-score distribution, convergence ratio), and alerts if any drifts >2σ would close the observability loop.

2. IMP-2. Explicit hysteresis check. Before archiving knowledge entries, run a "resurrection" pass — re-query the codebase for the entry's evidence anchors. If found, reset `lastConfirmed`. This adds path-independence.

3. IMP-3. Consider a "causal graph" diagram as `docs/spec/feedback-loops.md`. Visualize: triage → dispatch → synthesis → compound → triage (reinforcing); convergence gate → fixative → reaction → convergence (balancing if effective); trust scores → selection → precision → trust (reinforcing or balancing depending on difficulty normalization). Without the picture, each loop is separately reasonable and the system drifts.

4. IMP-4. Phoenix moment: when a review produces 0 findings for N consecutive runs, force a "full-slot override" — run all eligible agents without budget caps. This detects whether the zero findings are real (good) or the pool has collapsed (bad).
