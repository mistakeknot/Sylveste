Verdict: needs-changes

### Summary

The flux-drive skill is a layered orchestrator whose core architectural seam is the SKILL.md → phase-file routing model with progressive disclosure. The model is sound in intent, but several structural issues erode the separation of concerns it claims. Dual main files (SKILL.md + SKILL-compact.md) encode the same orchestration twice with subtle drift, the phase files have numbered steps that no longer match the order they appear in the orchestrator text, and dead Composer references and monorepo-leaked paths keep the plugin from being portable. The reaction phase is fully realized in `phases/reaction.md` but silently dropped from the compact variant — a load path that begins at SKILL-compact.md will not know to run it. Knowledge handling is documented, but temp-file and path conventions leak through abstractions.

### Issues Found

1. A1. P1: Dual-file orchestration partial-load ambiguity. SKILL.md opens with an HTML comment telling the reader to "load [SKILL-compact.md] instead" if it exists. But SKILL-compact.md covers Phases 1-4 in less depth, silently omits Phase 2.5 (Reaction Round) from its Phase 2 bullet list (see line 272-298), and re-routes to the same `phases/*.md` files. A reader that follows the "load the compact file instead" instruction will never see the Phase 2.5 orchestration instruction at all. Evidence: SKILL.md L262-265 explicitly names "Phase 2.5: Reaction Round" and directs to `phases/reaction.md`; SKILL-compact.md L272-298 "Phase 2" section enumerates 2.0–2.2c and jumps to Phase 3 with no mention of reaction. Fix: either include Phase 2.5 in SKILL-compact.md's phase list, or remove the "load compact instead" redirect and keep SKILL.md canonical.

2. A2. P1: Step ordering is reversed from text. In `phases/launch.md`, the steps appear in this document order: 2.0, 2.0.4, 2.0.5, 2.1, 2.1-research, 2.1a, 2.1d, 2.1c, 2.1e, 2.2, 2.2-challenger, 2.2a, 2.2a.5–2.2c, then (bottom of the file) 2.1b. Readers looking for Step 2.1b (sliced content prep) will not find it in alphabetical order. Worse, Step 2.1c references "Step 2.1a" and "Step 2.1d" as if they run before it, but the file lists them both above 2.1c and also mentions in prose that 2.1b runs "when slicing is active" — which would logically happen before 2.1c (the temp-file write). Fix: reorder steps so document order matches numeric order; or rename to reflect actual execution sequence.

3. A3. P1: Composer references are dead control flow. `phases/launch.md` Step 2.0.4 says "If the Composer is available (`_COMPOSE_LIB_SOURCED=1`), query `compose_dispatch` for a pre-computed agent plan." Step 2.0.5 gates on `COMPOSER_ACTIVE=1`. Step 2.2 skips "if `COMPOSER_ACTIVE=1`". The skill-reviewer already flagged Composer as dead (Phase 0), but the references here are not decorative — they are branches in the dispatch state machine. A new contributor will try to implement the Composer hook, or worse, will leave the gate intact as a no-op. Fix: either rip out all Composer gates entirely, or replace with a single "deferred: see composer-backlog.md" pointer. Evidence: `_COMPOSE_LIB_SOURCED` doesn't appear anywhere else in the skill directory (I grepped the corpus mentally — launch.md is the only mention).

4. A4. P1: Hard monorepo paths leak through `phases/launch-codex.md`. The path resolution at L8-15 searches `~/.claude/plugins/cache` first (portable), but falls back to `~/projects/Sylveste/os/Clavain` (monorepo-specific) and `~/projects/Sylveste/os/Clavain` (same). For a user who installs the `interflux` plugin via the marketplace, the first path works; for a user with the cache empty and a different dev layout, the fallback is a dead end. This is a portability defect because `launch-codex.md` is a phase file of `interflux` (shipped in the plugin), not of Sylveste-the-monorepo. Fix: make the fallback env-var driven (`FLUX_CLAVAIN_ROOT` or similar) or remove the fallback and fail-fast with a clear error. Same issue at L44-45 for `create-review-agent.md`.

5. A5. P2: Lorenzen config path `interverse/interflux/config/flux-drive/discourse-lorenzen.yaml` in `phases/synthesize.md` Step 3.2 is read relative to cwd via `open(...)`. When flux-drive runs from a non-monorepo project (a user's own repo), this path is wrong — the config lives inside the `interflux` plugin, which is at `${CLAUDE_PLUGIN_ROOT}/config/flux-drive/`. Every other config read in the skill uses `${CLAUDE_PLUGIN_ROOT}`; this one is anomalous. Fix: use `${CLAUDE_PLUGIN_ROOT}/config/flux-drive/discourse-lorenzen.yaml`.

6. A6. P2: Reaction phase orphaned in compact variant. SKILL-compact.md Phase 2 list (L272-298) does not list Phase 2.5. SKILL.md explicitly instructs "Phase 2.5: Reaction Round — Read `phases/reaction.md`." A compact-only load misses the entire reaction orchestration. Either drop the "load compact instead" directive or ensure compact is a true superset of the main phase entry points.

7. A7. P2: CLAUDE.md self-contradiction on MCP server count. Line 7 says "2 MCP servers (exa, openrouter-dispatch)". The quick-commands block (L24) reads plugin.json and prints "['exa']" as an expected output — so the actual plugin.json has one MCP server named exa, and the openrouter-dispatch MCP is referenced elsewhere (e.g., `phases/launch.md` Step 2.2-challenger uses `mcp__plugin_interflux_openrouter-dispatch__review_with_model`). Either the plugin.json is missing the second server entry or the CLAUDE.md count is stale. Worth verifying.

8. A8. P2: Oracle dispatch is documented twice with divergent detail. `references/agent-roster.md` L65-74 shows the full Oracle Bash invocation with `--write-output` and error-stub fallback. `phases/launch.md` L174-178 repeats a shorter version, mentioning `DISPLAY=:99` and the partial-to-final rename convention. SKILL-compact.md L259-260 gives a third short-form version. Three sources of truth for one dispatch, likely to drift. Fix: keep the canonical recipe in `references/agent-roster.md`, replace the other two with one-line references.

9. A9. P2: Slicing classification fragments across files. `phases/slicing.md` is the algorithm; `phases/launch.md` Step 2.1c embeds "Case 1/2/3/4" with "use Method 2 from phases/slicing.md → Document Slicing" — but slicing.md no longer calls the method "Method 2" (it calls it "Method (Keyword, primary)"). SKILL.md Step 1.2d mentions "Read `phases/slicing.md` → Document Slicing" generically. Naming drift: launch.md says "Method 2", slicing.md says "(Keyword, primary)". A reader switching files can't tell they refer to the same algorithm. Fix: standardize on one method name (recommend dropping "Method 2" — slicing.md has the correct label).

10. A10. P2: AGENT_ID_MAP contract is implicit prose only. `phases/shared-contracts.md` L122-142 describes the token counting contract, mentioning "maintain a mapping of agent_name → agent_id for all dispatched agents" and "Pass it forward to synthesis via the same mechanism as slicing_map (in-memory or temp file)". Step 2.3 (launch.md L183-185) also mentions it. Step 3.4c (synthesize.md L299-329) consumes it. No schema, no helper script, no definition of what "the same mechanism as slicing_map" is (slicing_map itself has no defined storage). This is a leaky abstraction — the entire dispatch/synthesis seam depends on a prose contract. Fix: define a concrete temp-file schema (e.g., `/tmp/flux-drive-${TS}-agent-map.json`) or write to a file next to peer-findings.jsonl.

### Improvements

1. IMP-1. Decide between SKILL.md and SKILL-compact.md as canonical, not both. The current "if compact exists, load it instead" header is a design hedge. A single authoritative SKILL.md with `## Quick Reference` at the top and detailed steps below lets readers disclose progressively without encoding the same orchestration twice.

2. IMP-2. Move phase-file step numbers to reflect execution order, not logical groupings. The current "step-number as name, not position" convention works for a spec document but breaks for an orchestrator instruction that Claude executes top-to-bottom.

3. IMP-3. Consider extracting the "path resolution" logic (plugin cache + fallback) into a shared snippet at `references/plugin-paths.md`. Currently launch-codex.md, synthesize.md (CLAUDE_PLUGIN_ROOT uses), and progressive-enhancements.md (lib-trust.sh search) each redo path resolution with different fallback strategies.

4. IMP-4. Document the "Progressive Enhancement" contract as a first-class concept. The skill invokes qmd, lib-routing.sh, lib-interspect.sh, lib-trust.sh, fluxbench-challenger.sh, intercept, openrouter-dispatch MCP, interlens-mcp, interrank MCP — each with its own "skip silently if unavailable" fallback. A short reference page listing all enhancement points and their gate conditions would make the dispatch state machine legible.

5. IMP-5. Tighten the Findings Index contract. shared-contracts.md defines `- SEVERITY | ID | "Section Name" | Title`, but `references/prompt-template.md` also defines the same format with slightly different wording. synthesize.md expects to parse this. One canonical grammar (BNF or regex) in shared-contracts.md, referenced from the other two, would prevent silent synthesis failures.
