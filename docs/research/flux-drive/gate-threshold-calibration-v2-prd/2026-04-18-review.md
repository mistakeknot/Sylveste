# PRD review — gate-threshold calibration v2

The PRD inherits all findings from the brainstorm review synthesis (session_id rational subgrouping, window partitioning, consecutive-stable-windows, small-n safety, theme_source audit, fnr_threshold override — all present in acceptance criteria). This focused review targets only PRD-level concerns not covered by the brainstorm review.

## P1 findings

### Consecutive-stable-windows counter: behavior on empty drains undefined
F2's acceptance criterion says `consecutive_windows_above_threshold ≥ 3` is required for promotion, and that the counter "increments on each drain where threshold is exceeded; resets to zero on a drain where it is not." But a SessionEnd that produces zero new outcomes (exit code 2 path) is neither "above threshold" nor "below threshold" — the threshold isn't evaluated because there's nothing to evaluate. Pin one behavior:
- **Option A:** Empty drain is a no-op on the counter (treats consecutive as consecutive *drains with data*).
- **Option B:** Empty drain resets to zero (stability requires consecutive sessions, not consecutive data).

**Recommend A** — empty drains are common (short sessions, no gate fires) and shouldn't disturb the stability counter. Add explicit acceptance criterion.

### Concurrent SessionEnd hook invocations not addressed
Multiple Claude Code agents can have overlapping SessionEnd moments (e.g., two terminals ending within seconds). Both fire the hook → both open `gate.db` → race on drain. SQLite default `BEGIN` allows concurrent reads; two concurrent `BEGIN` transactions that both `UPSERT tier_state` can produce `SQLITE_BUSY`.

Fix: require `BEGIN IMMEDIATE` (write-lock at transaction start) + bounded retry with jittered backoff, or a file-level lock at `.clavain/gate.db.lock` acquired before the hook proceeds. Add acceptance criterion on F2: `calibrate-gate-tiers --auto` must tolerate concurrent invocation without corruption or double-advance of cursor.

## P2 findings

### theme_source 'registry' path is dead code
F1 derives `theme_source`: "explicit `bd state` theme label → `registry`; else `check_type` prefix → `inferred`; else `default`." But no theme registry exists. The `registry` branch triggers if `bd state <bead> theme` returns a non-empty value today — which is a bead label, not a registry. Either:
- **Clarify:** bead-label theme IS the registry (one-bead-at-a-time). OK as-is but document.
- **Stub:** accept that `registry` branch will rarely fire until a registry bead ships. Same semantics, cleaner naming: `theme_source='labeled' | 'inferred' | 'default'`.

**Recommend relabel** to `labeled` — "registry" promises infrastructure that doesn't exist.

### F3 backward-compat export tiebreak deferred
F3 AC says: "if multiple themes exist for same key, use most-recent or worst-case tier — documented tiebreak." This is a real decision, not a deferrable one. During v1→v2 migration, every v1 entry becomes `theme='default'` — so there's only one entry per key initially. Post-migration, new theme-aware outcomes populate additional rows. If `ic gate check` reads v1 JSON expecting one entry per `(check|p→p)`, the export MUST pick one.

**Recommend:** worst-case tier (hard wins over soft). Safety-leaning default; documented behavior; easy to reason about. Pin in F3 AC before plan.

### `ic gate check` JSON-schema tolerance unverified
F3 assumes the backward-compat JSON export is consumable by `ic gate check` without change. The v2 export adds no new fields to the v1 shape, so it should work — but not verified. Add to F3 AC: "Integration test: regenerate JSON from v2, run `ic gate check`, verify no parse errors and identical gate decisions vs. pre-migration."

### No observability hook for myyw.10 streak tracking
PRD mentions `drain_log.invoker` distinguishes auto vs manual but doesn't specify a read-facing command. myyw.10 will need `clavain-cli gate-streak` or similar to count consecutive auto-only drains. Either:
- Add F3 AC: "Expose `clavain-cli gate-streak` returning `{consecutive_auto: N, last_manual: ts}` so myyw.10 can read it without direct SQL."
- Leave for myyw.10 (it'll read `drain_log` via SQL directly).

**Recommend:** leave for myyw.10 to define its own read path. Over-speccing here couples the beads.
