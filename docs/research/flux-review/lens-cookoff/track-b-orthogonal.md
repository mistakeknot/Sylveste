## Actuarial Model Credibility

### Source Domain: Casualty actuarial credibility theory (Bühlmann-Straub, CAS credibility blending)

### Findings

- **[P1] No credibility weighting applied before anchor construction**: `build_anchor_suite.py` `compute_dilemma_scores()` (lines 47–130) blends all model votes via raw vote counts, with a hard-coded `LOW_BIAS_MODELS` set (line 27) as the only quality gate. Credibility theory requires Z = n/(n+k) per model-lens pair, where k is the credibility constant derived from cross-model variance. The current approach treats a 30-dilemma sample from `nemotron-3-nano-30b` as equally credible as 1,360 dilemmas from `gpt-5.4-nano`. Failure scenario: a model with high within-run stochasticity (e.g., non-zero effective temperature) contributes noise-weighted votes identical in magnitude to a stable model, inflating apparent agreement toward that model's attractor lenses. Fix: compute Z per model as `n_ok / (n_ok + k)` where k is estimated from cross-model variance of `agreement` scores; weight `corrected_votes` accumulation in `compute_dilemma_scores()` by each model's Z rather than by raw count.

- **[P1] No complement-of-credibility source defined**: `analyze_cookoff.py` `analyze()` (lines 31–273) and `build_anchor_suite.py` `select_anchors()` (lines 133–225) both treat cross-model raw agreement as the ground truth. When Z is low (few dilemmas, high variance), credibility theory requires blending observed frequency with a prior — here the natural complement is the empirical lens selection rate across all dilemmas × all models (the marginal lens distribution). Without this complement, sparse or noisy models pull the agreement metric toward the overall corpus mean silently, with no principled accounting. Specifically: the SBI "attractor" noted in README.md (lens #1, ~38% selection rate) is partly position bias and partly corpus prior — disentangling these requires a complement. Fix: in `compute_dilemma_scores()`, add a `prior_lens_rate` dict (marginal frequency of each lens across all ok results) and compute `blended_votes[lens] = Z * observed_rate + (1-Z) * prior_rate` before the 50% consensus threshold at line 101.

- **[P2] Between-model variance not decomposed**: `analyze_cookoff.py` lines 82–121 compute agreement per dilemma as `consensus_votes / total_votes`, but this conflates two variance sources: (a) prompt stochasticity — the same model answering differently if re-run (temperature > 0), and (b) genuine architectural divergence between model families. The pipeline makes exactly one call per (model, dilemma) pair (`lens_cookoff.py` line 343), so within-model noise is entirely invisible. For models with `temperature=0.0` (line 183) the within-model noise floor is theoretically zero for deterministic models, but in practice reasoning models and free-tier endpoints may not honor temperature. This means the `contested_lenses` list in `agreement_scores` (line 108) may include lenses that would converge on a second run. Recommendation: for a random 5% sample of dilemmas, run each model twice and record the duplicate results with a `is_reproducibility_check: true` flag; compute within-model flip rate before interpreting cross-model entropy.

- **[P2] Thin-lens problem unaddressed before anchor construction**: `build_anchor_suite.py` `select_anchors()` (lines 133–225) filters anchors by `corrected_agreement >= min_agreement` but does not screen for lenses with fewer than 15 total selections across the corpus. With 291 lenses, a long-tail of rarely-selected lenses has near-zero sample size and near-infinite credibility constant k — their apparent consensus in `corrected_consensus` (line 95–98) is an artifact of sparsity, not signal. A lens selected by 2 of 4 low-bias models on a single dilemma registers as 50% consensus even if that lens has 3 total selections across all 1,360 dilemmas. Fix: add a filter in `compute_dilemma_scores()` before building `corrected_consensus`: exclude any lens from `corrected_votes` where its corpus-wide total selection count is below a threshold (suggested: 15 selections, ~1% of 1,360 × low-bias model count).

- **[P2] No run-epoch column to detect temporal credibility drift**: `lens_cookoff.py` records each result to `cookoff_results.jsonl` (line 379) but the record schema (lines 364–375) contains no timestamp or run-epoch field. The pipeline supports `--resume` (line 332) which can span days. If OpenRouter or OpenAI silently updates a model between batch runs (a common occurrence), the pipeline conflates two model instances under one name. The agreement scores in `analyze_cookoff.py` would then mix pre- and post-update outputs for the same model identifier. Fix: add `"run_ts": time.time()` to the record at line 374, and add a check in `analyze_cookoff.py` `analyze()` that emits a warning if any model's results span more than 24 hours.

- **[P3] Anchor suite sampling skewed to modal lens; tails underrepresented**: `build_anchor_suite.py` `select_anchors()` lines 143–165 select anchors ranked by `corrected_agreement` descending, filling topic diversity first and then backfilling by agreement. This guarantees the suite is dominated by high-frequency lens scenarios — the credibility tail (dilemmas where the correct answer is an infrequently-selected lens) is systematically excluded. Credibility theory requires the calibration set to span the loss distribution, not just its mode. Recommendation: after the topic-diversity pass (line 157), add a tail-sampling pass that selects one anchor per lens where that lens's corpus selection rate is below the 25th percentile of all lens selection rates.

## Sensory Panel Protocol

### Source Domain: ASTM E18 trained descriptive panel methodology (flavor house / CPG sensory science)

### Findings

- **[P1] No within-model reproducibility check before cross-model aggregation**: `lens_cookoff.py` submits each (dilemma, model) pair exactly once (line 343: `work_items.append((dilemma, model_name, model_config))`). `analyze_cookoff.py` then computes cross-model agreement (lines 82–121) and `build_anchor_suite.py` promotes consensus lenses to the anchor suite — all without any within-model repeatability data. In sensory panel science, a panelist's output is only valid for aggregation once their within-panelist reproducibility exceeds r = 0.75 on check samples; below this, the panelist is in retraining, not contributing to the panel mean. Failure scenario: `glm-4.5-air` or `nemotron-3-super-120b` (both reasoning-adjacent free models) may produce different lens selections on the same dilemma on different days due to backend sampling variance or model state. If these models flip on 20% of dilemmas, their "agreement" with other models is partly noise — the pipeline cannot detect this and will promote noise-derived consensus lenses. Fix: in `lens_cookoff.py` `run_cookoff()`, insert a 2–3% blind duplicate set by adding `dilemma_idx` entries with a `is_duplicate: true` flag, running them at randomized positions in `work_items`, and computing per-model flip rate in `analyze_cookoff.py` before cross-model metrics.

- **[P1] Lens list presented in fixed numeric order for every model × dilemma pair**: `lens_cookoff.py` `build_lens_index()` (lines 58–72) builds the lens index once and reuses it identically across all 12 models and all 1,360 dilemmas. `call_model()` (lines 289–301) passes the same `lens_index` string to every call. The SELECTOR_PROMPT at line 80 explicitly references the first 10 items' scale tags. This creates a fixed-order primacy bias: lens #1 (SBI) is always first and benefits structurally from being the implicit anchor. The README notes SBI is selected ~38% of the time (note 4) and attributes this partly to position bias — but the pipeline uses fixed order, which means no per-run randomization can ever disentangle position from content signal. Failure scenario: anchors built on SBI dominance reflect the fixed-list artifact, not model judgment. When the lens library is reordered (version bump), every regression test based on SBI anchors fails not because the model changed but because the position bias target shifted. Fix: in `build_lens_index()`, accept an optional `seed` parameter; shuffle `lines` with that seed before joining. Log the seed in each JSONL record. For the full 1,360-dilemma run, use a per-dilemma seed derived from `dilemma_idx` so order is deterministic per dilemma but varied across dilemmas.

- **[P2] No panelist drift detection across ordinal batch position**: `lens_cookoff.py` processes dilemmas in the order they appear after filtering `action_type == "to_do"` (line 315), unshuffled. Batches of 5 dilemmas × all models are submitted in fixed order (lines 386–392). Models with context-window state (CLI models like `claude-haiku-cli` via `claude -p`) may accumulate implicit context across sequential prompts if the CLI retains any session state. Even for stateless API models, the ordinal position of a dilemma within the run is not recorded — if there is systematic drift (models selecting fewer lenses at positions 800–1360 due to API-side context-window or stochastic state), the pipeline cannot detect it. In sensory science, panelists at positions 40+ in a long session show statistically significant drift in attribute intensity ratings. Fix: add `"batch_position": work_items.index((dilemma, model_name, model_config))` to each JSONL record (or a sequential `call_seq` counter), then in `analyze_cookoff.py` add a drift probe: correlate `lens_count` with `call_seq` per model (Spearman rho). Flag any model with |rho| > 0.15 as drift-suspect.

- **[P2] Resume logic creates batch-boundary discontinuities with no exclusion flag**: `lens_cookoff.py` `run_cookoff()` resume logic (lines 331–337) reads completed `(dilemma_idx, model)` pairs and skips them, then processes remaining `work_items` starting from the interruption point. For CLI models (`claude-haiku-cli`, `codex-gpt5.4`, `gemini-cli`), a resume constitutes a fresh subprocess invocation — a clean context break equivalent to a panelist returning from a break. The first 50 dilemmas after a break may show selection pattern shifts that are boundary artifacts, not genuine model signal. The pipeline has no mechanism to flag these boundary records. Fix: record `"resumed_session": true` and `"session_start_seq": <seq>` in the JSONL for any result produced in a resumed run. In `build_anchor_suite.py` `select_anchors()`, filter out anchors whose `corrected_consensus` derives primarily from boundary-flagged records until drift analysis clears them.

- **[P3] No dimensionality reduction to detect perceptually redundant lenses**: `analyze_cookoff.py` treats all 291 lenses as independent attributes. In sensory panel work, trained attributes are routinely subjected to PCA to detect redundant dimensions (e.g., "buttery" and "fatty" collapsing into one factor). In this pipeline, certain lens pairs may be selected together so consistently across dilemmas and models that they effectively encode the same discriminative signal. Including both in the anchor suite inflates regression test counts without adding coverage. Recommendation: after `analyze()` returns `lens_frequency` data, add an optional `--pca` flag to `analyze_cookoff.py` that computes a lens co-selection matrix (binary vector per dilemma-model, PCA or UMAP), reports the top correlated lens pairs (r > 0.8), and flags them as candidates for lens library consolidation.

## Clinical Laboratory Proficiency

### Source Domain: CLIA/CAP laboratory proficiency testing, multi-instrument network harmonization

### Findings

- **[P0] Model version not recorded in JSONL output**: `lens_cookoff.py` records results at lines 364–375. The `record` dict includes `"model": result["model"]` (the key from the `MODELS` dict, e.g., `"deepseek-v3.2"`), but does not capture the model version string returned by the API in `data["model"]` or `data["system_fingerprint"]`. OpenRouter and OpenAI both return the resolved model version in their response payloads. Failure scenario: OpenRouter routes `"deepseek/deepseek-v3.2"` to a new checkpoint mid-run during the `--resume` scenario. Dilemmas 1–400 are answered by version A, dilemmas 401–1360 by version B. In `build_anchor_suite.py`, `compute_dilemma_scores()` (lines 47–130) aggregates both under `"deepseek-v3.2"`, producing a blended bias profile that is reproducible by neither version. The anchor suite becomes invalid the moment the provider rolls back. Fix: in `call_api_model()` at line 216 capture `data.get("model", "")` and `data.get("system_fingerprint", "")` and add them to the `_result()` return dict; write `"model_version"` and `"system_fingerprint"` into the JSONL record at lines 364–375. In `analyze_cookoff.py`, add a check that warns if any model's `model_version` values are not unanimous.

- **[P1] Models not stratified by architectural family before computing cross-model agreement**: `analyze_cookoff.py` `analyze()` computes `agreement_scores` (lines 82–121) across all available models indiscriminately. The `MODELS` dict in `lens_cookoff.py` (lines 32–54) mixes at least three distinct architectural families: (a) reasoning-adjacent (glm-4.5-air, nemotron series — these produce longer chain-of-thought outputs and show systematically different lens selection behavior as noted in the README's dropped-models table); (b) instruction-tuned RLHF (gpt-5.4-nano, claude-haiku-cli); (c) base/fine-tuned chat (qwen3.5-flash, gemma-4-31b). In clinical lab proficiency testing, mixing immunoassay and enzymatic instrument platforms into a single peer group inflates apparent between-method CV and masks genuine within-family consensus. The same effect applies here: if reasoning models consistently prefer fewer lenses while RLHF models prefer 2–3 lenses, pooling them will classify every dilemma as "contested" even when all RLHF models agree. Fix: add a `"family"` key to each entry in `MODELS` (e.g., `"rlhf"`, `"reasoning"`, `"chat"`). In `analyze_cookoff.py`, add a `stratified=True` code path that computes `agreement_scores` separately per family and reports family-level agreement alongside the global metric.

- **[P1] No external reference standard; models evaluated only against each other**: `build_anchor_suite.py` derives `expected_lenses` (line 189) from `corrected_consensus` — the intersection of low-bias model votes. There is no external validation against a human expert panel or a gold-standard lens selection. In clinical lab networks, a method that is not compared to a reference method has an uncharacterized bias — it may be precise (self-consistent) but systematically wrong. The corrected_agreement threshold of 0.7 (line 145) certifies precision, not accuracy. Failure scenario: all four `LOW_BIAS_MODELS` (line 27) have a shared training-data artifact that causes them to systematically under-select `[macro]` scale lenses. The anchor suite then sets `expected_lenses` to under-represent macro lenses, and any future model that correctly selects macro lenses will fail regression tests — not because it degraded, but because the reference was biased. Fix (minimal): add a `"reference_source": "low_bias_model_consensus"` field to the anchor suite metadata and document explicitly in the anchor suite JSON that `expected_lenses` certifies cross-model precision, not ground-truth accuracy. Longer-term: schedule a human expert labeling pass on a 50-dilemma stratified sample and compute bias of the anchor suite against it.

- **[P2] No pre-specified acceptance criteria for dilemma-level lens entropy**: `analyze_cookoff.py` routes dilemmas to either the agreement or disagreement bucket post-hoc, using a 60% threshold for consensus (line 106) and 20–59% for contested (line 108). These thresholds are not pre-specified as quality gates — they are output-display choices set during analysis. CAP requires allowable total error limits to be specified before the run for each analyte class, not derived from the data distribution. In this pipeline, the appropriate pre-specified limits would vary by `topic_group`: high-entropy dilemmas in `issue_crime_addiction` may be expected (genuine ambiguity), while high-entropy in `workplace` is more likely measurement error. Fix: add a `TOPIC_GROUP_ENTROPY_THRESHOLDS` dict to `analyze_cookoff.py` (or a config file) with per-topic acceptable disagreement limits; flag dilemmas exceeding their topic's threshold as `"exceeds_expected_entropy": true` in the exported JSON for expert review.

- **[P3] No corrective action trigger for total agreement failures**: `analyze_cookoff.py` `analyze()` computes per-dilemma agreement but has no automatic routing for dilemmas where all models disagree completely (agreement = 0 with non-empty votes). These are the clinical equivalent of a sample that fails all instruments — they should be routed to expert review, not silently passed to `build_anchor_suite.py` as near-miss candidates. Fix: in `analyze()` at line 254, add a `"total_disagreement_cases"` list of dilemmas where `agreement_scores[did]["score"] == 0` and `n_models >= min_models`; export this list separately and exclude these dilemmas from both anchors and near-misses pending investigation.

## Election Audit Risk-Limiting

### Source Domain: Risk-limiting audits (BRAVO, MINERVA, Ballot Comparison methods)

### Findings

- **[P0] No completion manifest verified before aggregation**: `analyze_cookoff.py` `analyze()` begins directly at line 31 by loading and grouping results, with the only completeness check being `len(rows) < min_models` (line 83). `build_anchor_suite.py` `compute_dilemma_scores()` has the same pattern (line 63). Neither function verifies that the expected `len(MODELS) × len(dilemmas)` records are present before computing agreement. Failure scenario: three models time out on dilemmas 200–400 (e.g., `glm-4.5-air` rate-limited, `nemotron-3-super-120b` 503, `gemini-cli` timeout). `lens_cookoff.py` writes an error record for these (line 381: `errors += 1`), which `analyze_cookoff.py` filters to `ok` (line 47: `ok = [r for r in results if not r["error"]]`). If the remaining 9 models happen to agree on these dilemmas, the pipeline promotes them to anchors with `n_models=9` and `corrected_agreement >= 0.7` — without signaling that the apparent consensus was computed on an incomplete panel. The anchor suite then contains dilemmas whose "ground truth" excluded exactly the models most likely to disagree. Fix: add a manifest check at the top of `analyze()` in `analyze_cookoff.py` before line 47: build an expected set of all `(dilemma_idx, model)` pairs and emit a named warning for each missing pair, along with a count of dilemmas with fewer than `len(MODELS)` model responses.

- **[P1] Anchor suite oversamples high-agreement cases and excludes narrow-margin dilemmas**: `build_anchor_suite.py` `select_anchors()` (lines 143–165) ranks anchor candidates by `corrected_agreement` descending and fills topic slots with the highest-scoring dilemma per topic. This systematically excludes dilemmas where the top lens wins with 55–65% model agreement — exactly the cases most sensitive to future model changes. In RLA, narrow-margin contests require larger sample sizes to certify, and are the most important contests to audit rigorously. The analogous insight: a lens calibration regression suite that only contains high-consensus cases will fail silently when a future model update shifts a borderline dilemma's lens selection, because no borderline dilemma is in the suite to catch the change. Fix: in `select_anchors()` after the topic-diversity first pass (line 159), add an explicit narrow-margin pass: select up to 20 dilemmas where `0.50 <= corrected_agreement < 0.65` and at least 2 lenses are within 15 percentage points of each other in `all_lenses_voted`. Label these `"type": "narrow_margin_anchor"` in the output.

- **[P1] Methodology not independently reproducible from recorded artifacts**: `build_anchor_suite.py` `select_anchors()` contains multiple implicit non-determinisms. Line 157: `sorted(by_topic.items())` iterates topic groups alphabetically, but within each topic group `candidates[0]` is the first element of a list built from `anchor_candidates` which is sorted by `corrected_agreement` descending — this is deterministic only if `corrected_agreement` values are unique. If two dilemmas in the same topic group have identical `corrected_agreement`, the tie-break is insertion order in the `defaultdict`, which depends on the iteration order of `scores.items()` — undefined in Python <3.7, dict-insertion-ordered in 3.7+ but not guaranteed to be reproducible across JSONL append orderings. The `near_miss_candidates` (line 175) sort has the same issue. If any randomization is introduced later, it will have no seed. Fix: add `metadata.random_seed` and `metadata.pipeline_version` fields to the anchor suite output at lines 216–224; use `sorted(..., key=lambda x: (x[1]['corrected_agreement'], x[0]))` as the stable tie-break throughout `select_anchors()` to ensure ordering is deterministic and reproducible from the raw JSONL.

- **[P2] No adversarial sampling to probe shared-bias artifact in cross-model agreement**: The pipeline's central hypothesis is that high cross-model agreement signals genuine lens relevance. But all 12 models share training data overlaps and may share systematic biases — the equivalent of a voting system that has the same firmware bug across all precincts. RLA audits this by testing the null hypothesis that the outcome is wrong; the pipeline should similarly probe whether its consensus lenses survive adversarial challenge. Specifically: for the top 20 anchor cases by `corrected_agreement`, check whether the consensus lens is also the highest-ranked lens by position in `build_lens_index()`. If position rank and consensus rank are correlated (Spearman rho > 0.5), the "agreement" may be primarily a shared position-bias artifact. Fix: add this correlation check to `analyze_cookoff.py` as a post-hoc audit step after `analyze()` returns; output it to `cookoff_analysis.json` under `"bias_audit"`.

- **[P2] No sequential stopping rule; full 1,360 × 12 run required regardless of evidence accumulation**: `lens_cookoff.py` processes all dilemmas and all models regardless of whether sufficient evidence for lens calibration has already accumulated for stable lenses. BRAVO-style evidence accumulation would allow the pipeline to certify a lens as "in calibration" after M dilemmas (where M is determined by the pre-specified confidence level and the empirically observed agreement rate), and stop running new dilemmas for that lens. At present, each run costs ~$1.73 for 500 dilemmas and scales linearly to ~$4.72 for 1,360. A sequential stopping rule could reduce cost by 30–60% for lenses with strong early consensus while spending more samples on contested lenses. This is P2 because the current cost is manageable, but the pattern becomes critical at scale. Recommendation: after each batch, compute a running `agreement_score` per lens; lenses with `running_agreement > 0.85` across at least 200 dilemmas can be removed from further sampling, reducing `work_items` for subsequent batches.

- **[P3] Audit trail lacks randomization seed and sampling parameter provenance**: `build_anchor_suite.py` writes `anchor_suite.json` with a metadata block (lines 215–224) that records `min_agreement`, `total_dilemmas_scored`, counts, and `topic_groups_covered` — but not the `--min-models` argument, the `LOW_BIAS_MODELS` set, or the `max_anchors` / `max_near_misses` values. A third party given only `anchor_suite.json` and `cookoff_results.jsonl` cannot reproduce the anchor suite without access to the source code at the exact version used. Fix: expand the metadata block to include `"min_models": args.min_models`, `"low_bias_models": sorted(LOW_BIAS_MODELS)`, `"max_anchors": max_anchors`, `"max_near_misses": max_near_misses`, and `"pipeline_commit": <git_sha>` (via `subprocess` or an env var) at output time.
