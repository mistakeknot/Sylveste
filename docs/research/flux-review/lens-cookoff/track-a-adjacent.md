## 1. Psychometric Instrument Design

### Findings

- **[P1] No criterion validity -- model consensus is treated as ground truth without justification.** `build_anchor_suite.py` uses `corrected_agreement >= 0.7` (line 145) to select anchor scenarios, and the `expected_lenses` field (line 186) is set from `corrected_consensus` -- the bias-corrected model majority vote. Nowhere in the pipeline is there a held-out validation against human expert lens selections or a documented argument for why 12-model consensus approximates "correct lens assignment." If the models share a common training-data prior (e.g., all RLHF'd against similar preference data), their agreement may encode shared bias, not genuine lens applicability. Recommendation: add a `criterion_source` field to anchor records (e.g., `"model_consensus"`) and document this limitation in the anchor suite metadata. Plan a held-out human evaluation on a 50-scenario sample to calibrate trust in the model consensus signal.

- **[P1] Position bias correction is binary inclusion/exclusion, not a graded correction, and the "low bias" model list is stale.** `build_anchor_suite.py` line 27 hardcodes `LOW_BIAS_MODELS = {"gpt-5.4-mini", "llama-4-maverick", "command-r-plus", "deepseek-v3.2"}`. Three of these four models (`gpt-5.4-mini`, `llama-4-maverick`, `command-r-plus`) are not in the active `MODELS` dict in `lens_cookoff.py` (lines 32-54) -- they have been dropped from the ensemble. The corrected agreement path (lines 77-85, 92-97, 109-113 in `build_anchor_suite.py`) filters to `low_bias_rows`, but when 0 of the 4 listed models are present in the run, `n_low` is 0, `corrected_consensus` is empty, and the `corrected_agreement` defaults to `1.0 if corrected_empty == n_low else 0.0` -- which evaluates to `1.0 if 0 == 0` i.e., `1.0`. Every dilemma gets `corrected_agreement = 1.0` and `corrected_consensus = []`. Then the anchor selection (line 145) requires `info["corrected_consensus"]` to be truthy (non-empty list), so no anchors pass. **The entire anchor extraction produces zero anchors with the current model lineup.** Recommendation: derive `LOW_BIAS_MODELS` dynamically from `analyze_cookoff.py`'s position bias grades rather than hardcoding, or at minimum update the hardcoded list to match the current ensemble.

- **[P2] Stimulus corpus may not represent Auraken's production input distribution.** The 1,360 dilemmas from `kellycyy/daily_dilemmas` are evenly distributed across 17 topic groups (80 each per README line 8), all framed as moral/ethical dilemmas with a binary to_do/not_to_do structure. Auraken in production receives open-ended user messages about career decisions, relationship dynamics, creative blocks, and organizational strategy -- domains that are structurally different from "should I do X" moral dilemmas. The anchor suite calibrated on this corpus will have coverage for moral-framework lenses but not for systems thinking, decision analysis, or creative-process lenses. Recommendation: supplement with 200-300 representative production-style prompts (anonymized) to cover the non-dilemma input space.

- **[P2] No test-retest reliability estimation.** `lens_cookoff.py` uses `temperature=0.0` (line 182) for API models, which reduces but does not eliminate stochasticity (sampling is still non-deterministic at temp=0 for many providers). CLI models (`claude -p`, `codex exec`, `gemini -p`) have no temperature control at all (lines 237-249). There is no mechanism to run the same dilemma-model pair twice to estimate intra-model reliability. If a model's lens selection for a given dilemma flips on re-run, the anchor/near-miss classification is not stable -- it measures one sample from a noisy distribution rather than a reliable signal. Recommendation: add a `--reliability-sample N` flag that re-runs N random (dilemma, model) pairs and reports intraclass correlation.

- **[P3] Near-miss extraction does not distinguish genuine domain ambiguity from model-specific topic blindspots.** `build_anchor_suite.py` lines 169-176 select near-misses based on the number of contested lenses and low raw agreement. If a single model (say `glm-4.5-air`) systematically selects zero lenses for a topic group due to training-data gaps while 11 others agree, that dilemma appears as "disagreement" when it is actually consensus + one outlier. Recommendation: add per-model outlier detection (e.g., flag a model as an outlier for a dilemma if its selection deviates from the other N-1 models by more than 2 standard deviations of pairwise Jaccard distance) and report whether near-miss disagreement is diffuse or driven by a single model.

## 2. Distributed Pipeline Reliability

### Findings

- **[P1] JSONL resume is vulnerable to truncated last line.** `lens_cookoff.py` resume logic (lines 330-336) reads existing results with `json.loads(line)`. If the process was killed mid-write (SIGKILL, OOM, power loss), the last line may be a partial JSON string. `json.loads()` will throw `JSONDecodeError`, crashing the resume path and losing all prior progress in the `completed` set. The same vulnerability exists in `analyze_cookoff.py` line 24 and `build_anchor_suite.py` line 39. Recommendation: wrap the resume reader in a try/except that skips malformed trailing lines:
  ```python
  try:
      r = json.loads(line)
  except json.JSONDecodeError:
      break  # truncated trailing line, skip it
  ```

- **[P1] Concurrent JSONL writes from asyncio tasks can interleave.** `process_item` (line 352) acquires `write_lock` (asyncio.Lock, line 349) before writing, which serializes writes within the async event loop. This is correct for single-process asyncio. However, the write itself is synchronous (`open(results_path_str, "a")` at line 378) within an async lock. If any exception occurs between acquiring the lock and completing the write (e.g., disk full), the lock is released without the file being in a consistent state. More critically, the `open` + `write` + implicit `close` is not atomic at the OS level -- a SIGKILL between `write` and `close` can lose the last record even with the lock held. Recommendation: write to a temp line variable and use `os.replace` on a temp file for atomic writes, or accept the current risk with a documented "last record may be lost on crash" note.

- **[P2] CLI subprocess timeout exists but process group cleanup is incomplete.** `call_cli_model` (line 258) uses `asyncio.wait_for(..., timeout=60.0)` and calls `proc.terminate()` on timeout (line 275). However, `terminate()` sends SIGTERM to the child process only, not to its process group. If the CLI tool spawns grandchild processes (e.g., `codex exec` may spawn a node subprocess), the grandchildren become orphans. After several timeouts, orphaned processes accumulate. Recommendation: use `os.setpgrp` in `preexec_fn` (or `start_new_session=True` on `create_subprocess_exec`) and `os.killpg` on timeout to kill the entire process group.

- **[P2] Single global semaphore for all API providers.** `RATE_LIMIT = asyncio.Semaphore(20)` (line 89) is shared across OpenRouter paid, OpenRouter free, and OpenAI direct. Free-tier OpenRouter has much lower rate limits than paid OpenAI. With 20 concurrent slots, the pipeline will hit free-tier 429s aggressively while under-utilizing paid API capacity. The retry backoff (lines 196-199) respects `Retry-After` headers but still wastes wall-clock time on retries that a per-provider semaphore would avoid. Recommendation: create per-provider semaphores (e.g., `OPENROUTER_FREE_LIMIT = asyncio.Semaphore(3)`, `OPENAI_LIMIT = asyncio.Semaphore(10)`) and select in `call_api_model` based on `provider` and whether the model ID contains `:free`.

- **[P3] No JSONL integrity check before downstream analysis.** `analyze_cookoff.py` and `build_anchor_suite.py` both read `cookoff_results.jsonl` with no duplicate detection. If the cookoff was interrupted and restarted without `--resume` (or if `--resume` had the truncated-line bug and failed, causing a manual re-run), the file may contain duplicate `(dilemma_idx, model)` pairs. Downstream analysis treats every row as a unique observation. Recommendation: add a dedup check at load time in `analyze_cookoff.py` -- if duplicates exist, keep the last occurrence and emit a warning.

## 3. Statistical Aggregation Methodology

### Findings

- **[P1] Agreement metric does not correct for chance.** `analyze_cookoff.py` lines 99-113 compute agreement as `consensus_votes / total_votes` -- the fraction of all lens votes that fall in the consensus bucket (>=60% of models agree). In a 291-label space where models select 0-3 lenses per dilemma, two models independently selecting the same lens from the full library by chance has probability ~(3/291)^2 per lens. Raw co-selection rates will be low, making even modest coincidental agreement look like strong consensus. The `build_anchor_suite.py` agreement score (lines 99-104) uses the same logic with a 50% threshold. Neither computes Fleiss's kappa, Krippendorff's alpha, or any chance-corrected metric. Recommendation: compute Fleiss's kappa for the multi-label setting alongside the raw agreement score. If kappa is low (<0.4) while raw agreement is high (>0.7), the "consensus" is driven by a small number of universally popular lenses (like SBI at 38%) rather than genuine per-dilemma agreement. This distinction is critical for anchor validity.

- **[P1] Bias correction is a step function, not a graded weight.** `build_anchor_suite.py` partitions models into `LOW_BIAS_MODELS` (hardcoded set, line 27) and all others. The corrected scores (lines 77-85) use only the low-bias subset; high-bias models are fully excluded. This is a step function at the bias threshold boundary. A model with 39% top-10 rate (just under "high" in `analyze_cookoff.py` line 228-229) gets full weight; a model with 41% gets zero weight. The corrected agreement is therefore highly sensitive to which side of 40% each model falls on -- a single re-run where one model's bias shifts from 39% to 41% flips its inclusion and changes the entire anchor set. Recommendation: use inverse-bias weighting (e.g., weight = 1 - top10_pct) so that higher-bias models contribute less signal proportionally rather than being fully excluded. This eliminates the threshold discontinuity.

- **[P2] Anchor threshold (0.7) is a hardcoded magic number with no sensitivity analysis.** `build_anchor_suite.py` line 137 accepts `min_agreement=0.7` as default. The README and code provide no rationale for 0.7 vs. 0.65 or 0.75. If the agreement distribution is clustered around 0.7 (which is plausible given the SBI attractor effect), a 5% shift in threshold could add or remove 30% of anchors. Recommendation: add a `--sensitivity` flag that re-runs anchor extraction at thresholds [0.5, 0.55, ..., 0.9] and reports how many anchors and topic groups are covered at each level. Include the sensitivity report in the anchor suite metadata so downstream consumers know how stable the set is.

- **[P2] Lens frequency counts treat each model's selections as independent, ignoring within-selection correlation.** `analyze_cookoff.py` lines 49-52 count each lens occurrence across all successful results. When a model selects [SBI, Cobra Effect, SCARF] for a dilemma, the co-occurrence of SBI and Cobra Effect is not random -- the dilemma's topic drives both selections. Treating them as independent inflates the apparent frequency of lenses that co-occur with popular lenses. For the "top 30 most selected" ranking (line 62), this means the ranking reflects co-occurrence with SBI as much as independent relevance. Recommendation: report both raw frequency and "unique dilemma" frequency (how many distinct dilemmas triggered the lens, regardless of how many models selected it) to separate per-dilemma relevance from per-model amplification.

- **[P3] No confidence intervals on any derived metric.** `analyze_cookoff.py` reports point estimates for lens frequency, agreement scores, and position bias percentages. With 12 models per dilemma, the effective sample size is small -- a lens selected by 3/12 models has a binomial 95% CI of [7%, 54%], meaning the 25% point estimate is compatible with both "rare" and "common." The anchor suite metadata (written by `build_anchor_suite.py` lines 214-225) includes `corrected_agreement` as a single number with no uncertainty band. Recommendation: add bootstrap confidence intervals (or exact binomial intervals) for per-dilemma agreement scores, at least for the top-50 anchor candidates.

## 4. Multi-Provider Model Behavior

### Findings

- **[P1] CLI models receive a structurally different prompt than API models.** `call_cli_model` (line 237) constructs the input as `f"System: {SELECTOR_PROMPT}\n\n{user_input}"` -- concatenating the system prompt into the user message as plain text. API models (`call_api_model` lines 176-180) receive the system prompt as a dedicated `system` role message. This is not a formatting difference -- it is a structural difference that affects how models parse the instruction. CLI models may treat "System:" as literal text rather than a system instruction, reducing instruction-following compliance. The `codex exec` path (line 249) passes this concatenated string via stdin, where the model receives it with no role framing at all. Recommendation: for `claude -p`, use `--system-prompt` flag if available. For `codex exec`, investigate whether structured input is supported. At minimum, document this asymmetry as a known limitation in the README.

- **[P1] No cardinality enforcement -- models can return arrays of any length.** `_parse_lens_array` (line 126) extracts a JSON array of integers and filters only for `isinstance(x, int)` (line 139). The prompt says "Pick 0-3 lenses" (line 78), but there is no enforcement. A model returning `[1, 2, 3, 4, 5, 6, 7]` has all 7 selections recorded. Models that over-select inflate their vote weight in the per-dilemma agreement computation (more lens votes per model = more influence on `lens_votes` Counter in `analyze_cookoff.py` line 89). `analyze_cookoff.py`'s per-model stats (lines 192-206) reports `Avg Lenses` but does not flag models exceeding the 0-3 cardinality instruction. Recommendation: add `result["selected_indices"] = result["selected_indices"][:3]` in `_parse_lens_array` or `call_model` to enforce the cardinality ceiling, and log a warning when truncation occurs.

- **[P2] No validation that OpenRouter returned the requested model.** `call_api_model` (lines 159-216) sends `"model": model_id` to OpenRouter but does not check the response's `model` field. OpenRouter documents that under load, free-tier requests may be served by a different model than requested. The pipeline records the result under the requested `model_name` (line 364 in `process_item`), not the actually-used model. If model substitution occurs silently, the corpus contains data from an unknown model attributed to a known model, contaminating agreement statistics. Recommendation: extract `data.get("model", "")` from the response and compare against `model_id`; if mismatched, record the result with `error="model_substituted"`.

- **[P2] `_extract_content` fallback chain may extract reasoning chain-of-thought as lens selection.** Lines 95-123 try `content`, then `reasoning_content`, then `reasoning_details`, then `reasoning`. If a reasoning model puts chain-of-thought text like "Let me think about which lenses apply... SBI is relevant because..." in `reasoning_content` and the final answer `[1]` in `content=None`, the function returns the reasoning text. `_parse_lens_array` (line 126) then searches this long reasoning string for `\[[\d,\s]*\]` -- which may match an incidental number sequence within the reasoning (e.g., "...comparing lenses [3, 7] to [12, 15]..."), extracting the wrong array. The function correctly prioritizes `content` over reasoning fields, but the README notes that some active models (glm-4.5-air) are reasoning models. Recommendation: when falling back to reasoning fields, apply `_parse_lens_array` but log a `reasoning_fallback` flag in the result record so downstream analysis can filter or weight these differently.

- **[P3] Safety-tuned model refusals are not classified distinctly from zero-lens selections.** If Claude Haiku refuses to engage with a morally charged dilemma and returns text like "I don't think I should help with this scenario," `_parse_lens_array` returns `[]`. This is recorded identically to a model that legitimately evaluated the dilemma and determined no lens applies. In `analyze_cookoff.py`'s per-model stats (line 198), both cases increment `empty_count`. For morally sensitive topic groups (`issue_crime_addiction`, `religion_custom`), safety-tuned models may show inflated "empty" rates that are refusals, not calibration signals. Recommendation: add a heuristic check for refusal patterns (e.g., raw_response contains "I cannot" or "I'm not able to" with no JSON array) and classify as `error="safety_refusal"` rather than a valid empty selection.

## 5. Information-Theoretic Lens Calibration

### Findings

- **[P1] Anchor suite optimizes for agreement (low information) while the downstream calibration task needs discrimination (high information).** `build_anchor_suite.py` `select_anchors` (line 133) sorts anchor candidates by descending `corrected_agreement` (line 147) and fills the suite from the top. High-agreement scenarios are definitionally low-information: every model agrees, so the scenario cannot distinguish between a good lens selector and a random one that happens to pick the popular lens. The near-miss set (lines 169-176) captures the high-information scenarios but is capped at `max_near_misses=50` (line 138) vs. `max_anchors=100`. The 2:1 ratio of anchors to near-misses inverts the information content ratio -- the 50 near-misses likely contain more calibration signal than the 100 anchors. Recommendation: rebalance to at least 1:1, and consider using information gain (entropy reduction given the scenario's lens distribution) rather than raw agreement as the primary anchor selection criterion.

- **[P2] No lens library redundancy analysis -- anchor suite may over-represent a few semantic clusters.** The 291-lens library is presented to models as a flat numbered list (`build_lens_index`, `lens_cookoff.py` lines 58-72). If 30 lenses are variations on "feedback frameworks" (SBI, Nonviolent Communication, Radical Candor, etc.), models will distribute votes across the cluster, and each variant gets a moderate vote count. The anchor suite then includes scenarios where each variant appears in `corrected_consensus`, but all 30 are calibrating the same conceptual territory. The net effect: 30% of the anchor suite covers "feedback frameworks" while "systems dynamics" gets 1 anchor. `select_anchors` enforces "at least 1 per topic group" (line 157) but topic group is a property of the dilemma, not the lens -- there is no lens-diversity constraint. Recommendation: cluster the 291 lenses by semantic similarity (e.g., embedding cosine distance) and add a lens-cluster diversity constraint to anchor selection analogous to the topic-group diversity constraint.

- **[P2] No pairwise model correlation analysis -- ensemble effective-N is unknown.** The 12-model ensemble is treated as 12 independent raters throughout `analyze_cookoff.py` and `build_anchor_suite.py`. If the two Nemotron variants (`nemotron-3-super-120b` and `nemotron-3-nano-30b`) have 0.85 pairwise agreement (plausible given shared architecture family), their combined vote weight should be ~1.15 independent raters, not 2.0. Similarly, `qwen3.5-flash` and `qwen3.6-plus` are from the same model family. Without computing the pairwise correlation matrix, the effective ensemble size may be 8-9 rather than 12, and agreement thresholds calibrated for N=12 are too lenient for effective-N=9. Recommendation: add a pairwise Jaccard similarity matrix computation to `analyze_cookoff.py`, report it, and use effective-N (e.g., from the eigenvalues of the correlation matrix) when setting agreement thresholds.

- **[P2] Topic group is likely a strong predictor of lens selection independent of content.** The README notes 17 topic groups, each with 80 dilemmas. If `issue_crime_addiction` dilemmas consistently trigger "Moral Disengagement" and "Restorative Justice" regardless of the specific scenario content, the pipeline is calibrating topic-to-lens mappings, not content-to-lens reasoning. `analyze_cookoff.py`'s topic coverage section (lines 158-188) reports per-topic top lenses but does not compute the mutual information between `topic_group` and `selected_lenses` to detect whether topic is doing the heavy lifting. Recommendation: compute I(topic_group; lens | dilemma_content_embedding) -- or as a simpler proxy, measure whether shuffling topic labels while keeping dilemma text fixed changes the lens frequency distribution. If it does not, topic is not the primary driver. If it does, the calibration suite is partially measuring topic recognition rather than conceptual framework selection.

- **[P3] The 100-anchor cap is arbitrary and may discard information.** `select_anchors` limits to `max_anchors=100` (line 137) with no criterion for why 100. If 200 scenarios pass the agreement threshold with good topic diversity, discarding 100 of them loses calibration signal for no documented benefit. If the downstream consumer (Auraken regression suite) can handle 200 scenarios, the cap is purely wasteful. Recommendation: make the cap a function of the downstream test budget (how many scenarios can be evaluated per regression run) rather than a hardcoded number, and document the rationale.
