## Song Dynasty Celadon Kiln Calibration

### Source Domain: Song Dynasty Longquan celadon kiln testing, 10th–13th century CE

### Structural Isomorphism

| Kiln domain | Pipeline |
|---|---|
| Test tile | Per-model selection result for one dilemma |
| Kiln position (thermal gradient) | Index position in the 291-lens list |
| Glaze recipe | The lens itself |
| Firing conditions (weather, fuel, draft) | Model idiosyncrasy / training characteristics |
| Reference tile archive | `anchor_suite.json` |
| Position-biased inspection (kiln mouth) | Position bias toward early-indexed lenses |
| Per-firing results | Per-model results for the same dilemma |

### Findings

- **[P1] Position bias is modeled as a uniform binary, not a gradient**: `analyze_cookoff.py` classifies models into three grades — low (<15%), medium (15–40%), high (>40%) — based on the single threshold `idx <= 10` (line 216–228). The kiln master knows that the thermal gradient is not binary; it decays from the firebox through a long tail. A model at 22% top-10 selection rate and a model at 38% are both graded "medium" and treated identically in `LOW_BIAS_MODELS` in `build_anchor_suite.py` (line 27). If one of those models has a sharp cliff at index 5 and the other has a gentle slope through index 40, a single threshold cannot distinguish them, and the bias-corrected agreement scores carry unmodeled variance. Recommendation: add a per-model bias profile — at minimum, compute top-10 and top-30 selection rates separately in `analyze_cookoff.py`'s position bias block and report both before collapsing to a grade.

- **[P1] Cross-model agreement may reflect shared firing season, not glaze quality**: `build_anchor_suite.py` selects anchors on `corrected_agreement` (line 144–147), which averages votes from `LOW_BIAS_MODELS = {"gpt-5.4-mini", "llama-4-maverick", "command-r-plus", "deepseek-v3.2"}`. The kiln analogy is sharp: multiple firings in the same seasonal weather produce consistent but weather-biased results. GPT-5.4-mini (OpenAI), codex-gpt5.4 (also OpenAI via Codex CLI), and claude-haiku-cli (Anthropic) share substantial RLHF-era English training data. The current pipeline has no mechanism to test whether agreement clusters by model family. If Qwen and Gemini models agree with each other for a dilemma, and GPT and Claude models agree with each other on a different lens for the same dilemma, `agreement_scores` in `analyze_cookoff.py` (lines 86–121) computes a single numeric score that treats both clusters as equal noise. Concrete failure: anchors are selected that appear high-agreement but actually encode OpenAI/Anthropic shared-prior consensus rather than lens correctness. Recommendation: in `compute_dilemma_scores`, add a family-stratified agreement check — compute agreement within `{openai-family}` models and within `{non-openai-family}` models separately, and flag dilemmas where cross-family agreement is low even when within-family agreement is high.

- **[P2] Anchor suite is built exclusively from stability cases, not boundary cases**: `select_anchors` in `build_anchor_suite.py` requires `corrected_agreement >= min_agreement` (default 0.7) and `corrected_consensus` to be non-empty (lines 143–145). The kiln master's reference archive includes tiles that always fire perfectly (stability verification) but also tiles near the boundary zones where glaze behavior is sensitive to small chemistry changes (boundary sensitivity). The current anchor suite contains no near-miss scenarios in its `anchors` list — near-misses are separately collected but play no role in the anchor regression function. If the pipeline drifts subtly on boundary dilemmas, the anchor suite will not detect it. The near-miss records written to `anchor_suite.json` carry no `expected_lenses` field (lines 199–210), meaning they cannot be used for regression comparison. Recommendation: add a `boundary_expected_lenses` field to near-miss records using the single most-agreed-upon contested lens (from `all_votes`), and include a small fixed quota of high-quality near-misses in the anchor regression suite specifically as drift-detection probes.

- **[P3] The dilemma set was not designed for uniform lens coverage**: `analyze_cookoff.py` reports under-served topics (lines 179–188) but the 17 topic groups of `daily_dilemmas.json` were constructed for a moral dilemma research dataset, not for coverage of the 291-lens space. The kiln master places tiles at regular intervals across the kiln's full length, not just at the mouth — but the dilemma set may cluster in regions of lens space that correspond to interpersonal and ethical domains, leaving lenses from other domains (professional, ecological, systemic) under-represented. Does the current topic-coverage analysis in `analyze_cookoff.py` show which specific lenses appear in zero dilemma results? The "lenses never selected" comment on line 69–70 explicitly skips this computation. Adding that count would reveal how much of the 291-lens space has no test coverage at all.

## Talmudic Machloket Adjudication

### Source Domain: Talmudic dialectical jurisprudence, Brisker analytical method, ~200–500 CE and subsequent codification

### Structural Isomorphism

| Talmudic domain | Pipeline |
|---|---|
| Machloket (structured disagreement) | Cross-model disagreement on lens selection |
| Machloket b'svara (principled disagreement) | Models applying different defensible frameworks to the same dilemma |
| Machloket b'metziut (factual disagreement) | Models misreading the dilemma or selecting an inapplicable lens |
| Minority opinion (Shammai preserved alongside Hillel) | Single-model lens selection no other model chose |
| Stam (anonymous consensus) vs. attributed reasoning | Vote count vs. per-model attribution |
| Sugya (dialectical unit with internal topology) | Disagreement cluster structure |
| Halacha l'ma'aseh (binding practical ruling) | Bias-corrected scores consumed by downstream selection |

### Findings

- **[P1] All disagreements are collapsed to a single numeric score — principled machloket and factual error are indistinguishable**: `analyze_cookoff.py`'s agreement computation (lines 86–121) and `build_anchor_suite.py`'s `compute_dilemma_scores` (lines 48–130) reduce every inter-model disagreement to a vote fraction. A dilemma where seven models select "Utilitarianism" and five models select "Deontology" — two defensible frameworks that genuinely illuminate different aspects of the dilemma — produces the same agreement score as a dilemma where seven models select "SBI Feedback" and five models select a lens that does not apply to the situation at all. The Talmud's distinction is that the first case is machloket l'shem shamayim (preserved forever as both illuminate the law) and the second case requires resolution. The pipeline treats both as "disagreement = low calibration quality." Concrete failure: defensible near-miss dilemmas with high philosophical tension are filtered out of the anchor suite because their corrected_agreement falls below 0.7, while actually-ambiguous dilemmas with coincidental high agreement are promoted to anchors. Recommendation: add a `contested_lens_applicability` field to near-miss records in `build_anchor_suite.py` — even a simple heuristic (do the contested lenses share a topic_group tag?) would distinguish principled from arbitrary disagreement without requiring semantic analysis.

- **[P1] Model-family correlation in agreement is neither tested nor discounted**: The 12 models in `MODELS` (lens_cookoff.py, lines 33–53) include at minimum two OpenAI-family models (gpt-5.4-nano, codex-gpt5.4), two Anthropic-family models (claude-haiku-cli via Claude CLI, claude-embedded in Codex), and multiple Qwen-family models (qwen3.5-flash, qwen3.6-plus). When `compute_dilemma_scores` counts that 8 of 12 models agree on a lens, it is treating each model as an independent rabbinic authority. But if 4 of those agreements come from models in the same training lineage, the effective independent agreement is smaller. The Talmudic analogy is exact: Hillel's students always agree with Hillel — that is not 20 independent opinions, it is one. No code in the pipeline computes within-family vs. cross-family agreement for any dilemma. Recommendation: define a `MODEL_FAMILIES` dict in `build_anchor_suite.py` (OpenAI-family, Anthropic-family, Qwen-family, Google-family, Nvidia-family) and add a `cross_family_agreement` field to `compute_dilemma_scores`. Anchors with high raw agreement but low cross-family agreement should be flagged.

- **[P2] Single-model minority lens selections are silently discarded**: `compute_dilemma_scores` requires `v / n >= 0.5` for `corrected_consensus` and `v / n >= 0.2` for `contested` (lines 89–91, 94–96). A lens selected by exactly one model — a single dissenting opinion — is excluded from both categories. In `build_anchor_suite.py`, the near-miss filter requires `len(info["contested_lenses"]) >= 2` (line 171). A dilemma where 11 models select one lens and 1 model selects a structurally distinct lens is classified as high-agreement and promoted to anchor, when the minority selection may indicate the one model that perceives the dilemma's most important dimension. The Talmud preserves Shammai because Shammai's reasoning applies in edge cases. Recommendation: in `compute_dilemma_scores`, add a `minority_selections` field listing lenses with 0.05 <= v/n < 0.2, with model attribution preserved. This costs nothing in storage and unlocks future analysis of whether specific models are systematically productive minority selectors.

- **[P3] The operational output (anchor_suite.json) does not document the analytical reasoning that produced it**: The `anchors` records in `anchor_suite.json` carry `expected_lenses`, `corrected_agreement`, and raw vote counts, but no record of which specific models agreed and which disagreed. A downstream consumer of the anchor suite knows that "7 of 9 models agreed on lens X" but cannot determine if that agreement came from a diverse cross-family consensus or from a family cluster. The Talmud's distinction between stam (anonymous) and attributed reasoning exists precisely because attribution carries epistemic weight. Recommendation: add a `agreeing_models` and `disagreeing_models` list to each anchor record in `select_anchors` (`build_anchor_suite.py`, lines 179–196), assembled from the `rows` data already available in `compute_dilemma_scores`.

## Polynesian Star Compass Calibration

### Source Domain: Carolinian/Micronesian etak wayfinding tradition, 32-house sidereal compass, pre-contact Pacific navigation

### Structural Isomorphism

| Navigation domain | Pipeline |
|---|---|
| Sidereal compass (32 star houses) | 291-lens library |
| Multiple independent indicators (stars, waves, birds, clouds) | Multiple independent bias detection methods |
| Reference voyage (known island-to-island bearing) | Regression anchor with known-correct lens |
| Ambiguous horizon zone (star on boundary between two houses) | Near-miss (models split between two adjacent lenses) |
| Seasonal drift (stellar positions shift relative to horizon) | Model version drift (model updates change bias profiles) |
| Etak (moving reference island, dynamic calibration frame) | Library-update re-anchoring |

### Findings

- **[P0] Position bias detection uses a single method that cannot separate positional from content effects**: `analyze_cookoff.py` detects position bias by counting what fraction of a model's selections fall in `idx <= 10` (lines 213–228). This is a single indicator — equivalent to the navigator using only one star to confirm a bearing. The critical problem: if early-indexed lenses in `lens_library_v2.json` are more general in scope (SBI appears to be lens #1 with 38% selection rate per the README), a model selecting them frequently may be responding correctly to their genuine breadth, not to their index position. The pipeline cannot distinguish these two causes because it has only one detection method. Concrete failure: the pipeline grades a model as "high bias" and excludes it from `LOW_BIAS_MODELS`, but the model is actually selecting genuinely applicable general lenses. Its exclusion from bias correction degrades corrected agreement scores on dilemmas where those general lenses are correct. The README documents this exact tension (finding #4: "SBI attractor — lens #1 gets selected ~38% of the time — a combination of genuine relevance and position bias") but the code has no mechanism to disentangle the two. Recommendation: as a second independent indicator, randomize lens index order for a held-out 10% of dilemma runs in `lens_cookoff.py` — if a model's selections track the same lenses regardless of their index position, its selections are content-driven, not position-driven. This can be done by shuffling `build_lens_index()` output and storing the shuffle mapping alongside results.

- **[P1] Regression anchors have no external ground truth — the compass is calibrated against reference voyages that were themselves recorded with the same compass**: `select_anchors` in `build_anchor_suite.py` selects anchors based on `corrected_agreement >= 0.7` where `corrected_agreement` is derived from votes by `LOW_BIAS_MODELS`, and `LOW_BIAS_MODELS` was itself identified by `analyze_cookoff.py`'s position-bias grading. The entire calibration chain — from raw cookoff results through bias grading through anchor selection — is self-referential with respect to the models being calibrated. If the low-bias models share a systematic content preference (preferring frameworks with Western philosophical pedigree, for example), the anchors encode that preference as "correct." The navigator calibrating against reference voyages recorded with a miscalibrated compass will have a consistent compass that points consistently wrong. No external signal (human expert, held-out validation set, structured test cases designed to have unambiguous answers) enters the pipeline anywhere. Recommendation: add a small (20–30 scenario) expert-validated set where lens applicability was determined by human judgment independent of model outputs. Even a rough human-curated ground-truth subset would break the self-referential loop for anchor validation.

- **[P2] Near-miss analysis does not map the global boundary topology of the lens space**: `analyze_cookoff.py` and `build_anchor_suite.py` both identify near-miss dilemmas (scenarios where models disagree) but treat each as an isolated incident. The output `near_misses` list in `anchor_suite.json` contains per-dilemma contested lens lists, but no aggregation across dilemmas. The navigator who finds that a star is ambiguous between house 12 and house 13 on twenty separate voyages does not record twenty ambiguous sightings — the navigator concludes that house 12 and house 13 share a boundary that requires a secondary indicator. If lens A and lens B appear as contested pairs across 30 dilemmas, they are taxonomically adjacent in a way the library does not explicitly encode, and the disambiguation rule for that boundary could be extracted and applied. No code aggregates `contested_lenses` across the near-miss set to identify which lens pairs are systematically confused. Recommendation: add a post-processing step in `analyze_cookoff.py` that counts co-occurrence of lens pairs in `contested_lenses` lists across all near-miss dilemmas and reports the top confused pairs. Ten lines of Counter logic at the end of `analyze()` would produce this.

- **[P3] The calibration framework has no re-anchoring plan for library updates**: `build_lens_index()` in `lens_cookoff.py` (lines 58–72) reads `lens_library_v2.json` at runtime. If a lens is added, removed, or renumbered, all previously recorded `selected_indices` in `cookoff_results.jsonl` refer to stale index mappings, and the `anchor_suite.json` expected_lenses lists (stored as name strings) may now match different index positions. The etak navigator knows that when the reference island moves, the entire bearing system must be re-referenced. The pipeline has no versioning of the lens library against the cookoff results. Does `cookoff_results.jsonl` record the library version at time of run? It does not — no `lens_library_version` field appears in the record schema (lines 364–375 of `lens_cookoff.py`). Recommendation: record a `lens_library_hash` (md5 of `lens_library_v2.json`) in each result record, so that future analysis can partition results by library version.

## Mesoamerican Obsidian Knapping Platform

### Source Domain: Pachuca green obsidian prismatic blade production, Mesoamerican lithic technology, ~200 BCE–1500 CE

### Structural Isomorphism

| Knapping domain | Pipeline |
|---|---|
| Polyhedral core | The 291-lens library as a prepared structure |
| Platform selection | Model lens selection from the library index |
| Platform prominence (visually obvious vs. technically optimal) | Description saliency vs. actual lens applicability |
| Core preparation geometry (set before knapping begins) | Library construction order and description style |
| Conchoidal fracture propagation | Error amplification through self-referential calibration |
| Hinge fracture (premature termination, almost-correct angle) | Near-miss selection (adjacent but not optimal lens) |
| Reference core collection at known preparation stages | Anchor suite for calibration verification |

### Findings

- **[P1] Position bias correction assumes the library is content-neutral across index positions, but the library's construction may encode a generality gradient**: `analyze_cookoff.py` grades position bias and identifies `low_bias_models` (lines 223–235) based solely on selection frequency by index. `build_anchor_suite.py` uses these grades to define `LOW_BIAS_MODELS` (line 27), which gates the `corrected_agreement` scores used for anchor selection. The knapper's insight: the most prominent platform is often not the best, but it is also true that platform geometry is constrained by the core's preparation. If `lens_library_v2.json` was constructed with more general/universal lenses early (SBI at position 1 with 38% selection is the documented example), the measured position bias conflates two effects: (1) models are biased toward early indices regardless of content, and (2) early lenses are genuinely more applicable to more dilemmas. Applying position bias correction to the second effect degrades selection quality for dilemmas where those general lenses are correct. The code in `analyze_cookoff.py` has no test for content-position correlation. Concrete failure: `corrected_agreement` scores systematically underweight SBI and other high-frequency early-index lenses even when they are the correct selection, causing anchors for those lenses to be excluded or downgraded. Recommendation: compute the mean `scale` tag distribution (macro/meso/micro from `build_lens_index()`) across the top-10, 11-50, 51-150, and 151+ index ranges. If `macro` lenses cluster in early positions, position bias and generality bias are confounded, and the bias correction needs a content-stratified baseline.

- **[P1] Anchors are bootstrapped from previous pipeline outputs, creating a self-referential fracture path**: The `LOW_BIAS_MODELS` set in `build_anchor_suite.py` (line 27) is a hard-coded constant derived from the results of `analyze_cookoff.py` on a previous run. The `corrected_agreement` values in `compute_dilemma_scores` are computed using these constants. The `expected_lenses` fields in anchor records come from `corrected_consensus`, which comes from `corrected_agreement`, which comes from `LOW_BIAS_MODELS`. This is a closed loop: the ground truth encoded in `anchor_suite.json` is derived from the same models whose behavior the anchor suite is supposed to validate. If gpt-5.4-mini has a systematic content preference that was not captured as position bias (e.g., preference for frameworks familiar from RLHF training), that preference propagates unchanged into `expected_lenses` and becomes what the pipeline calls "correct." Like a hinge fracture that propagates from a flawed platform through the entire blade, this error amplifies silently — every future run that shows high agreement with anchors is validating agreement with the original model's biases. Recommendation: at minimum, document this dependency explicitly in `build_anchor_suite.py` with a comment marking `LOW_BIAS_MODELS` as "derived from analyze_cookoff.py run on YYYY-MM-DD" and add a warning if the set is older than 30 days relative to the current cookoff results.

- **[P2] Per-model selection volume varies widely but all models receive equal vote weight**: `build_anchor_suite.py`'s `compute_dilemma_scores` counts votes per lens per model (lines 68–75), treating each model's selection as a single ballot. The README documents that some models select 0 lenses on most dilemmas (step-3.5-flash: 0 lenses every time; llama-4-maverick: 97.8% zero-lens selections). Among retained models, `analyze_cookoff.py`'s per-model stats (lines 192–206) show `avg_lenses` per model — if one model averages 2.8 lenses per dilemma and another averages 0.4, the first model casts 7x as many total votes, making it the de facto dominant voice in every `lens_votes` Counter. A knapper who strikes 3 blades from every platform gets 3x the influence on platform assessment statistics compared to a knapper who strikes 1. The pipeline does not normalize for this. Recommendation: in `compute_dilemma_scores`, add a per-model normalization option — either cap each model's contribution to 1 vote per lens per dilemma (binary) or weight votes by `1 / model_avg_lenses` computed from the full results set. Expose as a `--normalize-votes` flag in `build_anchor_suite.py`.

- **[P3] Near-miss lens selections are not checked for taxonomic adjacency**: `build_anchor_suite.py` identifies near-miss dilemmas by `len(info["contested_lenses"]) >= 2` (line 171) but does not distinguish between a near-miss where the contested lenses are taxonomically adjacent (hinge fracture: almost the right angle) versus one where the contested lenses are from unrelated domains (random fracture: the platform had a hidden flaw). A dilemma where models split between "SBI Feedback" and "Situation-Behavior-Impact" is a meaningful near-miss revealing a genuine lens-boundary problem. A dilemma where models split between "Game Theory" and "Attachment Theory" likely reflects input ambiguity. Does the lens library's 1,779-edge graph (`AGENTS.md` mentions typed edges) include enough adjacency data to check this? If so, checking whether contested lens pairs share edges in the graph would classify near-misses as taxonomic-boundary vs. semantic-ambiguity cases.

## Ottoman Divani Calligraphy Repertoire

### Source Domain: Ottoman imperial court calligraphy, divani script tradition, 15th–19th century CE

### Structural Isomorphism

| Calligraphy domain | Pipeline |
|---|---|
| Theoretical repertoire (300+ forms) | The 291-lens library |
| Working repertoire (60-80 forms actually used) | Each model's effective lens set across 1,360 dilemmas |
| Master-lineage bias (form preferences reveal training) | Model-family bias (selection profiles reveal training provenance) |
| Prestige hierarchy (noble forms over-used in court documents) | Desirability gradient (attractive lens descriptions over-selected) |
| Mashq (calibration practice on canonical texts) | Anchor suite regression check |
| Compositional context (adjacent letters affect form choice) | Lens interaction (co-selected lenses affect each other's appropriateness) |

### Findings

- **[P1] Desirability bias in lens descriptions is not measured and is fully confounded with position bias**: `analyze_cookoff.py` measures position bias (fraction of selections in top-10 indices, lines 213–228) but has no measurement of description-level attractiveness. The Ottoman hattat knows that certain forms carry prestige independent of their fitness — they are over-used because they look authoritative, not because they serve the text. In the lens library, frameworks with recognizable names (Utilitarianism, Game Theory, Cognitive Behavioral) may be over-selected across all models regardless of index position because LLMs trained on academic text have stronger priors for these names. A lens named "Situation-Behavior-Impact" that appears at index 1 has both position advantage and name-recognition advantage — the pipeline cannot attribute its 38% selection rate to either cause. Concrete failure: a model with 12.8% top-10 selection rate (gpt-5.4-nano per README) is graded "low position bias" and placed in `LOW_BIAS_MODELS`, but if that model's selections are driven by prestige/familiarity bias that happens to be uncorrelated with index position, its "corrected" scores are not corrected at all — they carry a different unmodeled bias. Recommendation: compute a per-lens desirability baseline — for each lens, count how often it is selected when it appears at a randomized position in a shuffled index (this requires the index-shuffle experiment already recommended under the Polynesian lens). Until that experiment is feasible, flag lenses with selection rates >3 standard deviations above mean as requiring manual review for description-attractiveness.

- **[P1] Effective repertoire breadth per model is not measured, and narrow-repertoire models have equal vote weight**: `analyze_cookoff.py` reports per-model `avg_lenses` and `empty_pct` (lines 200–206) but does not compute the count of unique lenses each model selects across all 1,360 dilemmas. If one model selects from 230 of 291 lenses and another selects from 45, the second model's votes represent a restricted and likely biased view of lens space. In `build_anchor_suite.py`, both models contribute equally to `lens_votes` and `corrected_votes`. A hattat who uses only 40 of 300 forms is not making 300 free choices — the 260 unused forms are invisible to them. Their agreement with another hattat who also uses 40 forms (possibly the same 40 due to shared master lineage) artificially inflates apparent consensus. The pipeline does not measure this. Recommendation: add a `unique_lenses_selected` count to the per-model stats block in `analyze_cookoff.py` (one additional Counter per model, ~5 lines). If any model uses fewer than 50 unique lenses across 1,360 dilemmas, flag it as a narrow-repertoire model and exclude it from `LOW_BIAS_MODELS` regardless of its position-bias grade.

- **[P2] Co-selection patterns are decomposed into independent per-lens votes, losing lens interaction information**: `compute_dilemma_scores` in `build_anchor_suite.py` iterates `for lens in r["selected_lenses"]: lens_votes[lens] += 1` (lines 72–75), treating each lens selection as an independent ballot. A model that selects `{Utilitarianism, Deontology}` for a dilemma is recorded as two separate votes, indistinguishable from a model that selected Utilitarianism for a different reason and Deontology incidentally. The hattat's compositional insight: adjacent letter forms constrain and inform each other — a selection pair carries information that neither selection alone contains. If multiple models consistently co-select two specific lenses for the same dilemma, that co-selection pattern indicates the dilemma is genuinely two-dimensional in a way that single-lens analysis cannot capture. This information is currently lost. The near-miss candidate list (lines 169–175) does not include co-selection frequency as a signal. Recommendation: in `analyze_cookoff.py`, add a co-selection Counter — for each result with `len(selected_lenses) >= 2`, record all pairs using `itertools.combinations(selected_lenses, 2)`. Report top-20 co-selected pairs in the analysis output. This is a 10-line addition that unlocks structural information about the lens library's implicit clusters.

- **[P3] Anchor performance is not tracked per model across runs — mashq drift is undetectable**: `anchor_suite.json` records `expected_lenses` and `corrected_agreement` at the time of suite generation, but no mechanism exists to compare a specific model's selections on anchor dilemmas between cookoff runs. If claude-haiku-cli's behavior changes between Claude model updates (which happen silently when using `claude -p` without a pinned model version), its anchor selections will change, but the pipeline will not detect this because it only checks aggregate agreement, not per-model longitudinal consistency. The mashq analogy is exact: the hattat detects personal drift by comparing their practice sheet against the master exemplar — but this requires the practice sheet from the previous session to be preserved. `cookoff_results.jsonl` is appended to on resume (line 337) and never preserves previous-run anchor baselines separately. Recommendation: when `build_anchor_suite.py` generates an anchor suite, also write a `per_model_anchor_baseline.json` file recording each model's selections on each anchor dilemma. On subsequent runs, compare new per-model anchor selections against this baseline and report drift per model.
