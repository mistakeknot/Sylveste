## What it is

A bash installer that adds two Claude Code hooks to detect context compaction and inject a plain-text reminder telling Claude to re-read AGENTS.md. The core insight: after compaction, Claude loses all project-specific rules and conventions, leading to "post-compaction rampage" behavior. The solution uses a marker-file bridge between PreCompact and UserPromptSubmit hooks to reliably deliver the reminder.

**Language/Stack:** Bash 4.0+, jq, python3 (for atomic JSON manipulation)

**Quality:** medium

- 10 test files covering banner, deps, hook generation, JSON escaping, settings manipulation
- Installer is a 2938-line monolith (workaround version) — more complex than the core mechanism warrants
- Well-documented README with architecture diagram and troubleshooting
- Idempotent installer with backup/restore, lock files, self-update
- Works around a real Claude Code bug (#15174, #13650): SessionStart `compact` matcher stdout not injected

## Architecture

The entire project is two shell scripts generated by an installer:

| Component | Role |
|---|---|
| `claude-precompact-marker` | PreCompact hook: writes `~/.local/state/claude-compact-reminder/compact-pending` with timestamp |
| `claude-prompt-compact-check` | UserPromptSubmit hook: checks for marker, outputs reminder to stdout (injected into context), deletes marker |
| `install-post-compact-reminder-workaround.sh` | 2938-line installer with full CLI (--status, --doctor, --template, --interactive, --update, --restore, --uninstall) |

The core mechanism is ~20 lines of bash. The other 2918 lines are installer/CLI UX.

## Core Patterns

### 1. Marker-File Event Bridge

The key transferable pattern: bridging two hook events that can't share state directly. PreCompact fires before compaction (can't inject context — too early). UserPromptSubmit fires on next user message (can inject context via stdout). A filesystem marker file bridges the temporal gap.

```
PreCompact → write marker → [compaction happens] → UserPromptSubmit → read marker → inject → delete marker
```

This is a general-purpose pattern for any "detect event X, act on event Y" hook coordination.

### 2. One-Shot Consumption

The marker file is deleted after first read — the reminder fires exactly once per compaction, not on every subsequent message. This prevents reminder fatigue while ensuring delivery.

### 3. Confirmation Requirement

All message templates require Claude to **state what rules it found** before proceeding. This isn't just a notification — it's a gate. Without the confirmation requirement, Claude could "see" the reminder and ignore it.

### 4. Template System

Four preset messages (minimal, detailed, checklist, default) covering different tradeoffs:
- **minimal:** single line, lowest context cost
- **detailed:** step-by-step recovery protocol (read AGENTS.md, check tasks, review git log, verify uncommitted changes)
- **checklist:** markdown checkboxes for structured compliance
- **default:** balanced 4-line version

## Transferable to Sylveste

### Pattern 1: Compaction-Aware Context Recovery (HIGH VALUE)

**Current state in Clavain:** Clavain's SessionStart hook (hooks/session-start.sh:290-291) already detects compaction via `_hook_source == "compact"` and injects a brief warning:

```
"Context was compacted. Task-notifications from background agents received after
this point may reference work already completed or reviewed."
```

This is a stale-notification warning, not a rule-recovery protocol. Clavain doesn't tell Claude to re-read CLAUDE.md/AGENTS.md or verify conventions after compaction.

**What to adopt:** Add a compaction-specific recovery protocol to the SessionStart additionalContext when `_hook_source == "compact"`. This is simpler than post_compact_reminder's two-hook workaround because Clavain's SessionStart hook already fires on compact and successfully injects context (the bug they work around doesn't affect additionalContext injection — it only affects raw stdout).

**Implementation:** Add ~5 lines to session-start.sh inside the `_hook_source == "compact"` branch to inject a recovery reminder alongside the existing stale-notification warning.

### Pattern 2: Marker-File Event Bridge (MEDIUM VALUE — REFERENCE PATTERN)

Clavain doesn't currently need this pattern because its SessionStart hook works for compact events. But the pattern is valuable as a reference for future cross-hook coordination (e.g., PreToolUse writing state that PostToolUse reads).

**Action:** Document the pattern in Clavain's solutions/patterns directory for future reference.

### Pattern 3: Confirmation Gate (HIGH VALUE)

The insight that Claude should **confirm** it read the rules — not just receive a reminder — is important. A reminder without confirmation is a polite suggestion. A reminder with confirmation is a gate.

**Current state in Clavain:** No confirmation requirement after compaction. The stale-notification warning is informational only.

**What to adopt:** Include "Confirm by briefly stating the key CLAUDE.md rules you found" in the compaction recovery prompt. This transforms a passive notification into an active verification step.

### Pattern 4: SKILL-compact Files (ALREADY DONE)

Clavain already has SKILL-compact.md files (646 lines across 17 skills) that provide condensed versions of skills for post-compaction contexts. This is a more sophisticated version of post_compact_reminder's approach — rather than just telling Claude to re-read one file, Clavain pre-computes compact versions of each skill.

The SKILL-compact approach is strictly better for skill retention. Post_compact_reminder's contribution is the *rule recovery* aspect (re-reading AGENTS.md/CLAUDE.md), which is complementary.

## Avoid

### 1. Two-Hook Workaround Mechanism

Clavain's SessionStart hook already injects `additionalContext` on compact events. The PreCompact + UserPromptSubmit marker-file bridge is unnecessary complexity for Clavain. The bug post_compact_reminder works around (SessionStart stdout not injected) doesn't apply to `additionalContext` injection.

### 2. Monolith Installer Pattern

The 2938-line installer script bundles everything (dependency checking, atomic JSON manipulation, backup/restore, self-update, shell completions, TUI template picker) into a single file. This is fine for a standalone tool but doesn't fit Clavain's plugin architecture where these concerns are separated (hooks.json for registration, individual scripts for logic, ic publish for distribution).

### 3. AGENTS.md-Only Focus

Post_compact_reminder assumes all project rules live in one file (AGENTS.md). Clavain has a richer context hierarchy: CLAUDE.md (project root + subproject), AGENTS.md, MEMORY.md (auto-memory), PRIME.md (beads priming), plus sprint state and companion context. A compaction recovery protocol should reference the appropriate files for the current context, not hardcode one file.

### 4. Global-Only Scope

The tool installs to `~/.claude/settings.json` with no per-project customization. Clavain's hook system is per-plugin with project-level override capability.

## Verdict

**Relevance: integration (partial)**

The core insight (compaction destroys rule retention; recovery requires active confirmation, not passive notification) is directly applicable to Clavain. The specific implementation (two-hook marker-file bridge) is unnecessary because Clavain's SessionStart hook already handles compact events correctly via additionalContext injection.

The highest-value takeaway is the **confirmation gate pattern** — requiring Claude to prove it re-read the rules before continuing. This is a ~30 minute change to session-start.sh that would meaningfully improve post-compaction behavior across all Clavain sessions.

The SKILL-compact files Clavain already has are a superior approach for skill-specific retention, but they don't cover project-level CLAUDE.md/AGENTS.md rules. The two approaches are complementary.
