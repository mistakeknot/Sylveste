# PRD: Priompt — Priority-Based Prompt Composition

## Problem

Skaffen's `SystemPrompt(phase)` returns a fixed string regardless of context window pressure. As conversations grow, there's no mechanism to drop low-priority prompt sections to stay within budget, and no way to optimize for Anthropic's prefix-based prompt caching. This leads to context overflow and wasted cache spend.

## Solution

Build `masaq/priompt`, a shared Go library that assembles prompt elements into a token budget using priority-based greedy packing. Each element has a base priority with optional phase-specific boosts. Cache-aware ordering ensures stable elements render first for prefix cache hits. Skaffen integrates by replacing its static SystemPrompt with Priompt-backed composition.

**Import direction:** `masaq/priompt` has zero imports from Skaffen or any agent package. All F5 integration code lives in `os/Skaffen/internal/session/` (or a new `internal/prompt/` package), calling into `masaq/priompt` as a downstream consumer. This preserves masaq's invariant: rendering primitives only, no agent/provider/tool imports.

## Features

### F1: Core Renderer
**What:** The `Element` struct, `Render()` function, and greedy sort algorithm that assembles elements into a prompt within a token budget. Includes the stable-prefix partition (F3) as part of the core sort — they are implemented together.
**Acceptance criteria:**
- [ ] `Element` struct with `Name`, `Content`, `Priority`, `PhaseBoost`, and `Stable` fields
- [ ] `Render(elements, budget, opts...)` returns `RenderResult` with `Prompt`, `Included`, `Excluded`, `ExcludedStable`, `TotalTokens`, `StableTokens`
- [ ] Greedy sort: elements partitioned into stable (first) and dynamic (second), then sorted by effective priority (descending) within each partition, included until budget exhausted
- [ ] Elements with empty `Content` are silently skipped
- [ ] Equal-priority elements sorted by `Name` for deterministic output (partition applied before name tie-break)
- [ ] Element names must be unique within a Render call — duplicate names are undefined behavior (documented)
- [ ] Separator inserted between adjacent included elements only — never before first or after last
- [ ] Separator tokens count against the budget; a non-empty separator always contributes at least 1 token (use `max(1, len(sep)/ratio)`)
- [ ] N included elements produce N-1 separators
- [ ] `budget <= 0`: returns `RenderResult` with empty `Prompt`, all elements in `Excluded`/`ExcludedStable`, `TotalTokens: 0`
- [ ] `nil` or empty elements slice: returns empty `RenderResult` with no error
- [ ] Repeated `Render()` calls with identical inputs and options produce byte-identical `RenderResult.Prompt`

### F2: Pluggable Tokenizer
**What:** A `Tokenizer` interface with a character-ratio heuristic default, replaceable by consumers.
**Acceptance criteria:**
- [ ] `Tokenizer` interface: `Count(s string) int` — must return non-negative integer; `Count("")` returns 0
- [ ] `CharHeuristic` struct with configurable `Ratio` field (default 4 chars/token)
- [ ] `CharHeuristic` with `Ratio <= 0` falls back to the default ratio of 4 (no panic, no div-by-zero)
- [ ] `WithTokenizer(t)` option overrides the default tokenizer in `Render()`
- [ ] Default tokenizer is `CharHeuristic{Ratio: 4}` when no override is provided
- [ ] `renderConfig` struct is unexported (lowercase) — callers cannot construct `Option` values by inspecting the concrete type

### F3: Cache-Aware Ordering
**What:** A `Stable` flag on elements that partitions rendering order so stable elements form a consistent prefix for Anthropic prompt cache hits.

Note: F3 is implemented as part of F1's sort algorithm (they cannot be separated). This section documents the stable-specific acceptance criteria.
**Acceptance criteria:**
- [ ] Elements with `Stable: true` render before elements with `Stable: false`
- [ ] Within each partition (stable/dynamic), elements are sorted by effective priority
- [ ] `RenderResult.StableTokens` reports the token count of the stable prefix (included stable elements only)
- [ ] If any stable element is excluded due to budget pressure, `RenderResult.StableTokens` is set to 0 (partial stable prefix invalidates cache)
- [ ] `RenderResult.ExcludedStable` lists names of excluded stable elements (distinct from `Excluded` which lists dynamic drops) — dropping a stable element is a higher-severity signal
- [ ] Stable ordering is deterministic across calls with identical element set and priorities
- [ ] Consumers must not mark elements with dynamic content as `Stable` — documented as a usage constraint (the library does not detect content changes)

### F4: Phase Boost
**What:** String-based phase tags that adjust element priority without importing Skaffen's Phase enum.
**Acceptance criteria:**
- [ ] `Element.PhaseBoost` is `map[string]int` (phase tag to priority adjustment); `nil` and empty map both mean "no boost"
- [ ] `WithPhase(tag)` option activates phase boosts during rendering; `WithPhase("")` and no `WithPhase` call are both treated as "no active phase" (phase boosts are inert)
- [ ] Effective priority = `Priority + PhaseBoost[activePhase]` (0 if no boost for active phase)
- [ ] Negative effective priority is valid — the element participates in sorting at a very low rank
- [ ] Phase boost can make a low-priority element survive budget pressure (positive boost) or drop a high-priority element (negative boost)

### F5: Skaffen Integration
**What:** Replace Skaffen's static `SystemPrompt` with Priompt-backed composition, wiring budget from the caller. All F5 code lives in `os/Skaffen/` — masaq/priompt has no Skaffen imports.
**Acceptance criteria:**
- [ ] New `PriomptSession` in `os/Skaffen/internal/session/` wraps `JSONLSession` and implements `agent.Session`
- [ ] `SystemPrompt` signature extended to accept budget: `SystemPrompt(phase tool.Phase, budget int) string` — callers compute budget before calling
- [ ] Budget computed in the agent loop (`loop.go`) as `contextWindowSize - outputReserve - existingMessageTokens`, passed to `SystemPrompt`
- [ ] Context window sizes added to router config (model-to-window lookup: opus=200K, sonnet=200K, haiku=200K)
- [ ] `existingMessageTokens` estimated using the same `CharHeuristic` tokenizer
- [ ] If computed budget <= 0, log a budget-exhausted evidence event and call `Render` with budget=0 (returns empty prompt, all excluded)
- [ ] `RenderResult.ExcludedStable` triggers a distinct evidence signal (stable drop = budget config problem, not normal operation)
- [ ] `RenderResult.Excluded` emitted as standard dropped-element evidence
- [ ] `JSONLSession` and `NoOpSession` are not modified — `PriomptSession` is a new implementation
- [ ] System prompt + tool definitions marked `Stable: true`; phase context and project context marked dynamic
- [ ] At least one positive behavioral test: given sufficient budget, `SystemPrompt()` returns non-empty prompt containing system instructions

## Non-goals

- **Nested priority scopes** (Cursor's JSX tree model). Flat elements are sufficient for v1.
- **Fractional knapsack** or two-pass reserves. Greedy sort is optimal for ~10-20 elements.
- **tiktoken-go dependency**. Ships with char heuristic only. Consumers inject exact tokenizers via the `Tokenizer` interface if needed.
- **Dynamic budget callbacks**. Budget is computed once before `Render()`.
- **Message truncation**. Priompt handles system-prompt-level composition, not conversation history truncation (though F5's budget computation accounts for message token count).

## Dependencies

- `masaq/` module (existing, with `replace` directive in monorepo). `priompt` is a package inside the masaq module, not a standalone module.
- `os/Skaffen/internal/agent` — `Session` interface signature change (add `budget int` parameter) for F5 only
- No external dependencies for `masaq/priompt` itself (pure Go, stdlib only)
