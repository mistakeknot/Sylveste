# PRD: interop — Unified Integration Fabric

## Problem

Sylveste's external system integrations are fragmented: interkasten owns Notion sync (TypeScript, 27 MCP handlers), interject reads GitHub (one-way), Auraken bridges signals to Intercore. There is no cross-system orchestration, no GitHub bidirectional sync, no event bridging, and no unified conflict resolution. Adding a new system (Google Drive) means building another standalone plugin with no shared infrastructure.

## Solution

Build **interop**, a Go daemon that replaces interkasten and becomes the single hub for all external system sync. Adapter-based architecture where each system implements a standard interface. Event-driven hub routes events between adapters, detects collisions, resolves conflicts via a neutral SyncJournal, and persists operation history for audit and recovery.

## Architecture

Single Go binary with goroutine-isolated adapters. Event-driven hub (webhook-first, fsnotify for local files, polling as fallback). Long-running daemon on zklw via Docker Compose. Caddy reverse proxy for webhook ingestion. MCP server mode for Claude Code sessions.

### Core Components (from flux-review)

- **EventBus**: In-process event routing with per-entity serialization (`map[EntityKey]chan Event`), CollisionWindow (5s TTL for opposing-source events), Emit validation (semantic contamination barrier), three-lane priority dispatch (express/urgent/routine)
- **SyncJournal**: Persistent adapter-independent operation log (SQLite). `Begin`/`Complete`/`MarkFailed`/`ResolveConflict`. Neutral conflict arbiter, retry queue, and audit trail
- **AncestorStore**: Persisted common ancestors for three-way merge (SQLite, indexed by entity ID). First-class component, not implementation detail
- **Event type**: `ID`, `EntityID`, `Type` (typed constants), `SourceAdapter`, `Timestamp`, `SchemaVersion`, `RoutingHints` (cross-system IDs), `TracePoints`, `Payload` (sealed interface)
- **Adapter interface**: `Start(ctx)`, `Stop()`, `HandleEvent(Event) error`, `Emit() <-chan Event`. Optional capabilities via Go interface assertion (`HealthChecker`, `StateSyncer`). Typed errors: `ErrMalformed`, `ErrTransient`, `ErrConflict`, `ErrShuttingDown`

### Operational Contracts

- Graceful shutdown: stop listener → cancel context → drain in-flight (30s) → flush SyncJournal → close MCP → exit
- Health/readiness: `/health` (liveness), `/ready` (all adapters operational)
- Per-adapter: dedicated `*http.Client` + `net/http.Transport`, bounded input channel (1000), panic recovery with `bus.Requeue()`
- Structured JSON logging via `slog`
- Docker Compose with `stop_grace_period` > drain timeout, 512M memory limit

## Features

### F1: Core Daemon + Event Bus
**What:** Go binary skeleton with event bus, SyncJournal, AncestorStore, CollisionWindow, graceful shutdown, health endpoints, and structured logging.
**Acceptance criteria:**
- [ ] `interop serve` starts daemon, listens on configured port
- [ ] EventBus routes events between registered adapters with per-entity serialization
- [ ] CollisionWindow detects opposing-source events within 5s TTL and routes to ConflictResolver
- [ ] EventBus.Emit validates events before enqueueing (non-empty EntityID, non-nil Payload, valid Type)
- [ ] SyncJournal persists Begin/Complete/MarkFailed/ResolveConflict to SQLite
- [ ] AncestorStore persists and retrieves common ancestors by entity ID
- [ ] SIGTERM triggers graceful shutdown sequence (drain, flush, exit)
- [ ] `/health` and `/ready` endpoints respond correctly
- [ ] `slog` JSON structured logging with adapter context

### F2: Beads Adapter
**What:** Bidirectional sync between event bus and beads via `bd` CLI — issue create/close/update, state changes, comments.
**Acceptance criteria:**
- [ ] Polls beads state via `bd list --format=json` on configurable interval (default 30s)
- [ ] Emits events for bead state changes (created, closed, claimed, status change)
- [ ] Handles incoming events: creates beads from GitHub issues, updates state from Notion changes
- [ ] Uses `bd` CLI exclusively — no direct Dolt access
- [ ] Deduplication: tracks last-seen state hash per bead to avoid emitting no-change events
- [ ] Typed errors for `bd` CLI failures (ErrTransient on timeout, ErrMalformed on parse failure)

### F3: GitHub Adapter
**What:** GitHub App with webhook receiver for bidirectional sync — Issues, PRs, comments, repo files.
**Acceptance criteria:**
- [ ] GitHub App webhook receiver with HMAC signature verification (`X-Hub-Signature-256`)
- [ ] Caddy forwards `X-Hub-Signature-256` header correctly
- [ ] Issues: create/close/update/comment sync with beads and Notion
- [ ] Repo files: bidirectional sync of markdown files with Notion pages
- [ ] PR events: status changes forwarded to beads
- [ ] Webhook delivery deduplication via `X-GitHub-Delivery` header (7-day TTL)
- [ ] Per-adapter `*http.Client` with rate limit awareness (respect `X-RateLimit-Remaining`)

### F4: Notion Adapter
**What:** Port interkasten's Notion sync into Go adapter — pages, databases, blocks with webhook support.
**Acceptance criteria:**
- [ ] Notion webhook receiver with signature verification
- [ ] Pages ↔ local markdown files (bidirectional)
- [ ] Databases ↔ beads issues (bidirectional)
- [ ] Block-level event debouncing (2s window per entity before emitting)
- [ ] Three-way merge for page content using AncestorStore
- [ ] Multi-workspace support via config (token resolution chain)
- [ ] Handles Notion API rate limits with exponential backoff

### F5: Local Filesystem Adapter
**What:** fsnotify-based file watcher for bidirectional markdown sync with other adapters.
**Acceptance criteria:**
- [ ] Watches configured directories via fsnotify
- [ ] Emits events on file create/modify/delete with content hashing (SHA-256) for change detection
- [ ] Handles incoming events: writes/updates/deletes local files
- [ ] Ignores dotfiles, `.git/`, `node_modules/`, configurable exclusion patterns
- [ ] Debounces rapid file changes (500ms window per file)
- [ ] Path validation on all write operations (prevent directory traversal)

### F6: MCP Server + Claude Code Plugin
**What:** MCP tools for interop management and a Claude Code plugin with skills/commands.
**Acceptance criteria:**
- [ ] MCP server exposes tools: sync-status, trigger-sync, list-conflicts, resolve-conflict, adapter-health
- [ ] Claude Code plugin with `.claude-plugin/plugin.json` in interverse directory conventions
- [ ] Skills: sync management, conflict resolution, adapter configuration
- [ ] Plugin loads without collision with other interverse plugins

### F7: interkasten Migration + Archival
**What:** Migrate interkasten's sync state into interop, then archive interkasten.
**Acceptance criteria:**
- [ ] Migration tool reads interkasten's tracked databases, WAL state, conflict history
- [ ] Converts to interop's SyncJournal + AncestorStore format
- [ ] Dry-run mode shows what will migrate without writing
- [ ] Post-migration verification: all tracked entities present in interop
- [ ] interkasten plugin archived (removed from active plugin discovery)

## Non-goals

- **Google Drive adapter** — day-2, not day-1. The adapter interface supports it; implementation is separate work.
- **Auraken adapter** — day-2. Auraken's bridge.py already sends signals to Intercore; interop can consume those via Intercore events later.
- **Real-time collaborative editing** — interop syncs document state, not keystrokes. Conflicts are resolved at sync boundaries.
- **Custom webhook server** — Caddy handles TLS termination and reverse proxy. interop only handles HTTP behind Caddy.
- **Replacing Intercore event bus** — interop's event bus is internal to its adapters. Intercore remains the system-wide signal bus.

## Dependencies

- Go 1.22+ with standard library (net/http, slog, database/sql)
- SQLite via `modernc.org/sqlite` (pure Go, no CGO)
- `bd` CLI for beads access
- GitHub App credentials (webhook secret, app ID, private key)
- Notion integration token + webhook configuration
- Caddy reverse proxy on zklw (existing infrastructure)
- Docker Compose on zklw (existing infrastructure)

## Open Questions

1. **Identity mapping storage** — config file (YAML), SQLite table, or derived from cross-system IDs in RoutingHints? Config file is simplest for day-1.
2. **Notion webhook setup** — Notion's webhook API requires a public URL during registration. Does Caddy expose a stable subdomain (e.g., `interop.zklw.example.com`)?
3. **Conflict resolution UI** — MCP tools suffice for agent-driven resolution. Do we need a web UI for human conflict review? Probably day-2.
4. **Full-sync reconciliation frequency** — flux-review recommended every 6 hours. Is that aggressive enough for active repos?
5. **interkasten deprecation timeline** — immediate archive after migration, or run both in parallel for a grace period?
