# Brainstorm: Flux-Gen P0/P1 Severity Calibration

## Problem

Generated agents produce **0% P0/P1 findings** across 120+ runs (per staging.md:197). All P0/P1 comes from 6 core agents. This means ~half the review fleet (project-specific agents) contributes no high-severity signal at all.

## Root Causes (3 structural gaps)

### 1. Template lacks domain-specific severity thresholds

`generate-agents.py:167-173` includes boilerplate:
```
- P0/P1: Issues that would cause failures, data loss, or broken functionality in production
- P2: Issues that degrade quality or create maintenance burden
```

But core agents have **domain-specific escalation rules**:
- fd-correctness: "Start with issues that corrupt persisted data"
- fd-safety: Risk classification matrix (High/Medium/Low → P0/P1/P2)
- fd-user-product: "Start by stating who the primary user is"

Generated agents get *definitions* but no *decision methodology* for their domain.

### 2. Domain profiles missing P0/P1 criteria tables

Protocol spec (protocol.md:201) says: "Extract Review Criteria section (table with P0/P1/P2 checks)" — but **0/11 domain profiles** contain these tables. The `claude-code-plugin` profile has rich Injection Criteria (per-agent bullets) but no severity classification.

### 3. No review sequence or persona in generated agents

Core agents have:
- Named personas (Julik, etc.) that establish voice and judgment style
- Explicit review sequences (what to check first)
- "What NOT to flag" boundaries
- Failure Narrative Method (for fd-correctness)

Generated agents have:
- Generic persona: "You are a specialist reviewer focused on X"
- Flat review_areas as numbered sections (no ordering rationale)
- anti_overlap only when LLM generates it (rare)

## Proposed Fix (3 interventions)

### A. Enrich `render_agent()` template with severity calibration block

Add a new section between Review Approach and Success Criteria:

```markdown
## Severity Calibration

### What makes a P0 in {domain}
{domain_p0_criteria — generated by LLM from focus + review_areas}

### What makes a P1 in {domain}
{domain_p1_criteria}

### What stays P2/P3
{domain_low_criteria}

### Decision method
When in doubt: describe the failure scenario. If it wakes someone at 3AM → P0/P1.
If it degrades quality over weeks → P2. If it's a style preference → P3.
```

The LLM prompt that generates agent specs should produce `severity_calibration` alongside `review_areas`.

### B. Add P0/P1/P2 criteria tables to domain profiles

Backfill all 11 domain profiles with severity criteria tables. These get injected into agent prompts during flux-drive dispatch, giving generated agents domain-specific severity anchors even without per-agent calibration.

Example for `claude-code-plugin.md`:

```markdown
## Review Criteria

| Priority | Criterion | Check |
|----------|-----------|-------|
| P0 | Plugin manifest validity | `plugin.json` schema conformance |
| P0 | Hook injection safety | No user content executed as shell |
| P1 | Hook error handling | All hooks have trap/cleanup logic |
| P1 | Frontmatter correctness | Required YAML fields present and valid |
| P2 | Skill cross-references | `/namespace:command` format |
| P2 | Token budget | Skills under 100 lines |
```

### C. Enhance LLM prompt for flux-gen to request decision methodology

The prompt that generates agent specs (the LLM call in flux-gen command) should explicitly request:
- `severity_calibration`: domain-specific P0/P1/P2 criteria
- `review_sequence`: what to examine first and why (ordering rationale)
- `persona_name`: a named reviewer identity (optional but proven effective)
- `failure_scenarios`: 2-3 concrete failure scenarios that would be P0/P1

## Approach Options

1. **Template-only fix (A)** — Smallest change, biggest impact. Generated agents get severity calibration block. ~1hr.
2. **Template + domain profiles (A+B)** — Template fix plus backfilling domain profiles. Compound effect: generated agents get both per-agent AND per-domain calibration. ~2hr.
3. **Full fix (A+B+C)** — All three interventions. Most complete but requires changing the LLM prompt format and updating flux-gen command. ~3hr.

## Recommendation

**Option 3 (full fix)** — The LLM prompt change (C) is the highest-leverage intervention because it makes the LLM *think about severity* during agent design rather than bolting it on after. Combined with template enrichment (A) and domain profile backfill (B), this creates a complete severity pipeline.

## Validation

Success metric: next flux-drive run with project agents produces at least 1 P0/P1 finding from a generated agent (on content that warrants it). Can test against the garden-salon brainstorm doc or the interflux codebase itself.
