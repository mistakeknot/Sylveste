# Persona/Lens Ontology — Unify the Three Stores

## What We're Building

An **object-first ontology graph** that unifies three existing persona/lens stores into a single canonical knowledge asset, with three view projections over one backing store.

**Today's fragmentation:**
1. **fd-* agents** — 660 invokable reviewer personas in `.claude/agents/` with persona + task context + review questions. Generated by `/flux-gen`, `/flux-review`, `/flux-explore`. Frontmatter has `tier`, `domains`, `use_count`. No cross-links.
2. **Auraken lenses** — 291 structured framings in `auraken-web/data/lenses.json` with `forces`, `solution`, `questions`, `discipline`, `bridge_score`, `community_id`, `effectiveness_score`. Already extracted from flux-review outputs (source field: `flux-review-ep11`).
3. **interlens plugin** — 288 FLUX cognitive lenses served via MCP (`search_lenses`, `combine_lenses`, `get_central_lenses`, `get_dialectic_triads`, etc.). Well-developed API.

**Target:** One graph, seven object types, three views. All existing entries ingested, cross-linked, deduplicated. Backed by **Apache AGE on Postgres** (verdict: adopt — see `assess-ontology-stores-2026-04-21.md`).

**MVP milestone for the first epic child:** measurable triage lift in `/flux-drive` when it queries the graph instead of filename-glob + tier heuristics. A/B test P0/P1 detection vs. current selection on a held-out corpus.

**This sprint's deliverable:** not code — a **scoped epic** with children beads and a migration plan. The ontology program is 2–4 months; forcing it into a single sprint would either overreach or underspec.

## Why This Approach

**Why unify at all.** The three stores are **latently the same thing** expressed at different maturity levels. Auraken's `bridge_score` and `community_id` are graph reasoning waiting for a graph. fd-agents are personas-who-wield-lenses (Akan goldweight persona → fuzzy-metrology lens) with the link implicit in the prompt. interlens already ships `find_bridge_lenses` and `get_dialectic_triads` — mature graph semantics with no graph DB underneath. Unifying collapses three partial implementations into one asset.

**Why Palantir-style object-first ontology.** Typed objects with typed first-class relationships scale to the complexity we already have latent in the data (persona ↔ wields ↔ lens ↔ bridges ↔ lens ↔ derives-from ↔ source). Prior-art assessment confirms this framing is *right-sized, not overengineered*, for ~1200 entities growing to 10K. Adjacency-list-on-Postgres would force application-layer graph logic we'd eventually want to rip out.

**Why Apache AGE + Postgres.** Auraken already runs Postgres + pgvector. AGE is a Postgres extension — zero new ops surface, Cypher queries in the same DB as embeddings, schema-first DDL for ontology discipline, manual bi-temporal via timestamp columns. TerminusDB is more elegant for audit/versioning but a separate service. Neo4j Community's GPLv3 blocks commercial paths. See assessment for full comparison.

**Why epic-not-sprint.** User has ADHD and many parallel Sylveste subprojects (per memory). Durable anchors (beads, docs) are how work survives. Three-to-six scoped children beads lined up in execution order outlast any single session's context. The MVP (triage lift) is itself valuable standalone even if follow-on views slip.

**Why flux-drive triage as MVP.** It's the dogfood loop with the fastest feedback cycle. Every sprint I run hits `/flux-drive`. If the ontology improves agent selection, the signal shows up in days, not quarters. Hermes and catalog views are user-facing surfaces that need Hermes/Auraken pivot work upstream (sylveste-heh8) before they can consume the ontology.

## Key Decisions

### D1. Scope: stop-and-plan-epic
This sprint closes as scoping. Bead `sylveste-b1ha` becomes the epic parent. Children beads are created in the strategy + plan steps (not brainstorm — brainstorm captures the shape, strategy formalizes the epic PRD, write-plan enumerates children).

### D2. Model: two linked entities
Persona and Lens are distinct types with many-to-many `wields` relationships. One persona can wield many lenses; one lens can be embodied by many personas. This matches how fd-agents implicitly work and preserves Auraken's lens-as-thinking-tool purity (some lenses have no natural human persona — they're pure framings).

### D3. Storage: Apache AGE on Postgres
Per `docs/research/assess-ontology-stores-2026-04-21.md`. Reuses Auraken's infrastructure. Graph queries in Cypher, relational queries in SQL, embeddings in pgvector, all in one DB.

### D4. Object types for V1 (7 of 8)
- **Persona** — archetype/character (Akan goldsmith-assayer, agent-systems architect)
- **Lens** — framing / thinking tool (forces, solution, questions)
- **Domain** — cross-cutting tag (agent-systems, orchestration, compliance)
- **Discipline** — formal field of study (organizational psychology, systems theory)
- **Source** — provenance (flux-review-ep11, research report 2026-03-25-novel-frameworks, manual authoring)
- **Evidence** — specific citations backing a lens's claims
- **Concept** — named idea a lens references (emergence, feedback loop, enabling constraint)
- **Deferred:** Task-context (non-portable — it's a fd-gen artifact, not an ontology fact; retained as a property on Persona for provenance, not promoted to first-class)

### D5. Consumers: three views over one graph
- **flux-drive triage view** (MVP): Cypher queries for agent selection — domain match × discipline coverage × effectiveness × community neighborhood
- **Hermes conversational view** (V2): persona+lens pairs for live reasoning, filtered by scale (micro/meso/macro) and conversational state
- **Catalog browse view** (V3): public/user-facing surface, likely consuming an HTTP API that projects the graph to JSON

### D6. Relationships as first-class (with properties)
- `Persona —[wields {affinity, generated_at}]→ Lens`
- `Lens —[bridges {strength}]→ Lens` (the bridge_score already in Auraken)
- `Lens —[contradicts]→ Lens`, `Lens —[supersedes]→ Lens`
- `Lens —[derives-from]→ Source`
- `Lens/Persona —[in-domain]→ Domain`, `Lens —[in-discipline]→ Discipline`
- `Lens —[cites]→ Evidence`, `Lens —[references]→ Concept`
- `* —[same-as {confidence, method}]→ *` (semantic dedup across stores)

### D7. Dedup before unification is a non-goal
The graph's `same-as` relationship captures semantic equivalence without collapsing entries. A Persian-medicine-assay lens and an Akan-goldweight-metrology lens might be "same-as" at 0.8 confidence but stay as distinct entries because their personas and task contexts differ. This respects the cultural/disciplinary richness rather than homogenizing it.

### D8. interlens's MCP surface is an adapter, not the source of truth
Post-migration, the interlens MCP server reads from the AGE graph (or a cached projection) rather than from a bundled JSON. The MCP tool contracts (`find_bridge_lenses`, `get_dialectic_triads`) stay stable — only the backing query changes. Existing interlens consumers see no breakage.

### D9. Versioning: bi-temporal + explicit schema_version
AGE doesn't natively do bi-temporal. `valid_from`/`valid_to` columns on entities and relationships — plus `schema_version: semver` on every entity (per G6 below) so schema migrations don't collapse into instance-level timestamps. `WHERE valid_to IS NULL` partial index on the hot path. If audit requirements grow (compliance, external publishing, lens provenance disputes), evaluate TerminusDB as an auxiliary git-for-data layer in V2.

## Gated Decisions (committed post flux-review)

The 2-track flux-review (10 agents, 61 findings, 9 P0, synthesis: `docs/research/flux-review/persona-lens-ontology-brainstorm/2026-04-21-synthesis.md`) surfaced nine load-bearing commitments the original brainstorm left as aspirational language ("idempotent", "embedding-based", "measurable lift", "bi-temporal"). These are now **gates** — the strategy PRD fails review if any is unresolved, and children beads fail plan review if they elide them. The synthesis verdict was *SHIP-READY to strategy with the nine P0s captured as gates*.

### G1. Cypher benchmark spike is Epic child #1 (not DDL)
Before schema DDL is written, load 10k then 100k synthetic edges into AGE and `EXPLAIN ANALYZE` the MVP triage query (`Persona × Lens × Domain × Discipline match with effectiveness filter and 2-hop community neighborhood`). If the plan is not index-friendly at 100k edges, redesign before Phase 2 ingestion begins — potentially collapsing the query to fewer hops, denormalizing a hot path, or reconsidering AGE itself. The `EXPLAIN ANALYZE` transcript is a committed artifact on the child bead. *(Addresses ACQ-01 P0.)*

### G2. Ingestion idempotence: explicit key per importer
- fd-agents importer: frontmatter `name` is the idempotence key; `MERGE (p:Persona {name: $name}) SET ...` upsert semantics
- Auraken lenses importer: JSON `id` field; same MERGE pattern
- interlens importer: JSON `id` field; same MERGE pattern
- Regression test: run each importer twice against a clean DB, assert node count stable, assert no `same-as` edges at cosine 1.0 between self-and-self
- Dry-run mode reports the would-change set before writing *(Addresses MIS-01 P0.)*

### G3. `same-as` edge requires source-independence + curator promotion
Schema additions:
- `same-as` gets `source_independence: bool` and `corroborator_count: int` in addition to `confidence` and `method`
- Automatic dedup pass emits `candidate-same-as` only; a curator review step promotes to `same-as`
- Embedding input is essence-text (definition + forces + solution); `task_context` is stripped before comparison to avoid cross-store asymmetric similarity inflation
- Tier inheritance via same-as traversal is **blocked** unless `source_independence = true AND corroborator_count ≥ 2` *(Addresses the triple-convergence finding: isnad tier laundering + perfumery two-roses + sibu 互見 + Track A SDC-01/02.)*

### G4. `bridges` edge is fully specified in DDL
- Directionality: `directed: bool` (most bridges are asymmetric — Lens_A illuminates Lens_B but not vice versa; some are symmetric)
- Temporal activation: `activation_delay: enum[immediate, short, medium, long]` — encodes the base/heart/top-note timing from perfumery; meaningful for Hermes conversational pacing
- Auraken `bridge_score` (float on Lens) → AGE `bridges.strength` (float on edge) transform specified: `strength = bridge_score_source × bridge_score_target / norm_constant` with norm_constant documented in migration
- Not retrofittable after 1000+ bridges edges exist *(Addresses quipu cord-typing P0 + perfumery temporal P1 + quipu bridge_score transit P1.)*

### G5. Single query authority: flux-drive triage as the iemoto reference
- The `/interflux:flux-drive` triage query is the canonical reference implementation
- Selection logic is extracted into a new versioned module: `packages/ontology-queries/` (exact location TBD in strategy)
- The module publishes typed query functions: `selectPersonaeForTask`, `expandByBridges`, `rankByEffectiveness`, etc.
- Hermes view and Catalog view are **adapters** over this module — they MUST NOT reimplement selection logic; they may compose, filter, or project the module's output
- Any change to canonical query semantics goes through the module's versioned API (semver breaking = all three consumers coordinate)
- Without this, the ontology becomes three drifted schemas within 18 months *(Addresses noh iemoto P0 + perfumery projection drift P2 + isnad effectiveness_score ownership P1 + ACQ-05 MCP caching concern. Highest-leverage single commitment per synthesis.)*

### G6. `schema_version` on every entity, partial-index on `valid_to IS NULL`
- Migration 001 adds `schema_version: semver` to all entity tables (Persona, Lens, Domain, Discipline, Source, Evidence, Concept)
- `WHERE valid_to IS NULL` partial index on all entity tables — the hot path is "current state only" and must not scan historical rows
- Distinguishes schema evolution from instance evolution; irreversible to retrofit after ingestion writes a million rows with NULL `schema_version` *(Addresses sibu P0 schema-vs-instance + noh P2 schema_version + ACQ-04 partial-index P1.)*

### G7. Evidence nodes are graded; `cites` carries transmitter tier
- `Evidence.strength_grade: enum[sahih, hasan, da'if, mawdu]` — borrowed from Islamic hadith jarh wa-ta'dil grading, precisely because the problem is the same (source credibility tiering)
- `cites` edge gains `transmitter_tier: int` (0 = original author, 1 = faithful summary, 2+ = citation-of-citation)
- A lens citing a Nature paper (`sahih`, tier 0) is distinguishable from a lens citing a Medium summary of that paper (`hasan`, tier 2) — tier inheritance rules in G3 operate on `strength_grade`, not just `confidence`
- Part of initial DDL, not a follow-up migration *(Addresses isnad jarh wa-ta'dil P0.)*

### G8. Lens identity: immutable node + `supersedes` + `lens_identity_uuid`
- Lens nodes are **immutable** once committed. Edits create a new Lens node with a `supersedes` edge pointing to the predecessor.
- Every Lens carries a `lens_identity_uuid` — stable across the supersedes chain — so consumers can ask "give me the current Lens with identity X" and receive the tip of the chain
- Same rule applies to Persona (edits create new Persona node + supersedes) to keep the model consistent
- Determines ingestion behavior (re-import of edited Auraken lens → new node + supersedes, not overwrite), dedup behavior (same-as compares tip-of-chain, not ancestors), and triage query shape (always filters to current via partial index)
- Freeze in strategy step *(Addresses OSD-02 P1 — elevated to gate because it pervades every downstream decision.)*

### G9. Source becomes a Transmission chain
- Replace monolithic `Source` node with a `Transmission` chain: `Source —[transmitted_via]→ Transmission —[prior]→ Transmission ...`
- Transmission carries: `transmitter` (who), `transmitter_tier` (credibility 0-3), `transmission_method` (manual-author, llm-extract, research-import, import-auraken, import-interlens), `prior_transmission` (FK to upstream Transmission or NULL for origin)
- A lens extracted by LLM from flux-review-ep11 has a reconstructible chain: `origin_source → transmission:llm-extract → transmission:auraken-ingest → transmission:ontology-import`
- Enables provenance audit queries: "show me all lenses whose transmission chain touches LLM extraction without human review" *(Addresses MIS-03 P1 source precedence + isnad Source-as-label P1 + transmission chain P1 — cross-track convergence Rank 5.)*

### G10 (measurement pre-registration, from P1 package but non-negotiable)
Before any triage code is written for Epic child #4 (flux-drive view):
- **Primary metric:** review-coverage-per-diff (did the triaged agents collectively touch every changed subsystem?). Secondary: P0/P1 count per run, user-accepted-verdict rate, **cost-per-finding** (to guard against the pick-more-agents confound).
- **Baseline:** git SHA of the current `/flux-drive` triage logic frozen at Epic #4 day 1, reproducible via a pinned invocation
- **Corpus:** 30-diff paired held-out set, labeled for ground-truth "right agents" by at least one reviewer
- **Ship/abandon thresholds:** commit to specific numbers before running the A/B — e.g., "ship if review-coverage-per-diff improves ≥ 15% at constant-or-lower cost-per-finding, abandon or redesign if below 5%" *(Addresses TLM-01 through TLM-05. Pre-registration is how measurement stays honest.)*

### G11 (auxiliary — Domain/Discipline audit before DDL)
Before committing Domain and Discipline as separate V1 types, run a concrete audit:
- Extract all existing `domains` values from 660 fd-agents frontmatter
- Extract all existing `discipline` values from 291 Auraken lenses
- Compute string-similarity and embedding-similarity overlap
- If > 30% overlap, collapse to one type with `kind: enum[informal-tag, formal-field]` and `formalization_level: float`
- If < 30% overlap, keep separate — both tracks converged on this decision gate *(Addresses OSD-01 adjacent + sibu 經部/子部 distant, Rank 4 convergence.)*

## Proposed Epic Shape (to be formalized in strategy + plan)

Working sketch — strategy step refines. Children beads created under `sylveste-b1ha` as the epic parent. Reordered post flux-review: benchmark + audit come **before** DDL so that schema decisions are made against measured behavior, not assumed behavior.

1. **Cypher benchmark spike** — Load 10k → 100k synthetic edges; EXPLAIN ANALYZE the MVP triage query; decide AGE-viable vs. redesign. Commit transcript as acceptance artifact. (G1, ~1 week)
2. **Domain/Discipline audit + `ontology-queries` module skeleton** — Audit existing 660+291 `domains`/`discipline` values (G11); scaffold the versioned query module before any view consumes it (G5). (~1 week, runs parallel-possible with #1)
3. **Schema + DDL** — 7-type ontology with all gated fields (G6 schema_version, G3 same-as, G4 bridges, G7 Evidence, G8 Lens immutability, G9 Transmission chain). Includes partial indexes, identity UUIDs, curator-promotion workflow stubs. (~1.5 weeks)
4. **Ingestion pipeline** — Three importers with explicit idempotence keys per G2; dry-run mode; per-entity transactions with manifest log for partial-failure replay; second-run regression test. (~2 weeks)
5. **Semantic dedup pass** — Calibration corpus (≥ 50 labeled pairs) built first; embedding model + threshold committed as artifacts; emits `candidate-same-as` only per G3; curator review tool for promotion; essence-text-only embedding input. (~1.5 weeks)
6. **flux-drive triage view (MVP)** — Query layer consumes `ontology-queries` module (G5). Pre-registration doc (G10 primary metric, baseline SHA, corpus, ship/abandon thresholds) committed before any triage code is written. A/B against frozen baseline on the 30-diff corpus. (~2.5 weeks)
7. **interlens MCP adapter** — Swap interlens MCP server's backend from bundled JSON to AGE via the `ontology-queries` module — validates the adapter-over-module pattern in G5 with a real second consumer. (~1 week)
8. **[Deferred to V2]** Hermes conversational view (must be an adapter over G5, not a reimplementation)
9. **[Deferred to V3]** Catalog browse surface (also an adapter over G5)

Total MVP-through-interlens migration: ~10-11 weeks on one focused lane (up from ~7 — the gates add real work, but skipping them adds much more work later). Longer if interleaved with other sprints.

## Open Questions

Resolved-by-gates are removed; remaining open questions:

- **Plugin home.** Does this live in `interlens` (extended), a new `interontology` plugin, or as an Auraken module? interlens feels natural given its existing lens-focus, but the ontology serves more than lenses. Worth a strategy-step discussion. The `ontology-queries` module (G5) might live in its own package independent of any plugin wrapper.
- **Hermes-pivot dependency.** Can flux-drive view land before Hermes overlay stabilizes, or do we need Hermes work to inform Persona schema? Likely independent — flux-drive consumes Persona+Lens, Hermes chooses them; both can evolve in parallel. Hermes view is explicitly deferred (G5 mandates it be an adapter).
- **Authoring UX.** Editing 1200+ entries across a graph is painful without an admin UI. Do we need one in V1, or does the graph stay code-authored (markdown ↔ importer ↔ AGE) with dedup curator review via CLI? G3's curator-promotion workflow at least needs a CLI affordance in V1.
- **Deprecation policy.** 660 fd-agents include dead-end experiments. Post-ingestion, do we soft-retire (tier → `retired`) or hard-delete? Lean soft-retire — ontology remembers. G8's immutable+supersedes model makes deletion structurally awkward anyway; a `retired` tier plus `valid_to` set is the cleaner path.
- **Scale headroom.** AGE scales to ~10M entities per the assessment. At current 1200 + embeddings + relationships, we're nowhere near limits. Revisit if lens generation industrializes. G1's benchmark spike will surface actual scaling behavior rather than trust the assessment's number.
- **Curator promotion workload.** G3 requires a human in the loop for candidate-same-as → same-as promotion. How many candidate-same-as edges will a real dedup run produce? If thousands, curator review becomes a bottleneck; if dozens, trivial. Strategy step should estimate from a dry-run on a sample of 100 records before committing to the workflow.
- **Fixative-triad schema.** Perfumery's Persona + Lens_A + Lens_B stabilization-triad (Track C P1) — promote to V1 schema or defer? Adds genuine expressive power but at schema complexity cost. Recommend: defer to V2 unless flux-drive triage queries need it; revisit after dedup reveals whether triadic relationships are common.
- **jarh wa-ta'dil matrix scope.** G7 grades Evidence; should the full authentication matrix (mutawatir, sahih, hasan, da'if, mawdu) also apply to Lens entries themselves, or only to the Evidence nodes they cite? Recommend: Evidence only in V1; revisit if Lens-level credibility becomes a consumer concern.

## Appendix: Why this probably isn't overengineered

Memory warned me to check: "Check assessment docs before building infrastructure" + "adopt mature tools, don't rebuild." The assessment agent flagged directly: *"The 'Palantir-style' framing is apt but right-sized — you're building personal-scale Foundry infrastructure."*

The three existing stores collectively encode object-typed semantics already; a graph DB surfaces what's already there. We are not inventing ontology — we are giving a name and typing discipline to a graph that's been implicit in the schemas all along.
